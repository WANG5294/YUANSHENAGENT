## 章节 2.1：上下文窗口作为有限资源

上下文窗口（context window）不是一个特性。它是一个预算——你花掉的每一个 Token，都意味着后续工作可用的 Token 少了一个。

这种认识很重要，因为开发者习惯性地将上下文视为"可用空间"。模型有 200K Token 的上下文？那空间大了去了。把整个代码库喂给它。附上每一个相关文件。再加上错误日志、文档、项目历史。模型会找出什么重要。

它不会找出什么重要。它所做的，是在每次迭代中处理完整的 200K Token，计算成本随输入长度的平方增长。在 20K Token 时，模型快速且准确。在 100K Token 时，它变慢了，并开始丢失前面内容。在 180K Token 时，它很昂贵，明显变慢，并且容易遗漏几千个 Token 之前的指令。上下文窗口不会断崖式下降——它是逐渐退化的。开发者不会在退化开始时收到错误消息。模型悄无声息地变差，而且这种变差很容易被归因于其他原因。

n² 问题：注意力机制（attention）——允许语言模型将上下文中任何 Token 与其他任何 Token 关联起来的机制——的计算成本随序列长度的平方增长。上下文翻倍，计算量翻四倍。这就是为什么性能是平滑退化而不是突然崩溃。这也是为什么实际可用的上下文对于高质量输出来说远小于技术最大值。一个 200K Token 的上下文窗口并不意味着你可以可靠地使用 200K Token。它意味着模型会尝试处理 200K Token，并在该范围的高端产生退化的输出。

退化在实践中的表现：智能体重复它已经完成的工作，重新读取它已经读过的文件，产生的代码忽略了在会话开始时指定的早期架构决策，或者询问两千个 Token 之前已经回答过的问题。这些都不像编译错误那样明显错误。它们是细微的错误，以一种累积的方式导致会话无法推进。

在 OpenTalon 中，上下文管理是构建连贯软件的开发会话与产生局部正确但全局不连贯的代码之间的区别——这正是第 1 章的置信陷阱，但现在我们理解了其机制。当 Claude Code 在一次数小时的会话中构建 CLI 时，上下文窗口会被工具结果、中间推理和文件内容填满。如果没有刻意的管理，会话会在实现完成之前退化。防止这种退化的策略是第 2 章和第 12 章的主题。第一步是理解当会话结束时，什么能够存活下来。

上下文管理是区分智能体工程（agentic engineering）与氛围编码（vibe coding）的技能。氛围编码就是向模型提问、接受输出、然后继续前进。智能体工程是设计信息环境——模型读取什么、何时读取、以及什么在会话之间持久化——以便在整个项目的生命周期中产生连贯的输出。第 2 章的其余部分构建了这种设计的工具。
