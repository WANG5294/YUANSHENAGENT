## 章节 2.3：记忆层级

并非所有持久化都是平等的。记忆层级（memory hierarchy）将四个层级从最可靠到最不可靠排列——而你选择的层级决定了你的信息实际存活多久。

四个层级，从最持久到最不持久排列：

**第一层——CLAUDE.md 文件。** 最可靠的层级。这些文件在每个会话开始时被读取，在 `/compact` 操作后被重新读取，并且在会话期间生成的任何子智能体都可以访问。它们无限期存活，因为它们是在版本控制下的普通文件。如果信息需要在下周、下个月或者在别人打开这个项目时可用，它应该放在 CLAUDE.md 文件中。这是不会退化的层级。

**第二层——自动记忆笔记（Auto-memory notes）。** Claude Code 在会话期间写入 `~/.claude/projects/[project-hash]/memory/` 的笔记。这些比单个会话更持久，但比 CLAUDE.md 持久性差——它们可以被手动清除，默认不在版本控制下，并且同一项目的其他开发者无法访问。对于独立开发者来说，自动记忆对中期连续性来说是可靠的。对于团队来说，则不然。

**第三层——会话记忆摘要（Session Memory summaries）。** 在会话中存活 `/compact` 操作的背景笔记。当上下文窗口填满并且 `/compact` 运行时，Claude Code 会写入关键决策、文件路径和任务状态的摘要。会话继续使用压缩后的上下文，但摘要保留了基本结构。会话记忆在关闭终端并重新打开后不会存活。它是工作日内的连续性，不是跨天的连续性。

**第四层——对话历史（Conversation history）。** 最不稳定的层级。对话历史只在会话运行且上下文窗口未被清除时存在。它包含一切：每个提示、每个响应、每次工具调用、每次文件读取。它是信息最丰富的层级，也是最短暂的。当会话结束时，它就消失了。

这个层级有一个压倒一切的实践含义：如果你通过聊天消息向 Claude Code 传达了某些信息，而它没有被记录在上层中，那么可以认为它已经丢失了。一个在对话中做出的但未被写入文件的决策，是下一个会话将不知道的决策。它会被重新做出，可能以不同的方式。

这听起来像是显而易见的事。在实践中，人们很容易把聊天消息当成持久的。"我告诉过 Claude Code 全程使用异步 SQLAlchemy。" 你把它写进 CLAUDE.md 文件了吗？如果没有，下一个会话不会知道。依赖聊天记忆来保存架构决策的开发者会发现这些决策被从未听说过它们的智能体推翻、忽略或矛盾。

在 OpenTalon 中，每一个持久化决策都存在于一个文件中——永远不在聊天消息中。架构决策存在于 `opentaion-state.md` 中。项目约定存在于根 `CLAUDE.md` 中。写作状态存在于 `progress.md` 中。当一个新的 Claude Code 会话在这个项目上启动时，它会读取这些文件并立即定位方向。它不需要被告知已经构建了什么、要遵守什么约定、或者从哪里开始——因为所有这些信息都在记忆层级的第一层，在我们刻意更改它之前，它都会一直存在。

这就是使长期运行的智能体项目保持连贯的设计原则。下一节解释了为什么需要一个独特的术语——上下文工程（context engineering）——来命名应用这一原则的学科。
