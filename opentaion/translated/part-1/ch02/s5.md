## 章节 2.5：从头编写 OpenTalon CLAUDE.md

理解 CLAUDE.md 文件的最佳方式是阅读一个必须实际工作的文件。

如果你下载了本书的仓库，你已经有了它：项目根目录下的 `CLAUDE.md` 是 Claude Code 用来编写你到目前为止读到的每一个章节的实际配置文件。它不是模板或教学示例——它是生产基础设施。本节将逐段介绍它的十个部分，解释每一个结构性决策，并以开启 OpenTalon 的行动结束：初始化项目目录并第一次打开 Claude Code。

### 写作前检查清单（第 3 节）

检查清单是第一个需要理解的东西，因为它以可操作的形式表达了本书的核心问题。Claude Code 在会话之间没有记忆。检查清单就是替代品。在编写任何章节之前，Claude Code 必须阅读 `progress.md`（我们停在了哪里？）、目标章节的大纲条目（本节必须完成什么？）、前一个章节文件（上一节说了什么？）和 `opentaion-state.md`（现在存在什么代码？）。

检查清单上的每一项之所以存在，都是因为它的缺失在开发过程中导致了真实的问题。大纲条目防止范围漂移。前一个章节防止突兀的语调断档。状态文件防止 Claude Code 编写关于尚不存在的代码的内容。检查清单不是形式主义。它是最小可行的记忆替代品。

当你编写自己的 CLAUDE.md 时，你的版本的这个检查清单会不同——但你需要一个。问题是：Claude Code 在每个会话开始时需要读取什么才能避免盲目开始？

### 技术栈部分（第 4 节）

技术栈部分使用了"固定（fixed）"这个词。这个词的选择是有意的且重要的。"固定"意味着：不要辩论这些选择。不要建议替代方案。未经明确批准不要使用其他库。

这种坚定性的原因是架构连贯性。OpenTalon 跨越 21 章和多个会话。如果技术栈发生漂移——如果某一章的代码使用 `uv` 而另一章使用 `pip`，或者某一章引用 `shadcn/ui` 组件而该组件已不在技术栈中——不一致性会累积到 110 个章节中，变得无法维护。"固定"这个词防止模型做出局部合理但全局违反本书连贯性的决策。

对于你自己的项目，技术栈部分应该列出不仅要用什么，还要不用什么——并且对两者都要具体。

### 唯一规则（第 10 节）

CLAUDE.md 的最后部分是最重要的：系统服务于工作，永远不要反过来。每一个配置、每一个技能文件、每一个钩子——如果它造成了损害产出的摩擦，就改变它。这条规则的存在是为了防止 CLAUDE.md 变成它原本设计要支持的工作的约束。

这是覆盖所有其他规则的规则。第 21 章将在一个完整项目的全部上下文中回到这条规则。

### 里程碑：初始化 OpenTalon 项目

这是 OpenTalon 软件开始的时刻。创建目录结构，放置 CLAUDE.md，然后运行 `git init`。从这一刻起，书中的每一个代码示例都是真实的。

```bash
# 创建项目目录
mkdir -p opentaion/{cli,web,api}
cd opentaion
git init
```

```
Initialized empty Git repository in /path/to/opentaion/.git/
```

现在将书仓库根目录下的 CLAUDE.md 复制到 `opentaion/` 目录中，或者使用上述结构创建一个新的。如果你是从头开始写自己的，从这五个必需部分开始：项目标识、仓库结构、会话前检查清单、技术栈和唯一规则。随着项目的发展添加其余部分。

```bash
# 验证结构
ls -la opentaion/
```

```
drwxr-xr-x  cli/
drwxr-xr-x  web/
drwxr-xr-x  api/
-rw-r--r--  CLAUDE.md
```

从项目根目录打开 Claude Code：

```bash
cd opentaion
claude
```

```
✓ Claude Code v1.x.x
  Project: opentaion
  CLAUDE.md: loaded (root)
  Memory: no previous session
>
```

### 刚刚发生了什么

Claude Code 在启动时读取了 CLAUDE.md。它现在知道了项目的标识、技术栈约束和会话前检查清单。它还不知道代码库——目录是空的。但它知道代码库将按照什么规则被构建。

这就是里程碑 1 的全部目的：不是产出代码，而是建立将指导后续每一个编码会话的信息环境。CLAUDE.md 不是已完成工作的文档。它是未来工作的契约。

在 OpenTalon 中，你刚刚初始化的 CLAUDE.md 正是 Claude Code 在第 9 章实现 CLI 智能体循环时、在第 11 章搭建 Web 平台时、以及在第 19 章部署到 Railway 时将要读取的同一个 CLAUDE.md。每一个会话都将继承你刚刚建立的约束。对这些约束的更改将在版本控制中被跟踪——CLAUDE.md 的 Git 历史就是项目的决策日志。

第 3 章将使用这个已初始化的项目来演示第一个真正的智能体任务：让 Claude Code 探索空仓库并描述自己的开发环境。
