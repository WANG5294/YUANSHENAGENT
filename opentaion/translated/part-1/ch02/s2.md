## 章节 2.2：什么在会话之间存活

Claude Code 有遗忘症（amnesia）。每一次会话都是从零开始。

这不是一个 Bug。这是语言模型工作方式的一个后果：对话历史存在于上下文窗口中，会话结束时上下文窗口被清空，下一个会话无法访问上一个会话中发生过什么。昨天花了三个小时实现 CLI 认证模块的模型，今天对此毫无记忆。它不知道哪些文件被创建了，哪些设计决策被做出了，哪些方法被尝试过又被放弃了。它就像项目不存在一样开始。

这让那些在单次会话中使用过 AI 助手并发现它们连贯的开发者感到惊讶。在一次会话内，Claude Code 是连贯的——它记得发生过的一切，因为一切都还在上下文窗口中。这种错觉在你第一次关闭终端、第二天重新打开、让 Claude Code 继续你上次停下的地方时被打破。它不知道你停在了哪里。它只知道你在新会话开始时告诉它的内容。

在会话之间存活的，精确且仅限写入磁盘的内容：

**CLAUDE.md 文件**——在每个会话开始时被重新读取。它们是最可靠的持久化形式，因为 Claude Code 被明确设计为读取它们。无论你在 CLAUDE.md 文件中写了什么——项目约定、技术栈决策、架构规则、禁止模式——都可以在该目录下运行的每一个会话开始时访问到。

**自动记忆笔记（Auto-memory notes）**——是简短的 Markdown 文件，Claude Code 可以写入 `~/.claude/projects/[project-hash]/memory/`，以在会话之间持久化观察结果。它们与 CLAUDE.md 不同：自动记忆笔记由 Claude Code 在会话期间更新；CLAUDE.md 由开发者编写和维护。

**会话记忆摘要（Session Memory summaries）**——是 Claude Code 在会话被压缩或恢复时自动写入的背景笔记。它们在同一个项目上下文内持久化，并且可以在 `/compact` 操作后存活。它们比 CLAUDE.md 更脆弱——它们不能存活所有会话边界——但它们弥补了积极开发中的项目和暂停的项目之间的差距。

**设置和权限会持久化**：`.claude/settings.json` 文件在会话之间存活，因此权限配置、钩子定义和工具限制不需要重新建立。

什么不会存活：工具调用历史、中间推理、模型在上一次会话中通过读取文件建立的"对代码库的理解"。昨天读取了项目中每一个文件的智能体，今天一个也没有读。

实际后果是：没有刻意持久化机制的每一个会话，都是盲开始的会话。智能体会读取它需要的文件，但除非有什么东西告诉它，它不会知道要读哪些文件。除非有什么东西记录它们，它不会知道上周做出了哪些决定。除非有什么东西跟踪，它不会知道项目正处于 20 个里程碑中的第 3 个。遗忘症不是一个模型可以解决的问题——它是一个开发者必须围绕设计的约束。

在 OpenTalon 中，这个遗忘症问题是 `book/progress.md` 和 `book/opentaion-state.md` 存在的原因。这两个文件是项目的持久化记忆。Progress.md 精确地告诉 Claude Code 写作停在了哪里以及接下来要做什么。Opentaion-state.md 精确地告诉 Claude Code 哪些代码存在、哪些不存在。这两个文件都不依赖于 Claude Code 之前读过它们——它们被编写为可以被没有会话历史的模型冷读时理解。这个设计要求不是偶然的。它是上下文工程（context engineering）的核心洞见，第 2.4 节将其命名并加以审视。

记忆层级——不同类型的信息存放在哪里以及它们可靠地持久化到什么程度——是这种工程的基础。下一节将描绘这个层级。
