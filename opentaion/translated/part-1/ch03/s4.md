## 章节 3.4：工具调用的经济学

每一次工具调用都有代价。理解成本结构会改变你设计任务的方式。

成本有三项：Token、延迟，有时还有金钱。Token 成本往往是最不直观的。当 Claude Code 运行一个打印 500 行日志输出的 Bash 命令时，这 500 行会被追加到对话历史中。后续的每一次迭代都必须处理这个现在已经长了 500 行的历史。一次冗长的工具调用就可能给后续的每一次迭代增加数千个 Token。成本是复利的。

这种复利正是为什么 Grep 和 Read 之间的选择比看起来更重要。一次找到包含某个函数调用的三行的 Grep 调用可能返回 150 个 Token。读取这三行所在的整个文件可能返回 4,000 个 Token。两者都给 Claude Code 提供了相关信息。但一个比另一个在上下文中留下了 3,850 个 Token 的空间，供后续所有内容使用。在二十次迭代时，这个差异是 77,000 个 Token——相当于在每个提示后为会话剩余时间增加了一篇短篇小说的长度。

延迟是第二个成本。工具调用需要时间：文件读取很快，测试套件运行很慢，网络抓取依赖网络。一个每次文件编辑后都运行完整测试套件的智能体，付出的延迟成本会使会话感觉缓慢，并在数十次迭代中累积。正确的设计通常是先批量编辑再运行测试，而不是单独验证每一次编辑。反馈循环是有价值的；每次按键都触发反馈循环则是痛苦的。

金钱成本是真实的，但依赖于模型。在 API 定价下，上述 Token 成本直接转化为费用。在订阅定价下，它们转化为速率限制消耗。两者都是有限的资源。一个重复读取同一个大文件的智能体工作流——因为开发者没有设计上下文来持久化相关信息——是一个在为重复付费的工作流。

多智能体工作流的复利成本是上述所有成本的乘数。三个并行智能体处理三个独立任务，有各自独立的上下文窗口。每个智能体支付自己的上下文累积成本。生成它们的编排器支付自己的上下文加上从每个工作者接收到的结果。一个三智能体的并行工作流通常比相同组合任务的单智能体工作流贵 2.5–3 倍，正是由于这种乘数效应。当任务真正独立且时间节省很重要时，这个成本是值得的。当开发者选择并行智能体只是因为感觉它更强大时，则不值得。

诚实的数字：智能体工作流大约消耗相同任务下单轮聊天交互的 10–20 倍的 Token。一个一直免费使用 Claude Chat 然后转用 Claude Code 进行开发工作的开发者，会很快遇到这个数字，而且这可能是一个冲击。10–20 倍的乘数来自于多轮对话的二次增长（每一轮都处理所有前一轮的完整历史）、工具输出通常很冗长、以及当智能体迭代向正确行为时的自我修正循环成倍增加 Token 消耗。

对于 OpenTalon 的开发者——我们 CLI 的用户——这些经济学正是本书包含一整章成本优化内容的原因。在第 20 章中，提示缓存（prompt caching）、模型路由（model routing）和 `/effort` 命令将把 OpenTalon 使用的每次会话成本降低大约 70%。工具调用的经济学不是一个抽象的问题。它们是我们正在构建的产品的定价模型。
