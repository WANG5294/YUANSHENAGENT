## 章节 3.1：为什么工具使用不同于文本生成

有了已建立的 CLAUDE.md 和已初始化的项目，问题从 Claude Code 知道什么转向它能做什么。

文本生成是预测。当语言模型产生一个函数时，它是在预测在当前上下文中最可能满足提示的 Token 序列。这个预测可能是正确的。它也可能看起来合理但实际上是错误的——一个可以编译但失败的函数、一个处理了快乐路径却在边界情况上崩溃的实现、一个因断言不正确而不是因行为正确而通过的测试。模型无法知道区别。它预测了 Token。这些 Token 是否对应可工作的软件，是预测过程之外的问题。

工具使用是世界中的行动。当 Claude Code 读取一个文件时，它接收到该文件中的实际字节——而不是对文件内容可能是什么的预测。当它运行测试套件时，它接收到测试套件的实际输出——而不是对测试可能说什么的合理描述。当它运行类型检查器时，它接收到实际的类型错误——而不是对哪些类型可能冲突的合理猜测。区别在于预测现实是什么与观察现实是什么之间的差异。

这种差异——预测与观察——在 AI 研究中有一个名字：接地（grounding）。一个接地的智能体，其推理是锚定在世界的实际状态上的，而不是锚定在世界可能是什么样子的统计模式上。每一次工具结果都为下一步推理提供了接地。在 Claude Code 读取一个文件之后，它的上下文中包含该文件的实际内容。在它运行测试之后，它的上下文中包含实际测试结果。"我预测了什么"与"实际是什么"之间的差距随着每次工具调用而缩小。

接地效应是累积的。一个已经读取了十二个文件并运行了三条命令的智能体，其上下文比一个只收到代码库描述的智能体在现实中接地得多。这就是为什么长时间的智能体任务——那些需要读取许多文件、运行许多命令、迭代代码直到测试通过的任务——可以产生一次性生成无法产生的结果。

并非所有工具的接地效果都相同。不可逆性频谱从安全延伸到有后果：读取文件没有副作用；写入文件可以撤销（版本控制）；运行测试套件不改变状态；删除文件在没有备份的情况下无法撤销；部署到生产环境会改变一个正在运行的系统。为智能体设计任务意味着设计智能体应该使用哪些工具以及以什么顺序使用——从观察开始（读取、搜索、分析），然后才进入行动（写入、执行、创建），这样在不可逆行动开始之前推理已经有良好接地。

这个设计原则产生了一个模式：给智能体提供能够验证自身工作的工具。测试、linter 和构建命令服务于质量，但它们还服务于更深层的东西：它们是智能体观察自身行动结果并了解行动是否正确的方式。没有验证工具的智能体是一个生成代码后就停止的智能体。有验证工具的智能体会生成代码、运行它、读取发生了什么、然后生成更好的代码。

在 OpenTalon 中，接地效应是 CLI 对于编码任务来说比聊天机器人更可靠的原因。当我们在第 9 章实现智能体循环时，循环会在编辑文件之前先读取文件，在报告成功之前先运行测试，并在继续之前检查自己的工具输出。这些检查中的每一次都是一次接地操作——智能体的预测与现实相遇并做出相应更新的时刻。循环的架构就是接地的架构。
