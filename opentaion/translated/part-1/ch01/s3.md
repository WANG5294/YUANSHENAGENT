## 章节 1.3：Claude Code 主循环的工作原理

终止条件比你想象的要简单。

Claude Code 的主循环一直运行到模型产生的响应不包含任何工具调用。此时，控制权返回给用户。模型根据其当前上下文决定任务是否完成。如果任务完成，它返回纯文本。如果任务未完成，它调用一个工具，观察结果，然后运行下一次迭代。这个循环是单线程的：一次一个迭代，每个迭代建立在前一个之上。

模型在每个迭代中看到的是完整的对话历史加上到目前为止每次工具调用的结果。不是摘要，不是压缩版本。而是自会话开始以来累积的消息、工具输入和工具输出的实际序列。这就是为什么上下文管理如此重要——每次工具调用都会增加历史记录，而历史记录在体积和成本上都在增长。

精确的机制：当你输入一个提示时，Claude Code 将其连同当前的对话历史一起发送给模型。模型的响应要么是 (a) 纯文本消息，结束本次迭代；要么是 (b) 一个或多个工具调用规范。如果响应包含工具调用，Claude Code 会执行每一个，收集结果，将它们追加到历史记录中，然后将整批内容送回模型进行下一次迭代。循环持续进行，直到模型产生一个没有工具调用的迭代。

这种架构的一个特性直接改变了你在长时间任务中与系统交互的方式：异步输入队列（asynchronous input queue）。当 Claude Code 正在运行时——执行工具调用、等待响应——你可以输入一条新的指令。Claude Code 会将输入排队，并在下一次迭代的边界处将其传递给模型。你不需要停止任务、等待它完成、然后用新指令重新开始。你可以在飞行中进行中途修正。这就是"打断（interrupting）"一个运行中的智能体的感觉背后的机制。

这种架构的简洁性是刻意的。一个具有清晰终止条件的单线程循环是可预测的。你可以推理它正在做什么以及为什么。更复杂的架构——多个并行智能体、嵌套编排、持久化后台进程——确实存在并且在特定上下文中很有用，但它们更难以调试，也更难以信任。Claude Code 的主循环用复杂性换取可控性。对于构建生产软件的独立开发者来说，这个交换是正确的。

实际的经济影响是：每次工具调用都消耗 Token 和时间。模型必须处理每次迭代中不断增长的历史记录。一个需要二十次工具调用的任务不是二十个独立操作——而是二十次迭代，每次处理一个比上一次略大的上下文。工具调用和成本之间的关系不是线性的。它比线性更糟糕：每次调用都增加了后续每次调用必须处理的上下文。这个算术使得上下文管理成为一个真正的工程问题，而不是一个学术问题。

在 OpenTalon 中，Claude Code 的主循环是我们在本书每一章中工作的环境。当 Claude Code 在第 9 章编写 CLI 智能体循环时，它正在使用自己的主循环来构建结构相似的东西。当我们在第 12 章观察到在长时间的实现会话中工具调用的累积时，我们将目睹同一个循环运行得比它被设计为高效运行的时长更长。理解这个循环有成本、有上下文、有终止条件，是智能管理它的先决条件。而这种管理正是第 2 章的主题。
