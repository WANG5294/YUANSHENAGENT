## 章节 1.5：创始问题：什么使一个工具具备智能体能力？

这个问题值得一个精确的回答：一个工具是智能体的，当它能够感知其环境、采取行动、观察结果并决定下一步做什么——而无需在每个步骤等待人类时。

这个定义中的每一个词都有分量。*感知*意味着真正的感知——读取实际文件、运行实际命令——而不是总结对环境的描述。*采取行动*意味着产生真正的改变——写入文件系统、执行代码——而不是生成文本然后由人类决定是否采取行动。*观察结果*意味着工具接收到来自自身行动的反馈——测试输出、错误消息、变更的状态——并且该反馈进入其上下文。*无需在每个步骤等待人类*是区分智能体与非常好的助手的关键：循环在任务开始和任务结束之间自主运行。

这个定义形成了一个频谱。自动补全位于一端：它预测下一个 Token，交付它，然后等待。它没有环境，不采取行动，也不观察任何东西。一个对话式 AI 助手——大多数人听到"AI 编码工具"时所想到的——位于更远一点的位置：它接收问题描述，返回解决方案描述。比自动补全好。仍然不是智能体的。人类仍然是反馈循环。

Claude Code 位于更远的位置。它感知真实的文件，执行真实的命令，并将真实的工具结果纳入其推理中。它自主地运行感知-推理-行动循环数十次迭代。它在任务完成时终止，而不是在人类决定停止给出指令时。根据上述定义，Claude Code 是智能体的——不是作为一个营销主张，而是作为其所作所为的结构性描述。

完全自主性（full autonomy）位于频谱的最远端：一个可以被赋予目标并无限期地追求它、处理所有障碍、做出所有决策、在任何时候都不需要人类参与的智能体。那不是 Claude Code。那不是 OpenTalon。那不是今天可靠存在的任何系统。完全自主的智能体在大约 23% 的复杂真实世界软件任务上取得成功。其余的都需要干预。理解工具在频谱上的位置，是不期望它们做它们不能做的事情的先决条件。

对于我们的项目，这个定义决定了 OpenTalon 的最小可行设计。CLI 必须能够感知文件。它必须能够执行命令并读取它们的输出。它必须在感知和行动之间维持自己的循环，而不需要用户将结果粘贴回提示中。任何低于这个标准的东西都只是一个智能包装器。区分真正智能体的三个能力——持久化工具使用、环境反馈、多轮规划——直接映射到这些需求上。

OpenTalon V1 将位于频谱的什么位置？在 Claude Code 和完全自主性之间，更接近 Claude Code 而不是完全自主性。它将在熟悉的代码库上处理明确指定的任务。它不会处理模糊的需求、新颖的架构或需要同时理解整个代码库的任务。这是正确的起点：一个足够有能力以有用、足够简单以理解、足够有限以诚实的系统。

创始问题已经得到回答。循环是真实的，并且在真实的工具上运行。现在还有第二个问题：循环在会话之间没有记忆。每次启动时，它都是冷启动。第 2 章是关于构建循环在跨真实项目的长度上保持连贯所需的记忆。
