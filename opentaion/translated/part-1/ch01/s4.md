## 章节 1.4：为什么 OpenTalon 会不同

理解 Claude Code 的主循环揭示了什么才是一个工具真正具备智能体能力——以及什么达不到这个标准。

大多数 AI 编码工具并不是智能体的。它们是智能包装器（smart wrappers）：一个语言模型连接到一个输入框，返回文本。输出可能是代码。用户将其粘贴到编辑器中，运行它，看到错误，将错误粘贴回输入框，收到修复，粘贴修复，然后重复。模型是聪明的部分。用户是循环。工具从不观察自己输出的结果。它从不根据代码运行后发生的情况修改自己的方法。它生成、停止、等待。

三个属性将真正的智能体工具与智能包装器区分开来。第一：持久化工具使用（persistent tool use）——在开发者实际环境中读取、写入和执行的能力，而不是在沙盒预览中。第二：环境反馈（environmental feedback）——观察行动结果（测试失败、编译错误、文件变更）并将该观察纳入下一步推理的能力。第三：多轮规划（multi-turn planning）——分解复杂任务、跟踪哪些步骤已完成、并在数十次工具调用中维持该计划的能力。

OpenTalon 将拥有全部三个。

在 OpenTalon 中，CLI 读取和修改用户项目中的文件。当它编写代码并运行测试时，它接收测试输出作为工具结果。这个结果会改变它接下来做什么。如果测试通过，它报告成功。如果测试失败，它读取错误、诊断问题、编辑相关文件、然后再次运行测试。这不是我们将要配置的一个特性——这就是感知-推理-行动循环在对真实环境拥有持久化访问权限时所做的。

OpenTalon V1 中的多轮规划比 Claude Code 的简单：它将在单个会话内跨工具调用维护一个简短的任务列表。它不会在会话之间持久化计划。它不会在途中分支或重新考虑。这些限制是刻意的。它们使系统可理解和可调试。更复杂的规划机制是在你理解了更简单机制的成本之后再添加的那种东西。

诚实的评估是：OpenTalon V1 将达不到 Claude Code 的复杂程度，而这正是构建第一个智能体系统的独立开发者的正确设计目标。Claude Code 背后有多年工程积累——广泛的工具定义、精细的权限系统、会话连续性、多智能体编排。OpenTalon 将拥有一个可工作的智能体循环、少量工具和一个清晰的架构。价值不在于功能集。价值在于理解构建每一块的成本是多少。

这让我们回到了置信陷阱。OpenTalon 将面临与第 1.1 节中描述的完全相同的失败模式。当我们在不同的上下文中构建 CLI 和 API 时，会出现集成不匹配。当 Claude Code 跨多个会话实现功能时，如果之前的决策没有被持久化到文件中，它会忘记它们。当我们添加一个涉及多个组件的功能时，一个组件的代码会自信地假设一个契约，而另一个组件则自信地违反它。这些不是假设——它们是第三部分要解决的真实问题。

OpenTalon 将与智能包装器不同，不是因为它避免了这些问题，而是因为它是由一个理解这些问题的人构建的。

这种理解始于一个问题。下一节精确地陈述了它，而答案将贯穿剩余二十章中的每一个技术决策。
