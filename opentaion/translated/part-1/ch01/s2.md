## 章节 1.2：感知-推理-行动循环（The Perception-Reasoning-Action Loop）

智能体的定义不在于它知道什么，而在于它用所学到的东西做什么。

这个区分很重要，因为它将智能体与其他所有类型的软件区分开来。函数接收输入并返回输出。聊天机器人接收消息并返回消息。智能体接收一个情境，对下一步该做什么做出判断，根据该判断采取行动，然后——这正是它成为智能体的关键——观察结果并用它来指导下一个判断。这个循环一直持续到任务完成。这个循环就是感知-推理-行动循环（perception-reasoning-action loop）。

三个相位不是比喻。它们是智能体实际执行的操作：

**感知（Perceive）**——智能体读取其环境。这意味着读取文件、运行命令并读取输出、搜索代码库、获取网页、查询数据库。感知是将外部状态转化为上下文的行动。没有感知，智能体只能根据一开始被告知的信息进行推理，而这往往是不完整的。

**推理（Reason）**——智能体思考它已经感知到的内容。问题是什么？解决它需要什么？每种方法的风险是什么？下一步应该发生什么？推理不是开发者编程的一个独立步骤——它来自模型对感知上下文的处理。推理的质量直接与感知的质量和完整性相关。

**行动（Act）**——智能体改变世界。写入文件、执行命令、创建测试、提交代码。与感知和推理不同，行动是有后果的。有些是可逆的。有些则不是。

名称中的"循环（loop）"是产生差异的部分：

```
        ┌─────────────────────────────────────────┐
        │                                         │
        ▼                                         │
   ┌─────────┐     ┌──────────┐     ┌────────┐   │
   │  感知    │────▶│  推理    │────▶│  行动  │───┘
   └─────────┘     └──────────┘     └────────┘
   读取文件        制定计划         编写代码
   运行命令        决定下一步       执行
   搜索            创建
```

智能体行动之后，它会再次感知——这一次包括其行动的结果。如果它写的代码有语法错误，接下来运行的测试就会显示该错误。智能体读取错误，推理原因，并采取行动修复它。这不是魔法。这是一个反馈循环。正是这个循环将一次性代码生成器转化为可以完成多步骤任务的东西。

将这与自动补全（autocomplete）对比。自动补全根据前面的 Token 预测下一个 Token——一次，然后停止。开发者决定如何处理这个预测。智能体在每个任务中运行几十次感知-推理-行动循环。每个循环都建立在上一个循环之上。开始时读取文件的智能体，到应用第三次编辑时已经是一个不同的智能体——拥有不同的上下文。

将这与聊天机器人（chatbot）对比。聊天机器人等待人类提供下一轮对话。人类阅读聊天机器人的输出，决定输入什么，然后输入。人类就是循环。智能体不等待。它行动之后，感知结果，并驱动自己的下一次感知。人类不再需要在每一步都参与——只参与边界：这里是任务，这里是任务完成时。

在 OpenTalon 中，当我们在第 9 章实现 CLI 的智能体循环时，这张图不是抽象概念——它是代码的字面结构。`run_loop()` 函数将调用 `perceive()`，然后 `reason()`，然后 `act()`，然后用更新后的上下文再次调用自身。现在理解了这个循环，意味着当我们写它的时候，我们不是在遵循指令。我们是在实现一个我们已经理解的东西。

这个循环不会永远运行。它有一个终止条件，这个条件内置于每一个使用语言模型的智能体的架构中。这个条件是下一节的主题。
