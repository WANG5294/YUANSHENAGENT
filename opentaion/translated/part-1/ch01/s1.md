## 章节 1.1：置信陷阱（The Confidence Trap）

这种失败模式一致到足以拥有一个名字。

开发者让 AI 编码助手实现一个功能。代码很快出现了。它通过了编译。基本的测试通过了。函数做了它名字所说的事。三十分钟后，在一次代码评审或生产事故中，问题浮出水面：新函数假设了一个六个月前就被代码库其余部分废弃的会话模型（session model），或者引入了一个只在两个模块同时加载时才出现的循环导入，或者重复了 AI 没有读取的工具文件中已有的逻辑。每个独立的部分都是正确的。系统却不是。

这就是置信陷阱（confidence trap）。陷阱不在于 AI 生成了糟糕的代码——而在于 AI 生成代码时，无论是否拥有足够的上下文来生成好的代码，都表现出同样的自信。一个没有读过代码库的人类工程师会有所保留："我觉得是这样工作的，但你去检查一下 auth 模块。"AI 助手不会有所保留。它产出一个完整、连贯、格式良好的函数，而输出内容的自信程度，并不能告诉你产生这个输出的上下文是否完整。

其机制是 Token 预测。语言模型通过预测给定前面 Token 后最统计可能的下一个 Token 来生成代码。这个优化是局部的：下一个 Token 应该与当前响应中之前的 Token 相匹配。它不是全局的：生成的代码不需要适应模型尚未完整读取的代码库架构。当上下文窗口只包含一个提示和一个函数签名时，模型生成适合该提示和签名的代码。这段代码是否适合系统的其余部分，是生成过程不会追问的问题。

考虑一个房子的类比。总承包商雇佣了熟练的专业人员：一名水管工、一名电工、一名木工。每个人在他们被分配的房间里都做得非常出色。水管工按照图纸精确地布置管道。电工按照图纸精确地铺设导线管。木工按照图纸精确地搭建墙壁框架。但没有人检查管道路线和导线管路线是否穿过同一面墙的同一空腔。每个房间独立来看都是正确的。这栋房子有一个结构性问题，只有在墙壁封起来之后才能发现。

当开发者把每个提示都当成独立任务来处理时，AI 辅助开发就会以这种精确的模式失败。"给这个函数加个缓存。""重构这个模块。""实现这个端点。"每个响应在局部都是连贯的——它在给定的上下文中解决了陈述的问题。代码库中积累着这些响应。架构的不连贯性也随之积累。

在 OpenTalon 中，这个问题在我们试图将 CLI 连接到 API 的那一刻就出现了。CLI 用一种认证流程模型编写的。API 用的是另一种。两者孤立来看都没错。两者都很自信。集成测试失败了，而 CLI 代码和 API 代码中没有任何迹象表明可能存在不匹配。在第 9 章中，当我们在编写任何实现文件之前先编写 OpenTalon CLI 规范时，我们正是在构建防止这种情况的机制：一个两个组件都读取的文档，这样它们各自的自信都建立在同一个共享模型之上。

问题不在于工具。问题在于使用工具时没有建立让工具安全的思维模型。这个思维模型有三个部分，它们是接下来三章的主题。第一部分是理解智能体（agent）到底是什么——不是"更好的自动补全"，而是一个具有特定架构、特定失败模式和特定优势的系统。这个架构从一个循环开始。
