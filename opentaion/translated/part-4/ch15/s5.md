## 第 15.5 节：实现就绪：将技术与 PRD 关联

架构已完成。`architecture.md` 描述了 Winston 设计的系统：组件结构、API 合约、数据库模式、部署计划。规划阶段在四个会话和三个智能体中产生了五个文档。在实现开始之前，还有一个问题：架构是否真的实现了 PRD？

这是第二次实现就绪检查——第一次在第 14 章中抓住了故事级的差距。这一次寻找的是架构级差距：PRD 指定了但架构未解决的需求，以及没有 PRD 需求证明其合理性的架构决策。

### 运行架构就绪检查

```
/bmad-agent-bmm-architect

"将 architecture.md 与 prd.md 和 epics-and-stories.md 进行审查。

验证：
1. PRD 中的每个功能需求都映射到 architecture.md 中的
   至少一个端点或架构决策
2. architecture.md 中的每个架构决策都由一个 PRD 需求
   证明其合理性
3. 数据库模式支持所有使用量查询端点

识别任何差距。"
```

Winston 审查这三个文档并生成一个差距报告。对于 OpenTalon，最可能的差距：

**差距 1——每日使用量端点需要零填充行。** 故事 3.1 的验收标准指出"使用量为零的天数显示高度为零的柱条（不是空白）。"这意味着 `/usage/daily` 端点必须在整个 30 天范围内每天返回一条记录，包括没有使用量发生的天数。数据库查询不能只是 `GROUP BY date`——它必须与一个日期序列连接并为缺失的日期返回零值行。Winston 向架构添加了一条注记："`/usage/daily` 查询需要一个日期序列连接。参见 architecture.md 中的实现说明。"

**差距 2——行级安全策略未指定。** 架构描述 usage_logs 应该只对拥有它的用户可读，但没有指定 Supabase RLS 策略。Winston 将策略添加到架构文档中：

```sql
-- usage_logs 的行级安全
ALTER TABLE usage_logs ENABLE ROW LEVEL SECURITY;

CREATE POLICY "用户只能读取自己的使用量日志"
ON usage_logs FOR SELECT
USING (auth.uid() = user_id);
```

**差距 3——流式延迟要求。** NFR-1 要求相对于直接 OpenRouter 调用的开销 < 200ms。架构文档描述了流式代理但没有解决 UsageLog 写入时机。在流式过程中同步写入数据库会增加延迟。Winston 更新了架构："在流式完成后使用 FastAPI 的后台任务异步写入 UsageLog 条目。不要让数据库写入阻塞流式响应。"

这三个差距通过在 Bob 读取之前更新 `architecture.md` 来解决。Bob 读取修正后的架构文档。他创建的故事文件将包括日期序列连接注记、RLS 策略和异步写入要求。

### 规划阶段产出了什么

实现阶段开始时没有架构模糊性。PRD 定义了系统必须做什么。架构定义了它将如何做到。故事定义了工作如何划分。UX 规格定义了用户看到什么。实现阶段中的每个智能体——Bob、Amelia、Quinn——每次会话开始时读取其中一个文档。

没有实现智能体需要做出架构决策。没有实现智能体需要从不完整的故事中推断用户体验。没有实现智能体会在实现中途发现两个 PRD 需求相互冲突。

这就是规划阶段的回报。不是完美的文档——差距报告显示有差距需要寻找。而是结构化的文档，将昂贵的实现中途意外的概率从几乎必然降低到几乎为零。

### 刚刚发生了什么

Winston 的架构审查发现了三个会导致实现问题的差距：一个缺失的每日图表零填充要求、一个缺失的 RLS 策略，以及流式过程中同步数据库写入的延迟风险。所有三个现在都已记录在 `architecture.md` 中。实现阶段从这些问题已解决开始。

规划阶段花费了大约六小时的实际时间——Mary 的会话、John 的 PRD 会话、John 的故事会话、Sally 的 UX 会话、Winston 的架构会话以及两次就绪检查。对于一个将花费大约 40 小时实现的 Web 平台来说，这是 15% 的规划开销。替代方案——用一个不成熟的计划进入实现——也会消耗那 6 个小时，但分散在发现会话、回溯和修复冲突中。规划阶段没有增加时间。它把时间移到了一个更便宜的阶段。
