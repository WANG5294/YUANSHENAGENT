## 第 15.4 节：API 设计：认证、密钥、计量、仪表盘

Winston 的架构文档指定了完整的 API 表面。本节讲解每个端点组的设计决策——不是实现代码，而是 Amelia 将实现且 CLI 和 Web 平台将消费的合约。

OpenTalon API 有四个功能区域：认证、API 密钥管理、LLM 代理和使用量查询。每个区域有不同的认证要求和不同的错误行为。

### 认证端点

```
POST /auth/register
Body: { "email": "user@example.com" }
Response: 200 { "message": "检查您的电子邮件以获取登录链接" }
需要认证：无
说明：触发 Supabase 魔法链接邮件。始终返回 200——
       不区分新邮件地址和已有邮件地址
       （防止用户枚举）。
```

```
GET /auth/callback
查询参数：token, type（来自 Supabase 魔法链接 URL）
Response: 302 重定向到 /dashboard
需要认证：无
说明：Supabase 在内部处理 token 交换。API
       路由存在是为了接收重定向并设置会话。
```

认证流程在 API 端是无状态的。Supabase 管理会话。API 在需要认证的路由上使用 Supabase 的 JWT 密钥验证 Supabase JWT。

### API 密钥端点

```
GET /api-keys
Response: 200 [{ "prefix": "abc12345", "created_at": "...",
                 "last_used_at": "...", "is_active": true }]
需要认证：Supabase JWT（Web 会话）
说明：仅返回密钥前缀，从不返回完整密钥。
       返回所有密钥（活跃和已吊销）用于审计可见性。
```

```
POST /api-keys
Body: 无
Response: 201 { "key": "opentaion_abc12345defg6789...",
                "prefix": "abc12345" }
需要认证：Supabase JWT
说明：生成新的 API 密钥。在此响应中返回完整明文密钥
       仅一次。仅存储 bcrypt 哈希 + 前缀。
       明文在此响应后从不存储或可检索。
```

```
DELETE /api-keys/{prefix}
Response: 204 No Content
需要认证：Supabase JWT
说明：设置 is_active = false。不删除行——
       已吊销的密钥保留在审计追踪中。
       如果前缀不属于已认证用户，返回 404。
```

### LLM 代理端点

```
POST /v1/chat/completions
Body: OpenAI 聊天补全请求格式
      { "model": "...", "messages": [...], "stream": true }
Response: 服务器推送事件流（OpenAI 格式）或 JSON
需要认证：X-Api-Key 头（OpenTalon API 密钥）
说明：验证 API 密钥，将请求代理到 OpenRouter，
       流式返回响应。从最终的 SSE 事件中提取
       使用量元数据。流式完成后写入 UsageLog 条目。
       无效/已吊销的密钥返回 401。
       如果 OpenRouter 不可达，返回 502。
       V1 中不返回 429 表示速率限制（无速率限制）。
```

代理端点是系统的关键路径。CLI 发出的每个 LLM 调用都经过此处。延迟要求（相对于直接 OpenRouter 调用的开销 < 200ms）约束了实现——流式传输期间不同步写入数据库。

### 使用量查询端点

所有使用量端点使用 Supabase JWT（Web 会话）进行认证。它们查询过滤到已认证用户 ID 的 `usage_logs` 表。

```
GET /usage/summary
查询参数：days=30（默认）
Response: { "total_tokens": 45320,
            "total_requests": 87,
            "date_range": { "start": "...", "end": "..." } }
```

```
GET /usage/daily
查询参数：days=30（默认）
Response: [{ "date": "2026-01-01",
             "input_tokens": 1200,
             "output_tokens": 450,
             "total_tokens": 1650 }, ...]
说明：在请求的时间段内每天返回一条记录，
       包括使用量为零的天数（零值行，而非省略）。
       仪表盘图表需要此项，因为"使用量为零的天数显示
       高度为零的柱条"这一验收标准。
```

```
GET /usage/by-model
查询参数：days=30（默认）
Response: [{ "model": "deepseek/deepseek-r1",
             "input_tokens": 32100,
             "output_tokens": 8400,
             "total_tokens": 40500 }, ...]
说明：按 total_tokens 降序排序。
```

### 数据库模式

两个表支持此 API：

```sql
-- api_keys 表
CREATE TABLE api_keys (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id     UUID NOT NULL REFERENCES auth.users(id),
    prefix      VARCHAR(8) NOT NULL,
    key_hash    TEXT NOT NULL,  -- bcrypt 哈希
    is_active   BOOLEAN NOT NULL DEFAULT true,
    created_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
    last_used_at TIMESTAMPTZ
);

-- usage_logs 表
CREATE TABLE usage_logs (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id         UUID NOT NULL REFERENCES auth.users(id),
    model           VARCHAR(100) NOT NULL,
    input_tokens    INTEGER NOT NULL,
    output_tokens   INTEGER NOT NULL,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

两个表都使用 Supabase 的 `auth.users` 作为用户引用。行级安全策略确保用户只能读取自己的行。
