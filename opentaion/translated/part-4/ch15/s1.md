## 第 15.1 节：Winston 设计完整系统

Winston 是架构师。Mary 获取了需求，John 将它们结构化为 PRD，Winston 决定如何构建这些需求所描述的系统。他在写出一个字之前读取 `prd.md`、`product-brief.md` 和 `epics-and-stories.md`。基于文件的手递意味着他以完整的上下文到达架构会话。

调用 Winston：`/bmad-agent-bmm-architect`。

### Winston 的方法

Winston 在一个约束下运作，这个约束塑造每一个决策：每个技术选择必须与 PRD 需求关联。如果没有 PRD 需求支持它，它就不属于 V1。

这个约束是 Winston 主要的反过度工程（anti-gold-plating）机制。一个构建 V1 工具的独立开发者面临着持续的压力来增加那些看似明显有用的能力——缓存、负载均衡、Webhook、管理面板、事件溯源。Winston 拒绝所有这些，除非它们与一个已记录的需求关联。不是因为这些功能不好，而是因为它们不是用户要求的，并且它们增加了达到 V1 的时间。

在架构设计过程中 Winston 对话的一个示例：

> **人类：** 我们是否应该为 API 密钥查找添加一个 Redis 缓存？我们将在每个请求上检查密钥。
>
> **Winston：** PRD 有 NFR-3：API 密钥哈希使用 bcrypt。它没有对密钥验证的延迟要求。一次 bcrypt 检查大约需要 100ms。你的 NFR-1 要求代理为 LLM 调用增加的总延迟 < 200ms。LLM 调用本身增加 500ms–5000ms。添加一个 Redis 实例来避免 100ms 的密钥验证并不能改善可观察的用户体验，并且增加了操作复杂性。V1 中不要使用 Redis。

这是正确的答案。Winston 根据需求进行计算，而不是根据一般的软件工程直觉。

### Winston 生成什么

Winston 生成 `architecture.md`——一个完整的系统设计，涵盖组件、数据流、数据库模式和部署架构。该文档足够具体，Bob 可以在不做架构决策的情况下从中创建实现故事。

OpenTalon Web 平台的高级组件设计：

```
组件：
  web/      — Vite + React SPA，部署到 Vercel
  api/      — FastAPI，部署到 Railway
  database  — Supabase（PostgreSQL + Auth）

数据流：
  浏览器 → web/（静态） → api/（REST） → Supabase（DB）
  CLI → api/（代理） → OpenRouter（LLM）

认证：
  Supabase Auth（魔法链接）
  会话：Supabase JWT，存储在 localStorage
  来自 web/ 的 API 请求包括 Authorization: Bearer ***
  来自 CLI 的 API 请求包括 X-Api-Key: <opentaion_api_key>

API 设计：
  /auth/register   POST — 触发魔法链接
  /auth/callback   GET  — Supabase 重定向处理
  /api-keys        GET, POST, DELETE — 密钥管理
  /v1/chat/completions POST — LLM 代理（OpenAI 兼容）
  /usage/summary   GET  — 聚合统计
  /usage/daily     GET  — 逐日细分
  /usage/by-model  GET  — 按模型细分
```

完整的 `architecture.md` 有三到五页，包括数据库模式、部署配置和错误处理策略。Winston 不做草图——他做规格说明。

### 在 OpenTalon 中

在 OpenTalon 中，Winston 的 `architecture.md` 是 Bob 生成故事文件之前的最后一个规划文档。它被提交到 `_bmad/artifacts/architecture.md`。它所描述的架构故意保持保守：没有微服务、没有消息队列、没有缓存层、没有 CDN 边缘函数。Vite 生成一个静态包。Railway 运行一个单一的 FastAPI 进程。Supabase 处理所有与数据库相关的事宜。一个独立开发者可以理解、调试和维护这个系统。第 15.2 节解释了指导这些选择的原则。
