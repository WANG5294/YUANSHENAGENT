## 第 14.3 节：Web 平台的史诗和用户故事

PRD 定义了系统必须做什么。史诗（epics）和故事（stories）定义了工作如何划分。John 同时产出两者——PRD 和故事分解是同一 PM 阶段的输出，而非独立的活动。

OpenTalon Web 平台的三个史诗对应 PRD 的三个主要功能领域：

**史诗 1：认证（Authentication）**——通过魔法链接进行用户注册、会话管理、退出登录。

**史诗 2：API 密钥管理（API Key Management）**——创建、查看（仅前缀）和吊销 API 密钥。

**史诗 3：用量仪表盘（Usage Dashboard）**——每日和按模型的 token 消耗可视化。

这三个史诗是自包含的，且大体上是顺序的。认证必须先于 API 密钥管理变得有意义。两者必须先于仪表盘有任何东西可以显示。

### 故事格式

每个故事遵循标准格式：作为一个[用户角色]，我想要[能力]，以便[收益]。每个故事都有具体的、可测试的验收标准——不是对期望行为的描述，而是可观察的结果。

验收标准是每个故事中最重要的部分。它们是 Bob（第 16 章）用来编写测试用例的内容，也是 Quinn 用来验证实现的内容。模糊的验收标准产生模糊的实现。

### 史诗 1：认证故事

**故事 1.1——用户注册**
> 作为一个独立开发者，我想用我的电子邮件地址注册，以便我可以访问 OpenTalon 的使用量追踪。
>
> 验收标准：
> - 输入有效的电子邮件并点击"注册"会在 5 秒内触发一封魔法链接邮件
> - 邮件包含一个链接，点击后创建一个 Supabase 会话
> - 会话创建后，用户被重定向到仪表盘
> - 尝试使用已注册的电子邮件注册会显示"检查您的电子邮件以获取登录链接"（不区分新用户和现有用户——安全考虑）

**故事 1.2——会话持久化**
> 作为一个已注册用户，我希望我的会话在浏览器刷新后持续存在，以便我不必反复登录。
>
> 验收标准：
> - 刷新仪表盘页面不需要重新认证
> - Supabase 会话 token 存储在 localStorage 中
> - 会话在 7 天不活动后过期

**故事 1.3——退出登录**
> 作为一个已注册用户，我想退出登录，以便在共享机器上清空我的会话。
>
> 验收标准：
> - 点击"退出登录"调用 `supabase.auth.signOut()`
> - 退出登录后，用户被重定向到注册页面
> - localStorage 会话 token 被清除

### 史诗 2：API 密钥故事

**故事 2.1——创建 API 密钥**
> 作为一个已注册用户，我想创建一个 API 密钥，以便我可以配置我的 CLI 通过 OpenTalon 路由请求。
>
> 验收标准：
> - 点击"创建密钥"生成一个新的 API 密钥
> - 完整的明文密钥只显示一次，在一个可复制的文本字段中
> - 关闭弹窗后，只显示密钥前缀（前 8 个字符）
> - 密钥作为 bcrypt 哈希存储——明文从不写入数据库

**故事 2.2——查看 API 密钥**
> 作为一个已注册用户，我想查看我活跃的 API 密钥，以便我知道哪些密钥正在使用中。
>
> 验收标准：
> - API 密钥面板显示一个表格，包含：前缀、创建日期、最后使用日期、状态（活跃/已吊销）
> - 密钥按创建日期排序，最新的在前
> - 没有密钥的用户看到一个空状态页面，带有一个"创建密钥"按钮

**故事 2.3——吊销 API 密钥**
> 作为一个已注册用户，我想吊销一个我不再使用的密钥，以便它不能再被用来代理请求。
>
> 验收标准：
> - 点击密钥上的"吊销"显示一个确认对话框
> - 确认吊销在数据库中设置 `is_active = false`
> - 已吊销的密钥在表格中仍然可见，状态为"已吊销"（审计追踪）
> - 使用已吊销密钥代理请求时返回 401

### 史诗 3：仪表盘故事

**故事 3.1——每日用量图**
> 作为一个已注册用户，我想看到过去 30 天的每日 token 消耗，以便我可以识别使用模式。
>
> 验收标准：
> - 图表显示过去 30 天每天一根柱条
> - 柱条代表当天的总 token（输入 + 输出）
> - 没有使用量的天数显示高度为零的柱条（不是空白）
> - 悬停柱条显示：日期、输入 token、输出 token、总 token

**故事 3.2——按模型细分**
> 作为一个已注册用户，我想看到按模型细分的 token 消耗，以便我可以比较不同模型的成本。
>
> 验收标准：
> - 表格每行显示一个使用的模型，按总 token 降序排列
> - 列：模型名称、输入 token、输出 token、总 token
> - 底部的合计行汇总所有模型
> - 表格反映与图表相同的 30 天窗口

剩余故事（3.3 到大约 3.6）涵盖日期范围筛选、加载状态和错误处理。John 在交棒之前生成所有故事。OpenTalon Web 平台的故事总数大约为 24 个——足够追踪所有功能行为，又不至于让规划开销超过实现时间。

### 在 OpenTalon 中

在 OpenTalon 中，这些故事被提交到 `_bmad/artifacts/epics-and-stories.md`。每个故事以上述格式编写：一行用户故事加上三到五个具体的、可测试的验收标准。John 在产生 PRD 的同一 PM 会话中生成此文档。在文档被提交之前，人类审查每个史诗的完整性——PRD 的所有功能需求是否至少在至少一个故事中得到了体现？第 16 章以 Bob 将此文档转换为 24 个自包含的故事文件开始。
