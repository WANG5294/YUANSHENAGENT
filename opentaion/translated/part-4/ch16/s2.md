## 第 16.2 节：故事解剖：零模糊性格式

Story-001.md 已在第 16.1 节中展示。本节解释为什么每个部分存在，以及没有它会出现什么问题。

一个 BMAD 故事文件有八个部分。每个部分服务于一个特定目的。去掉任何一个，Amelia 的实现质量都会以可预测的方式下降。

### 第一部分：故事 ID 和标题

```markdown
# 故事 001：用户注册
```

ID 是一个零填充的序号。它提供排序信息——story-001 在 story-002 之前——以及阻塞项（"story-012 被 story-008 阻塞"）的稳定引用。标题是与史诗文档中相同的简短描述，使得为给定故事找到故事文件变得容易。

### 第二部分：状态

```markdown
## 状态：ready
```

有效值：`draft`（Bob 仍在编写）、`ready`（Bob 已完成、人类已审查）、`in-progress`（Amelia 正在实现）、`review`（Amelia 已完成、等待人类审查或 Quinn 测试）、`done`（所有验收标准通过）。

Amelia 在开始之前读取状态。她不会实现状态为 `draft` 的故事。这防止了基于 Bob 尚未完成的故事进行实现——这会在不完整的指令基础上产生错误的实现。

### 第三部分：用户故事

```markdown
## 用户故事
作为一个独立开发者，我想用我的电子邮件地址注册
以便我可以访问 OpenTalon 的使用量追踪。
```

标准格式。"以便"子句确立了意图。当 Amelia 遇到故事未涉及的一个实现决策时，她参考"以便"子句来评估哪个选项更好地服务于意图。这是用户故事允许 Amelia 做出的唯一设计决策。

### 第四部分：上下文

```markdown
## 上下文
这是第一个故事。尚不存在认证。
Supabase 项目已创建，Auth 已启用。
web/ Vite 项目存在，带有一个空白的 App.tsx。
api/ FastAPI 项目存在，只有一个健康检查端点。
```

这是抗遗忘部分。Amelia 从头开始。没有这一部分，她不知道 Supabase 项目是否已创建、Vite 项目是否存在，或者之前的故事是否添加了她不应重新创建的文件。

上下文从 Amelia 的角度编写：在这个故事开始时，她需要了解代码库的什么状态？Bob 在上下文节中提到的一切在故事开始时都是真实的。未提到的一切都应假设为不存在。

### 第五部分：验收标准

```markdown
## 验收标准
1. GET / 显示注册页面（电子邮件输入框 + 按钮）
2. 提交有效电子邮件调用 supabase.auth.signInWithOtp()
...
```

编号的。具体的。可测试的。每个标准都是可观察的——对特定输入的操作产生特定输出。

验收标准是实现的检查清单。Amelia 在将故事标记为完成之前逐一检查它们。如果标准 3 说"成功状态显示'检查您的电子邮件以获取登录链接'"而她的实现显示"魔法链接已发送！"，标准失败。确切的文字很重要。

### 第六部分：技术说明

```markdown
## 技术说明
- Web：使用 Supabase JavaScript 客户端（supabase-js v2）
- 不使用路由——使用状态变量上的条件渲染
- 仅使用 Tailwind 类（无 shadcn/ui）
```

这一部分桥接了架构文档和实现。架构决策在这里，被翻译成这个特定故事的实现指令。Amelia 在实现 story-001 之前不会重读 `architecture.md`——相关的架构决策已经被 Bob 提取并放在这里。

当 Bob 编写这一部分时，他会问：`architecture.md` 中的哪些架构决策适用于这个故事？Amelia 在没有显式约束的情况下可能会做错什么？这些就变成了技术说明。

### 第七部分：测试要求

```markdown
## 测试要求
- 单元测试：RegistrationForm 渲染电子邮件输入框和提交按钮
- 单元测试：提交无效电子邮件显示验证错误
- API 测试：POST /auth/register 使用有效电子邮件返回 200
```

Quinn 使用这一部分作为她的测试计划。Amelia 在实现之前（TDD）或与实现同时编写测试——两者都可以接受，但测试必须在故事完成之前通过。测试要求足够具体，可以毫无歧义地编写；足够广泛，可以覆盖重要的行为而不过度指定。

### 第八部分：范围外

```markdown
## 范围外
- 登录（那是 story-002，处理魔法链接回调）
- 会话持久化（story-003）
```

这是防止范围蔓延的部分。Amelia 做事很彻底。没有显式的范围外约束，她会实现看似明显相关的行为——而她对它们确实相关是正确的。问题在于在 story-001 中实现它们会创建依赖关系，使 story-002 更难以独立实现和测试。

范围外部分是一个合约：Amelia 只实现验收标准中列出的内容，不超过这些。当她完成 story-001 时，注册页面提交并显示一条成功消息。它不处理魔法链接回调。它不持久化会话。它不导航到仪表盘。那些是故事 002、003 和 004。

### 在 OpenTalon 中

在 OpenTalon 中，每个故事文件都遵循这八个部分的结构。Bob 在 Amelia 开始之前生成所有 24 个故事文件。一致性意味着 Amelia 的阅读体验是可预测的——她知道在哪里找到上下文、在哪里找到验收标准、在哪里找到范围外约束。当她完成 story-001 时，她在故事文件中将状态标记为"done"并拾起 story-002。故事按顺序推进，每个都建立在前一个的输出之上，每个都准确告诉 Amelia 前一个留下了什么。第 16.3 节跟随 Amelia 完成 story-001 的实现。
