### 独立开发者的冲刺节奏

独立开发者的冲刺没有固定的时间盒。一个故事在完成时就是完成了，而不是在两周过去后。冲刺是相关故事的逻辑分组，而不是时间约束。

在实践中，OpenTalon Web 平台大约跨越三个冲刺：

**冲刺 1——认证 + API 密钥（故事 001–010）：** 核心认证流程、魔法链接、会话管理、API 密钥创建、查看和吊销。这个冲刺建立了安全基础。冲刺 2 和 3 中的任何内容没有它都无法工作。

**冲刺 2——代理 + 用量计量（故事 011–017）：** LLM 代理端点、用量日志记录、三个用量查询端点。这个冲刺实现了价值主张——计量和记录 token 使用量。

**冲刺 3——仪表盘 + 优化（故事 018–024）：** 用量图、模型细分表格、加载状态、错误处理、日期范围选择。这个冲刺使数据可见。

### 处理阻塞项

上面的冲刺状态注记显示了一个真实的阻塞项：story-012（每日用量图）发现 `pg_series()`，即生成日期序列所需的 PostgreSQL 函数，在 Supabase 的免费套餐中不可用。

当 Amelia 遇到此问题时，她不会默默地绕过去。她将故事状态更新为 blocked 并添加注记。Bob 创建一个缺陷故事（story-008b），描述变通方案：在 Python 中生成日期序列而不是在 SQL 中，然后将它与数据库结果合并。

Story-008b 在 story-012 恢复之前实现。sprint-status.yaml 显式反映了这种依赖关系。

### 里程碑

在 story-024 结束时，Web 平台可以工作：

- 用户可以访问注册页面，输入电子邮件，并收到魔法链接
- 点击魔法链接创建会话并重定向到仪表盘
- 仪表盘显示使用量数据（最初为空）并允许创建 API 密钥
- 代理端点接受来自 CLI 的请求并记录使用量
- 仪表盘图表显示累积的每日使用量

这是 V1 行为。不够精致，不是每个边缘情况都完整，但端到端功能正常。一个真实用户可以注册、获取 API 密钥、配置 CLI，并看着他们的使用量出现在仪表盘中。

### 在 OpenTalon 中

在 OpenTalon 中，当 sprint-status.yaml 中所有 24 个故事都处于"done"状态且完整测试套件通过时，达到里程碑 M15。平台在本地开发环境中上线。第五部分将"在本地工作"转换为"在生产环境中工作"——第 17 章自动化质量门禁，第 18 章完成测试金字塔，第 19 章将系统部署到 Railway 和 Vercel。BMAD 方法论的贡献在此结束：一个工作的 Web 平台，从编写第一行实现代码之前就已完整的规划包开始，逐个故事构建而成。
