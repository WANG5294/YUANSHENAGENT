## 第 16.1 节：Bob 创建超详细的故事文件

规划阶段产生了五个文档。实现阶段以一个智能体将这些文档转换为 Amelia 可以执行的格式开始：超详细的故事文件。

Bob 是 Scrum Master 智能体。他的工作是冲刺规划（sprint planning）——不是实现，不是审查，而是将规划文档翻译成自包含的实现指令。Bob 的输出决定了 Amelia 是成功还是挣扎。模糊的故事文件产生模糊的实现。

调用 Bob：`/bmad-agent-bmm-sm`。Claude Code 采用 Bob 角色并读取 `architecture.md`、`epics-and-stories.md` 和 `_bmad/templates/story-tmpl.md`。

### "超详细"意味着什么

一个传统的用户故事：

```
故事：用户注册
作为一个独立开发者，我想用我的电子邮件地址注册
以便我可以访问 OpenTalon 的使用量追踪。

验收标准：
- 注册发送一封魔法链接邮件
- 点击链接创建一个会话
```

同一个故事的 BMAD 故事文件：

```markdown
# 故事 001：用户注册

## 状态：ready

## 用户故事
作为一个独立开发者，我想用我的电子邮件地址注册
以便我可以访问 OpenTalon 的使用量追踪。

## 上下文
这是第一个故事。尚不存在认证。
Supabase 项目已创建，Auth 已启用。
web/ Vite 项目存在，带有一个空白的 App.tsx。
api/ FastAPI 项目存在，只有一个健康检查端点。

## 验收标准
1. GET / 显示注册页面（电子邮件输入框 + 按钮）
2. 提交有效电子邮件调用 supabase.auth.signInWithOtp()
3. 成功状态显示"检查您的电子邮件以获取登录链接"
4. Supabase 发送魔法链接邮件（在 Supabase 仪表盘中验证）
5. API 中的 POST /auth/register 触发 Supabase 魔法链接
   （对于 CLI 注册——Web 直接使用 Supabase 客户端）
6. 注册页面处理错误：无效的电子邮件格式显示
   "请输入有效的电子邮件地址"

## 技术说明
- Web：使用 Supabase JavaScript 客户端（supabase-js v2）
- API：POST /auth/register 调用 supabase.auth.admin.generateLink()
- 本故事不处理会话——那是 story-002
- 不使用路由——使用状态变量上的条件渲染：
  { showSuccess ? <SuccessMessage /> : <RegistrationForm /> }
- 仅使用 Tailwind 类（无 shadcn/ui，无外部组件库）
- supabase-js 已在 package.json 中（在项目设置中添加）

## 测试要求
- 单元测试：RegistrationForm 渲染电子邮件输入框和提交按钮
- 单元测试：提交无效电子邮件显示验证错误
- 单元测试：成功的 supabase 调用显示成功消息
- API 测试：POST /auth/register 使用有效电子邮件返回 200
- API 测试：POST /auth/register 使用无效电子邮件返回 422

## 范围外
- 登录（那是 story-002，处理魔法链接回调）
- 会话持久化（story-003）
- 登录后导航到仪表盘（story-004）
- 任何仪表盘内容——用户注册不需要仪表盘
```

区别不是美学上的。Amelia 在一个全新的上下文窗口中运行，只有这个故事文件以及它告诉她要读取的文件。没有上下文节，她不知道什么已经存在。没有技术说明，她会选择自己的实现方法——可能与架构冲突。没有范围外节，她可能在注册的同时实现会话处理，无意中耦合了两个本应独立的故事。

### 冲刺规划会议

Bob 为 OpenTalon 进行的规划会议以一个结构化的请求开始：

```
/bmad-agent-bmm-sm

"读取 _bmad/artifacts/ 中的 architecture.md 和 epics-and-stories.md。
为第一个冲刺创建故事文件：
故事 001 到 008，涵盖史诗 1（认证）和
史诗 2（API 密钥管理）的前两个故事。

使用 _bmad/templates/story-tmpl.md 作为模板。
将每个故事写入 _bmad/stories/story-NNN.md。

确保每个故事是自包含的：Amelia 必须能够
通过只读取故事文件和它明确引用的文件来
正确实现它。"
```

Bob 在一次会话中生成八个故事文件。人类在将每个故事标记为 sprint-status.yaml 中的"ready"之前进行审查。

### 在 OpenTalon 中

在 OpenTalon 中，Bob 在 Amelia 实现任何故事之前生成全部 24 个故事文件。完整的故事集被提交到 `_bmad/stories/`，并且 sprint-status 被更新为反映所有故事处于"ready"状态。这种前置加载是经过深思熟虑的——如果 Amelia 已经开始，Bob 的规划会话就会被实现问题污染。先创建所有故事意味着 Bob 在思考实现的全貌，而不仅仅是下一个立即的任务。第 16.2 节深入分析 story-001.md 的结构，并解释每个部分背后的设计决策。
