## 第 9.3 节：编写 OpenTalon CLI 规范

上一节描述了 SPEC.md 的作用。本节编写一个。OpenTalon CLI 规范在此完整呈现——就像它将出现在仓库中一样——因为阅读一个完整的规范的过程与编写它的过程同样具有教育意义。请注意不仅包含什么，还排除了什么。

下面的规范指导从第 9.4 节开始、贯穿第 12 章的 CLI 实现。这些章节中的每个架构决策都可以追溯到这个文档中的某一行。

---

### 规范：OpenTalon CLI — 版本 1

**编写时间：** 第 9 章 | **指导范围：** 第 9–12 章 | **状态：** 活跃

---

#### 目的

OpenTalon CLI 是一个用于 macOS 的基于终端编码代理。给定一个自然语言提示，它会推理当前目录的代码库，使用文件系统和 shell 工具来收集上下文并进行更改，通过 OpenRouter 调用 LLM 进行推理，并将结果报告到终端。

CLI 是一个独立的工具。它不需要 OpenTalon Web 平台或 API 即可运行。它只需要一个 OpenRouter API 密钥即可运行。

---

#### 非目标（V1）

以下功能不会包含在 V1 中。包含它们将增加范围，超出本书时间线内可以构建和测试的范围。

- **没有多文件差异视图。** 结果以单个文件编辑的形式显示，而不是统一差异。
- **没有对话持久化。** 每个会话是独立的。不保存会话历史。
- **没有插件系统。** 工具集固定为下面描述的六个工具。
- **没有 GUI 或 Web 界面。** 仅限终端。
- **不支持 Windows。** 仅限 macOS。Linux 可能可以使用但未经测试。
- **没有流式部分结果。** 输出在完成时显示，不会逐令牌流式传输。

---

#### 入口点

```
python -m opentaion "your prompt here"
```

或者通过已安装的 Homebrew 命令：
```
opentaion "your prompt here"
```

标志：
- `--model MODEL` —— 覆盖默认模型（默认：`deepseek/deepseek-r1`）
- `--max-turns N` —— 最大代理循环迭代次数（默认：10）
- `--effort [low|medium|high]` —— 思考预算（默认：medium）
- `--dry-run` —— 显示将要执行的操作而不实际执行

---

#### 工具集（固定）

CLI 向 LLM 暴露恰好六个工具：

| 工具 | 功能 |
|------|------|
| `read_file(path)` | 读取文件内容 |
| `write_file(path, content)` | 将内容写入文件（创建或替换） |
| `edit_file(path, old, new)` | 替换文件中的特定字符串 |
| `run_bash(command)` | 执行 shell 命令并返回 stdout/stderr |
| `glob_files(pattern)` | 列出与 glob 模式匹配的文件 |
| `search_files(pattern, path)` | 搜索文件内容中的正则表达式模式 |

所有工具都包含基本安全检查：
- `run_bash`：阻止匹配危险模式的命令（与 PreToolUse 钩子相同的模式）
- `write_file` 和 `edit_file`：在写入前扫描内容中的 API 密钥模式

---

#### 上下文管理

CLI 管理一个大约 100,000 个令牌的上下文窗口（保守估计以留出余量）。

上下文预算分配：
- 系统提示 + 工具定义：约 8,000 令牌（预留）
- 对话历史：最多 60,000 令牌，之后触发压缩
- 单个工具输出：上限为 10,000 令牌（超出时截断并附带通知）

当对话历史超过 60,000 令牌时，上下文管理器：
1. 汇总到目前为止的对话历史（约 1,000 令牌）
2. 用汇总替换完整历史
3. 从汇总继续会话

这种压缩对用户是不可见的——代理就像没有发生压缩一样继续运行。

---

#### OpenRouter 集成

CLI 调用 OpenRouter 的 API，该 API 与 OpenAI 兼容。它使用 `POST /v1/chat/completions` 端点。

所需配置（通过项目根目录的 `.env` 或环境变量）：
```
OPENROUTER_API_KEY=sk-or-...
OPENTAION_MODEL=deepseek/deepseek-r1       (可选，覆盖默认值)
```

客户端实现：
- 429（速率限制）时自动重试：指数退避，最多 3 次重试
- 超时：每次请求 60 秒
- 无流式传输（V1）

---

#### 错误处理策略

| 错误条件 | 行为 |
|---------|------|
| OpenRouter API 密钥缺失 | 打印错误，立即退出，返回码 1 |
| 工具调用失败（非零退出） | 在上下文中包含错误，继续循环 |
| 上下文压缩失败 | 打印警告，尝试继续 |
| 达到最大轮次 | 打印已完成和剩余工作的摘要，退出，返回码 0 |
| 网络超时 | 重试一次，然后打印错误并退出，返回码 1 |

---

#### 验收标准

完成意味着：

1. `uv run python -m opentaion "列出此目录中的 Python 文件"` 无错误运行并产生响应
2. 代理循环最多运行 `--max-turns` 次迭代然后终止
3. 超过 10,000 令牌的工具输出被截断并附带通知
4. 当历史超过 60,000 令牌时，上下文压缩自动触发
5. OpenRouter 重试逻辑在 429 时重试，如果第二次请求成功则成功
6. `--dry-run` 显示计划中的工具调用而不执行它们

### 编写规范之后

上面的规范就是仓库中 `cli/SPEC.md` 的内容。现在创建该文件。用消息"添加 CLI 规范（SPEC.md）"提交它。

在 OpenTalon 中，本文档是接下来四章的架构锚点。每个实现会话从阅读它开始。实现任务中的每个不明确之处都参照它来解决。当实现过程中做出的决策在规范中没有涵盖时——具体的压缩算法、确切的重试时机——该决策在添加到代码之前先添加到规范中。

规范并不完美。它需要更新。重要的是它在实现开始之前就存在了，并且当实现揭示了规范没有预料到的事情时，它会被更新。一个反映了实际构建的代码的规范，比一个描述了最初计划但与现有代码不同的规范更有价值。
