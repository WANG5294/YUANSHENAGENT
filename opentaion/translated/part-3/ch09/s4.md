## 第 9.4 节：计划模式作为契约性检查点

第 9.3 节编写的 SPEC.md 是长期契约——这个文档指导跨多个章节的多个会话。计划模式（Plan Mode）在更小的粒度上操作：它是单个会话的契约性检查点，在一个任务开始之前。

这个类比很精确。当 Claude Code 在计划模式下生成一个计划，你审查并批准它时，你正在进入一个契约：这就是将要完成的事情，并且你已经同意了它。一旦实现开始，计划就是参考。如果 Claude Code 在实现过程中偏离了计划，这种偏离就是违约——需要注意到并处理的事情，而不是默默接受。

### 计划模式的作用

计划模式（通过按 Shift+Tab 或告诉 Claude Code"切换到计划模式"激活）将 Claude Code 限制为只读操作。在计划模式下：

- Read、Glob 和 Grep 可用——Claude Code 可以阅读代码库
- Write、Edit 和 Bash 被阻止——Claude Code 不能做出更改
- MCP 服务器查询可用——Claude Code 可以检查 GitHub、查询数据库

计划模式下的 Claude Code 产生分析和建议的方法。它可以阅读每个相关文件，映射依赖关系，识别需要更改什么以及为什么。它不能做的事情是更改任何东西。计划先于行动。

这个限制主要不是安全机制——它是一个强制函数。当 Claude Code 被限制为规划时，它会生成更好的计划，因为它不能退回到"让我试试这个，看看是否有效"。它必须在执行之前承诺一种方法。

### 审查计划

当 Claude Code 生成计划时，用三个问题来审查：

**它是否引用了真实的文件？** 一个说"修改认证模块"而没有指定文件路径的计划是不精确的。一个说"编辑 `api/src/opentaion_api/routers/auth.py`，特别是 `handle_verify` 函数"的计划是精确的。引用实际文件名的计划可以对照真实的代码库进行验证。使用模糊引用的计划则不能。

**它是否符合规范？** 将每个计划中的更改与 SPEC.md 进行对比。如果计划建议添加规范中未包含的功能，质疑它。规范是范围的边界。如果范围需要改变，先更新规范。

**它是否识别了歧义？** 一个承认不确定性的计划，比一个对一切都显得自信的计划更诚实。一个说"OpenRouter 客户端的重试逻辑未指定——除非另有指示，我将使用 2 秒初始延迟的指数退避来实现"的计划，让你有机会确认或纠正这个假设。一个在不提及的情况下实现重试逻辑的计划，剥夺了你这个机会。

### 何时使用计划模式

计划模式会为一个工作会话增加十到十五分钟。在以下情况下，这个成本是值得的：

- 任务将修改超过两个文件
- 任务触及架构边界（添加 API 端点、更改数据 schema、修改代理循环）
- 范围不明确——你不确定完整的更改集是什么
- 上一个会话在没有干净检查点的情况下结束，你需要重新定位

在以下情况下，计划模式不是必需的：
- 任务在 SPEC.md 或 GitHub issue 中已完全指定——计划已经存在
- 更改局限于一个文件，范围明确
- 你在做探索性工作，目标是理解而不是实现

启发式方法：如果你会对 Claude Code 选择修改的内容感到惊讶，使用计划模式。如果更改如此清晰以至于意外是不可能的，直接编码。

### 当计划出错时该怎么办

计划在第一次通过时出错的情况比正确的情况更多。这正是审查它的目的——在执行之前捕获错误。

当计划出错时：
1. 识别具体的错误假设或建议的行动
2. 明确纠正它："计划假设认证模块在 auth.py 中，但实际上它被分散在 auth.py 和 dependencies.py 中"
3. 要求 Claude Code 修改计划的相关部分
4. 再次审查

不要执行你知道有错的计划。实现将反映计划的缺陷，而在实现之后修复它们比在计划中修复更昂贵。
