## 第 9.1 节：为什么没有计划架构就会崩溃

第一部分介绍了自信陷阱（confidence trap）：Claude Code 生成的文本看起来正确，因为每个标记与它前面的标记在局部上是一致的。第二部分构建了塑造它运行环境的基础设施。第三部分面对的是智能体软件开发中最困难的问题：跨会话的架构一致性（multi-session architectural consistency）。

单个会话可以产生高质量的代码。陷阱不在于一个会话。而在于会话的累积——每个会话冷启动，每个会话做出合理的局部决策，但这些决策并不知道之前会话中做出的决策。

### 崩溃模式

具体的失败模式是这样的：在第一个会话中，Claude Code 为 `UsageLog` 设计了一个数据库 schema，其中有一个 `user_id` 外键直接指向 `User` 表。这很合理。在第三个会话中，Claude Code 设计了 API 认证层，并决定 API 密钥属于用户，但使用量应该按密钥而不是按用户追踪。这也合理。在第五个会话中，Claude Code 编写了仪表板查询，发现 `UsageLog` 有 `user_id`，但查询需要 `api_key_id`。第一个会话的 schema 与第三个会话的决策不匹配。

两个会话都没有犯错误。每个都做出了局部一致的决策。架构上的不一致是第一个会话的假设与第三个会话的决策之间的差距——以及第一个会话和第三个会话彼此不知道的事实。

这不是假设。这是智能体软件开发的默认失败模式。上下文窗口在会话之间被清空。模型上下文中的架构也随之消失。会话之间唯一幸存的东西是写入磁盘的内容。

### 复合效应

崩溃是复合的，因为每个错误的决策都为下一个会话创造了基础。第五个会话通过添加一个未计划的 `api_key_id` 连接表来绕过 schema 不匹配。第七个会话添加了一个假设连接表结构的查询。第九个会话尝试添加缓存，发现连接结构与规范中的缓存方法不兼容。在每一步，合理的局部解决方案都为下一个会话创造了新的不一致性。

到第十二个会话时，代码库在技术上功能完整，但在架构上是不连贯的：一个个别正确的决策的集合，不能形成一个连贯的整体。调试变成了考古学。添加功能需要从没有连贯意图的代码中逆向工程意图。开发者开始绕过代理来处理任何触及核心架构的内容——这与初衷完全相反。

### 解决方案：SPEC.md

解决方案是一个书面的规范，它在会话之间幸存下来，并且 Claude Code 在每个实现会话之前读取它。这个文件不是大纲，不是 PRD，也不是文档。它是正在构建的组件的技术契约。

SPEC.md 精确地回答四个问题：
1. 这个组件做什么？（能力，用具体的术语）
2. 它不做什么？（非目标，明确陈述）
3. 确切的接口是什么？（函数签名、API 端点、数据 schema）
4."完成"的验收标准是什么？（可测试的、具体的陈述）

一个回答了这四个问题的 SPEC.md 创建了一个在会话之间幸存下来的共享参考。当第五个会话打开并要求 Claude Code 编写仪表板查询时，Claude Code 读取 SPEC.md，发现 `UsageLog` 有 `api_key_id`（因为 SPEC.md 被编写为指定正确的 schema），并正确编写了查询。会话间的一致性来自文件，而不是来自模型的记忆。

SPEC.md 与 CLAUDE.md 有两个不同之处：它是组件特定的（每个正在构建的组件有一个 SPEC.md），并且它是可变的（它随着规范通过规划过程而演变）。CLAUDE.md 一旦编写就相对稳定。SPEC.md 是一个活跃的工作文档。

### 与 OpenTalon 的关联

在 OpenTalon 中，CLI 的 SPEC.md 是将在第 9 章到第 12 章中指导一切的文件。它将指定代理循环、上下文管理方法、OpenRouter 集成、工具集和错误处理策略。在编写一行 CLI 代码之前，规范已经以书面形式存在。每个编写 CLI 代码的会话都首先读取 SPEC.md——因为 CLAUDE.md 的清单要求如此。规范是上下文窗口无法提供的架构记忆。
