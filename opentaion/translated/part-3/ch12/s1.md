## 第 12.1 节：/compact、/clear 与连续性

第 11 章的并行构建产生了一个包含三个功能组件的代码库。上下文挑战现在发生了变化：花了三章构建的代码库无法容纳在单个会话的上下文窗口中。长时间运行的会话、读取大量文件并累积工具调用输出的会话，会在任务完成之前撑满上下文预算。管理这一点是一种操作纪律，而非需要解决的技术问题。

两个命令和两个标志涵盖了活动开发中上下文管理策略的全部范围。

### /clear：全新开始

`/clear` 擦除所有对话历史。上下文窗口变为空。CLAUDE.md 文件在下次发送消息时重新加载。工具调用历史、已读取的文件内容、先前的决策——全部消失。

在以下情况下使用 `/clear`：
- 当前任务已完成，下一个任务不相关
- 会话累积了来自探索性工作的噪音，且这些噪音不再相关
- 先前任务中的上下文会主动混淆下一个任务——比如不再存在的测试产生的调试输出、已被删除的文件中的内容

`/clear` 的代价是丢失一切。当会话的工作已提交时——上下文中没有任何不在仓库中的内容——这种代价是可以容忍的。但在任务中途，累积的上下文（计划、决策、实现当前状态）是已完成工作的唯一记录，这时就不能容忍了。

### /compact：压缩延续

`/compact` 在不擦除对话的情况下对对话进行摘要。Claude Code 生成会话的紧凑表示：关键决策、活跃的文件路径、函数名称、仍然相关的错误消息以及当前任务状态。此摘要替换完整的对话历史。上下文被释放；连续性得以保留。

关键限制：`/compact` 是有损的。摘要不是转录。那些在对话中但未被纳入摘要的信息将丢失。会话早期建立的微妙约束（"不要在此文件中使用全局变量，因为会破坏测试模拟"）如果对摘要过程不够显著，可能无法在压缩后保留。

为压缩过程提供显式指导，以保留重要的内容：

```
/compact 保留：本次会话中修改的所有文件路径、当前失败的测试列表、在 context.py 中不使用全局状态的决定
```

指令限制了压缩过程优先考虑的内容。它不能保证完美保留——但它将摘要引向连续性所需的内容。

### 带指令的 /compact 与不带指令的对比

不带指令时，`/compact` 保留它推断为重要的内容。这对于具有明确焦点的短会话效果尚可。对于触及了许多文件、探索了多种方法并得出了特定约束的会话——无指导的压缩会遗漏东西。

编写 `/compact` 指令的开销是三十秒。重新发现因压缩而丢失的约束的开销是十到三十分钟。编写指令是值得的。

### 连续性标志

当一个会话未完成工作就结束时，对话状态会保存在 Claude Code 的历史记录中。两个标志用于恢复该状态：

**`claude --continue`** ——恢复最近的会话。对话历史加载，会话仿佛从未结束般继续。适用于返回被中断的会话。

**`claude --resume [SESSION_ID]`** ——按 ID 恢复特定会话。会话 ID 在 `/status` 的输出中显示。当你需要返回到两天前的会话，而非最近的会话时使用此标志。

两个标志都会重新加载对话历史。这个历史可能很大——一个带有许多工具调用的长会话在压缩前可能累积 50,000+ token 的历史。会话恢复了，但上下文预算已经部分消耗。如果会话在结束前已被压缩，恢复的会话从压缩摘要而不是完整历史开始。

### 决策启发式

| 情况 | 操作 |
|------|------|
| 任务已完成，无待办工作 | `/clear` |
| 上下文正在填满，任务完成了一半 | `/compact [带指令]` |
| 需要昨天的会话状态 | `claude --continue` 或 `claude --resume` |
| 开始一个完全不相关的任务 | `/clear` |
| 第二天早上返回被中断的工作 | `claude --continue` |

### OpenTalon 的关联

在 OpenTalon 中，每个主要实现会话的模式是：编写代码、运行测试、提交通过的代码，然后在下一个主要任务之前使用 `/clear`。压缩保留给任务足够复杂以至于跨越上下文边界的会话——那种读取二十个文件并在做出任何更改之前建立起详细心智模型的调试会话。这里的上下文管理不是 CLI 中的上下文管理器（它管理 token 预算）；而是开发者关于何时保留或重置会话状态的纪律。
