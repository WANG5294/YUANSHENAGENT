## 第 10.4 节：强制执行 TDD：Superpowers 与 tdd-guard

TDD 在被遵循时才有效。第 10.3 节描述的模式——先测试后实现、最小代码、不修改测试——只有在其执行纪律同样可靠时才可靠。Claude Code 在其自然倾向下，会先写代码再写测试。训练数据中包含的代码文件远远多于测试文件，而且"编写实现"的实例远远多于"先编写测试"。默认路径是先代码。TDD 需要覆盖这个默认。

本节介绍两个外部强制执行工具以及 OpenTalon 使用的方法——两者都不是。

### Superpowers 技能

Superpowers 是一个社区构建的 Claude Code 技能市场。它包括一个 TDD 强制执行技能，能自动检测 Claude Code 何时在未编写相应的失败测试的情况下编写了实现代码，并删除该代码：

```bash
# Install the Superpowers plugin marketplace
/plugin marketplace add obra/superpowers-marketplace

# Install the TDD skill from the marketplace
/plugin install superpowers@superpowers-marketplace
```

当 TDD 技能激活时，任何对源文件（非测试文件）的 Write 或 Edit 操作都会触发一个检查：是否存在对应的失败测试？如果没有，写入被阻止，Claude Code 被告知先编写测试。

其机制是一个具有测试/源文件区分意识的 PreToolUse 钩子。它是 CLAUDE.md 指令的结构性强制版本。

Superpowers 可用且有效，但它不是 OpenTalon 必需的。安装它增加了一个需要维护的依赖项和一层基础设施。对于构建将超越本书寿命的生产系统的团队来说，它可能值得投资。对于 OpenTalon 的构建，有一个更简单的方法。

### tdd-guard

`tdd-guard` 是一个开源的 PreToolUse 钩子项目，实现了先测试后执行的强制执行，无需完整的 Superpowers 技能系统。它作为一个独立的钩子脚本安装：

```bash
npm install -g tdd-guard
```

将其作为 PreToolUse 钩子配置在 settings.json 中，设置一个针对源文件上的 Write 和 Edit 操作的匹配器。当触发时，它检查正在被写入的文件是否有对应的测试文件，以及该测试当前是否失败。如果没有失败测试存在，写入被阻止。

像 Superpowers 一样，tdd-guard 可用且有效。权衡是相同的：一个需要理解和维护的额外依赖项，具有特定的行为。

### OpenTalon 的方法：CLAUDE.md + /effort

OpenTalon 既不使用 Superpowers 也不使用 tdd-guard。其执行方法分为两部分：

**CLAUDE.md 指令** —— 添加到根 CLAUDE.md：

```markdown
## Test-Driven Development (IMPORTANT)

IMPORTANT: This project uses test-driven development. The rule is:

1. Tests are written BEFORE implementation code
2. Never modify a test file to make tests pass — modify the implementation
3. Implement only the minimum code required to make the current failing tests pass
4. When a task says "implement X", write tests for X first, confirm they fail,
   then implement X

If you are about to write implementation code and no test exists for it,
stop and write the test first.
```

**`/effort medium` 用于 TDD 会话** —— 中等努力意味着 Claude Code 在编写实现之前会仔细推理测试需求。在低努力下，Claude Code 可能产生合理但不正确的测试。在中等努力下，测试更精确，实现更有针对性。

这种方法之所以有效，是因为 Claude Code 在 CLAUDE.md 指令被标记为重要，并且会话以加强它们的上下文开始时，确实会遵循这些指令。需要注意的是，它是基于指令的，而非结构性的。在长会话的末尾，当上下文饱和时，指令可能被不那么可靠地遵循。对于 OpenTalon 的构建——它在上下文填满时使用 `/compact`，在主要任务之间使用 `/clear`——这种退化是可管理的。

### 选择一种方法

正确的执行方法取决于项目的重要性和团队：

- **独立开发者，书籍项目**：CLAUDE.md + /effort。最低开销，对构建来说已经足够。
- **独立开发者，生产产品**：考虑 tdd-guard。当代码库将超过开发者对 TDD 纪律的专注时长时，结构性执行值得安装成本。
- **团队**：将 Superpowers 或 tdd-guard 提交到仓库。当多人贡献时，TDD 纪律必须是结构性的，因为没有任何个人足够可靠地手动执行它。

### 与 OpenTalon 的关联

在 OpenTalon 中，TDD 指令现在位于根 CLAUDE.md 中。每个实现会话开始于 Claude Code 阅读该指令。`/effort medium` 默认值适用于所有 TDD 会话。这两个控制对于构成 OpenTalon 构建的大约 20 个实现会话来说已经足够。
