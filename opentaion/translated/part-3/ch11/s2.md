## 第 11.2 节：编排器-Worker 模式

上一节确定了多智能体何时有意义。本节解释它如何实现——具体来说，是编排器-Worker 模式（orchestrator-worker pattern），这是 Claude Code 用于并行任务分解的结构化方法。

该模式有两个角色。编排器是一个 Claude Code 会话，它分析大型任务，将其分解为独立的子任务，并将每个子任务委托给一个 Worker。Worker 独立执行分配的任务并返回结果。编排器集成这些结果。

### 编排器

编排器是一个常规的 Claude Code 会话。它的区别不在于特殊的模式或配置——而在于它执行的任务。它不是直接实现功能，而是协调 Worker 对功能的实现。

编排器的职责：
1. 分析整体任务并识别真正独立的子任务
2. 为每个 Worker 编写清晰、完整的任务规格说明
3. 使用 Task 工具派生 Worker
4. 等待 Worker 完成（或处理部分完成）
5. 将 Worker 的输出集成到主上下文中

编排器不与 Worker 并发运行——它派生 Worker，等待它们的结果，然后处理这些结果。如果编排器派生三个 Worker，它会接收所有三个结果，然后进行集成推理。

### Task 工具

Task 工具是编排器派生 Worker 的方式。编排器用任务描述（可选的特定上下文）调用它。Worker 只接收：
- 传递给 Task 工具的任务描述
- 仓库的当前状态（它可以读取文件）

Worker 不会收到编排器的对话历史。每个 Worker 从一个全新的上下文窗口开始。这就是上下文隔离（context isolation）的好处：Worker 不会被编排器的 40,000 个 token 上下文污染。每个都从零开始。

编排器只收到 Worker 的最终消息——已完成工作的摘要。如果 Worker 写了二十个文件并执行了三十次工具调用，编排器只看到最终报告。完整的工作历史留在 Worker 的上下文中，之后被丢弃。

一个并行派生两个 Worker 的编排器提示：

```
我需要同时构建 OpenTalon 的 CLI 核心和搭建 Web 项目。

Worker A 任务：
"实现 OpenTalon CLI 智能体循环，文件位于 cli/src/opentaion/agent.py。
规格说明在 cli/SPEC.md 中。上下文管理器在 cli/src/opentaion/context.py 中。
OpenRouter 客户端在 cli/src/opentaion/llm.py 中。
使 cli/tests/test_agent.py 中的所有测试通过。
运行测试：cd cli && uv run pytest tests/test_agent.py -v
报告：哪些测试通过，实现了什么，下一步是什么。"

Worker B 任务：
"使用 Vite + React + TypeScript + Tailwind 搭建 OpenTalon Web 项目。
目标目录：web/。参考：web/CLAUDE.md 了解约定。
创建：vite.config.ts, tailwind.config.ts, src/App.tsx, src/Login.tsx, src/Dashboard.tsx
运行：cd web && npm run build 验证编译是否成功。
报告：创建了什么，构建是否成功。"

使用 Task 工具并行执行两个 Worker。
```

### 深度限制

Worker 不能派生 Worker。智能体树的最大深度为 1：编排器 → Worker。尝试派生子 Worker 的 Worker 将失败并报错。

这个限制是为了防止失控的智能体树，后者可能导致指数级的成本增长。一个拥有十个 Worker、每个 Worker 再派生十个子 Worker 的编排器，在完成任何工作之前就创建了一百个会话。深度为 1 的约束限制了成本和复杂性。

如果一个 Worker 的任务本身足够复杂，可以从多智能体处理中受益，正确的做法是将该任务返回给编排器，让编排器为其派生专门的 Worker。编排器是协调点；Worker 是执行点。

### OpenTalon 的具体示例

```
编排器提示：同时构建 OpenTalon CLI 智能体循环和
Vite Web 脚手架。规格说明在各自的
CLAUDE.md 和 SPEC.md 文件中。
```

Worker A 接收：
```
在 cli/src/opentaion/agent.py 中实现智能体循环。读取 cli/SPEC.md、
cli/src/opentaion/context.py 和 cli/src/opentaion/llm.py 获取上下文。
使 cli/tests/test_agent.py 中的测试通过。报告你的结果。
```

Worker B 接收：
```
搭建 web 项目。读取 web/CLAUDE.md 了解约定。
在 web/ 中创建文件结构。运行构建。报告你的结果。
```

两个 Worker 并行运行。编排器接收它们的报告并决定：如果都成功，进入集成；如果一个失败，派生一个专门的调试 Worker。

### OpenTalon 的关联

在 OpenTalon 中，编排器-Worker 模式在第 11 章中运行，以并行构建三个组件。编排器持有 SPEC.md 知识和集成检查清单。Worker 只持有构建其组件所需的内容。该模式的价值在于每个 Worker 从头开始——不受其他 Worker 的上下文污染，不受规划会话的累积历史影响——并产生专注的、范围明确的输出。下一节添加了 git 隔离层，防止 Worker 互相覆盖文件。
