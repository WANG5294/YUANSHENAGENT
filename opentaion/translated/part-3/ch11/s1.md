## 第 11.1 节：当单智能体达到上限时

前两章构建了顺序式、单智能体开发的规划和测试基础设施。单个会话搭配清晰的规格说明和测试套件，可以产出可靠的代码。但那只适用于一个组件。OpenTalon 项目有三个组件：CLI、Web 和 API。按顺序构建——先完成 CLI，再构建 Web，再构建 API——是安全的选择，但也是最慢的选择。这些组件是真正独立的，Web 的构建没有理由非要等 CLI 构建完成。

多智能体编排（multi-agent orchestration）正是为这种场景而生：任务真正独立，不共享中间状态，且无需协调即可并行推进。

### 单智能体的天花板

单个 Claude Code 会话只有一个上下文窗口（context window）、一个工作线程和一个模型的推理能力。这些限制并非人为设定，而是根本性的：上下文窗口容纳了智能体当前会话所知的一切——对话历史、已读取的文件内容、接收到的工具输出。当工作规模足够大时，上下文窗口就不够了。

对于 OpenTalon 的构建，单智能体的天花板体现在这里：构建 CLI 需要理解 Python 异步模式（async patterns）、Click 约定和上下文管理方法。构建 Web 需要理解 React、TypeScript 和 Tailwind 模式。构建 API 需要理解 FastAPI 路由、SQLAlchemy 和 Supabase 集成。同时理解所有这些领域，并且达到能做出正确实现决策的深度，在写下任何代码之前就会撑满一个上下文窗口。

### 表明需要多智能体的症状

以下三个症状表明任务已超出单智能体的容量：

**跨域上下文溢出（Cross-domain context overflow）。** 任务需要同时读取前端、后端和数据库代码，合并后的内容超出了实际的上下文限制。一个在做出决策前需要读取所有三个组件代码库的会话，已经没有空间留给实现了。

**真正独立的子任务（Genuinely independent subtasks）。** 子任务之间不共享状态。CLI 不会调用 Web 组件中的函数。Web 组件不会从 API 代码中导入。每个组件都可以独立构建和测试，集成在定义的接口边界处发生（在 OpenTalon 的情况下，是 HTTP 和环境变量）。

**从并行中受益的截止日期压力（Deadline pressure that benefits from parallelism）。** 三个顺序构建需要三倍的时间。三个并行构建只需要最耗时的那个构建的时间。当组件相互独立且时间节省很重要时，并行是正确的工具。

### 成本

多智能体并非免费。成本包括：

**Token 成本：每个并行任务增加 2–3 倍。** 每个智能体都有自己的上下文窗口。一个派生出三个 Worker 并集成其结果的编排器（orchestrator）需要为四个会话付费，而不是一个。额外成本大约是相同合并工作量下单智能体成本的 2.5 倍。

**协调开销（Coordination overhead）。** Worker 需要清晰、完整的任务规格说明。编排器必须检测 Worker 何时被阻塞、解读其输出并决定下一步行动。协调工作不出现在代码中，但会出现在 token 账单中。

**合并复杂性（Merge complexity）。** 三个并行智能体基于同一规格说明工作，本应产生互补的代码。实际上，集成时会出现细微的不对齐：API 合约（contract）因一个字段名不同而差异、跨组件的环境变量名不匹配、假定的默认值冲突。第 11.5 节涵盖了集成检查清单。

### 原则

当任务真正独立时使用多智能体。当任务共享状态时——即一个任务的输出是另一个任务的输入——单智能体更好。跨智能体协调状态的开销超过了在单个上下文中处理它的开销。

对于 OpenTalon，CLI、Web 和 API 组件满足独立性标准。它们共享一个规格说明并共享环境变量，但它们的代码在构建时不会相互调用。集成发生在运行时，通过 HTTP 进行。并行构建是合适的。

### OpenTalon 的关联

在 OpenTalon 中，第 11 章是三个组件同时构建的地方。CLI 智能体实现智能体循环和工具集。Web 智能体搭建 Vite 项目并构建仪表盘。API 智能体实现 FastAPI 端点和数据库模型。编排器分配这些任务，监控进度，并在所有三个组件完成后集成结果。接下来的几个小节构建编排基础设施——Task 工具模式和 git worktree 隔离——使这成为可能。
