## 第 11.4 节：智能体团队：实验性的群智能前沿

编排器-Worker 模式已经成熟且可靠。在 Claude Code 当前的能力中，还有另一种多智能体模型尚未成熟：智能体团队（Agent Teams）。本节描述它们是什么，它们与编排器-Worker 模式有何不同，以及为什么 OpenTalon 在核心构建中不使用它们。

将其纳入本书的原因是，今天实验性的东西明天就会成为标准实践。现在理解智能体团队，意味着当它们成熟时，过渡将是熟悉的，而非陌生的。

### 什么是智能体团队

智能体团队是多个独立的 Claude Code 实例，它们共享一个任务列表并可以相互发送消息。与编排器-Worker 模式（存在层次关系——编排器指示 Worker）不同，智能体团队是对等（peer-to-peer）的。团队领导智能体通过写入共享任务列表将任务分配给 Worker 智能体。Worker 智能体认领任务，执行它们，并将结果报告回任务列表。其他智能体可以观察此状态并做出响应。

这种模型是真正的群智能（swarm）：没有单个智能体掌握工作的全貌。每个智能体都有自己的上下文、自己的专长和自己的执行线程。协调通过共享任务列表发生，而不是通过控制型编排器的显式消息。

### 它们与编排器-Worker 的区别

| 维度 | 编排器-Worker | 智能体团队 |
|------|---------------|------------|
| 关系 | 层次型 | 对等型 |
| 持续时间 | 临时（任务范围） | 持久 |
| 协调 | 显式指令 | 共享任务列表 |
| 上下文共享 | 无（隔离） | 通过任务列表状态 |
| 稳定性 | 生产就绪 | 实验性 |

关键区别在于持久性。编排器-Worker 会话在任务结束时终止。智能体团队是持久的——智能体运行直到显式停止。这使得长期后台工作成为可能，但也创造了新的故障模式：无限循环的智能体、在同一个任务上冲突的智能体、上下文随时间无界增长的智能体。

### 实验性状态

智能体团队需要以下环境变量：
```bash
export CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1
```

该功能尚不稳定。文档明确指出了这一点：行为可能在 Claude Code 版本之间发生变化，已知存在边缘情况，且故障模式的理解程度不如编排器-Worker 模式。对于重要的工作——生产代码、书籍内容、任何有截止日期的工作——编排器-Worker 是正确的选择。

### 为什么 OpenTalon 不使用它

CLI + Web + API 的并行构建非常适合编排器-Worker 模式：三个定义明确、独立的任务，具有清晰的完成标准。智能体团队会增加协调开销（管理共享任务列表）、持久性复杂性（团队智能体何时停止？）和实验性风险（该功能可能行为异常），却没有任何额外收益。

多智能体 2–3 倍的成本溢价在编排器-Worker 方法中已经存在。智能体团队会在其之上增加额外的开销。对于有界且规格明确的并行任务，编排器-Worker 是高效的选择。

### 智能体团队的发展方向

智能体团队模型是多智能体向真正自主系统的自然延伸：一组可以在夜间运行、从待办事项中认领工作、完成工作并在早上报告结果的智能体。对于大规模自动重构、全面的测试生成，或跨大型代码库的文档综合——需要数小时而非数分钟的任务——持久化的对等模型具有真正的优势。

限制因素是可靠性。运行数小时的系统必须优雅地处理故障，在智能体被中断时不丢失工作，并且在多个智能体对同一工件做出贡献时产生连贯的输出。这些属性在实验性实现中尚不可靠。

当智能体团队稳定后，工作流程将改变：开发人员定义任务待办事项和团队配置，启动团队，然后稍后返回审查结果，而不是手动派生编排器并监控 Worker。人的角色进一步从执行转向审查。

### OpenTalon 的关联

在 OpenTalon V1 中，不使用智能体团队。并行构建使用编排器-Worker 模式和 git worktree。在 OpenTalon V2（第 21.5 节）中，当智能体团队成熟后，第六部分中的扩展任务——优化生产代码库的 API 成本——是基于团队执行的一个候选。本节是前瞻上下文，而非指令。下一节回到具体问题：将三个并行构建合并为一个连贯的系统。
