## 章节 P.1：元项目（Meta-Project）

本书正是用它所传授的工具写成的。

这不是一个思想实验，也不是某一章结尾的华丽收尾。你现在读到的书稿，是使用 Claude Code——也就是我们将花 21 章学习使用的同一个智能编码助手——起草、修改和组织的。每一节产生这些文字的会话，都遵循着本书所描述的系统：一个告诉智能体如何行为的 `CLAUDE.md` 文件、编码编写流程的技能文件、以及能在会话之间延续的进度跟踪。这个系统正在自身之上运行。这就是元项目。

本书的核心主张是：智能体工程（agentic engineering）是一门拥有可学习原则的学科——它不是一套提示技巧的集合，也不是等待工具升级的问题。提出这个主张容易，证明它却很难。证明就是我们所使用的、用来构建本书及其内部软件的系统。

这套软件就是 OpenTalon：一个可工作的 macOS CLI 编码智能体，附带有用户注册功能和 Token 用量仪表盘的 Web 平台。到第 21 章最后一页时，OpenTalon 将可以通过 Homebrew 安装，部署在 Railway 和 Vercel 上，并且任何有邮箱地址的人都可以使用。它不是玩具，不是伪代码。它能编译、运行和发布——而构建它所采用的每一项技术，都在首次应用的那一章中得到解释。

为什么要同时做两件事？因为一本描述智能体工程却不付诸实践的书，往好了说是缺乏说服力，往坏了说就是误导。第三部分的每一个模式之所以存在，都是因为我们在构建 OpenTalon 时需要它。第二部分的每一项配置之所以存在，都是因为我们在本书运行的开发环境中使用它。系统服务于工作，工作验证系统。如果这个循环听起来耳熟，没错——这正是 Claude Code 用来编写软件的同一个反馈循环，也是整本书所要论证的核心。

在 OpenTalon 中，这种元结构有一个具体的体现：仓库中同时包含书稿和正在构建的软件，而 Claude Code 同时编写两者。你在第 5 章将会读到的 `.claude/skills/` 目录，正是产出这些章节的同一个目录。跟踪哪些章节已完成的那个 `book/progress.md` 文件，正是我们在第 2 章中作为智能体遗忘问题解决方案所教授的同一机制。你可以一边阅读系统，一边阅读关于系统的描述。这不是巧合——这是将论点物理化。

这本书写给独立开发者（solo developer）。具体来说：那些独自发布产品的人，那些已经经历过 AI 生成的代码能编译但做错事的人，那些希望从内部理解智能体系统、而不是看营销页面的人。你不需要事先用过 Claude Code。你需要能舒适地阅读 Python 和 TypeScript，能在终端中运行命令，并且能容忍在构建完整答案的过程中暂时存在不完整的回答。

OpenTalon V1 会有粗糙的边缘。这不是免责声明——这正是重点。一个独立开发者的第一个智能体系统本就应该有粗糙的边缘。你从这些边缘中学到的东西，才是这本书真正要教给你的。

下一节将描述当本书结束时你将构建完成的内容：OpenTalon 的三个组件以及它们如何连接。从那里开始。
