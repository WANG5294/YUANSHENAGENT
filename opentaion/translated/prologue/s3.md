## 章节 P.3：你将构建完成的内容

在开始之前，这里是用具体术语描述的最终状态。

一个陌生人通过链接找到了 OpenTalon。她访问 Web 平台，输入邮箱地址，收到一个魔法链接。点击后进入仪表盘，创建了一个 API 密钥。密钥完整显示一次——她复制下来。在终端中，她运行 `brew install yourgithub/opentaion/opentaion`。不到一分钟，她就有了 CLI。她设置两个环境变量：`OPENTAION_API_KEY` 和 `OPENTAION_API_URL`。然后她运行了第一个任务：

```bash
$ opentaion "auth.py 里有类型错误，找出来并修复"
```

CLI 读取了 `auth.py`。它思考类型注解，识别出函数签名和调用点之间的冲突，并产生了一个编辑。它应用编辑，运行 `mypy auth.py` 验证修复，然后报告：

```
✓ OpenTalon v1.0.0  模型: deepseek/deepseek-r1  上下文: 6,840 tokens
→ 正在读取 auth.py...
→ 发现 authenticate() 和 verify_token() 中的 2 个类型错误
→ 正在应用修复...
→ 正在运行 mypy...
✓ 完成  (4 次工具调用 · 2.1 秒 · $0.00)
```

她检查仪表盘。6,840 个 Token 以 `deepseek/deepseek-r1` 为归属出现在今天日期下的柱状图中。她可以看到过去 30 天的总用量，按天和按模型细分。

这就是最终产品。本书中其他一切，都是从当下到那里的路径。

在完成状态下，OpenTalon 的仓库将包含大约 35 个源文件，分布在三个组件中：CLI 有 8 个 Python 文件，API 有 14 个 Python 文件（含迁移文件），Web 平台有 8 个 TypeScript 文件。总应用代码约为 3,500 行。测试套件将覆盖 CLI 的智能体循环、上下文管理器和工具实现；API 的认证、密钥管理、代理和用量端点；以及 Web 平台通过 Playwright 端到端测试的关键流程。大约 20 个测试文件。这不是全面覆盖——这是一个独立开发者能够实际维护的覆盖范围。

这是一个 V1 版本。这个描述是精确的，不是谦虚。仪表盘没有分页。CLI 不支持多个并发任务。错误信息功能齐全但未必友好。部署是手动的，没有完全自动化。这些不是失败——它们是一个独立开发者为了交付能用的东西、而不是设计完美的东西所做出的决策。每一个权衡都是经过深思熟虑的，并且在书中做出决策的那个地方都得到了解释。

这个完成的系统确实证明了一个论点：一个独立开发者，使用本书中的技术，可以设计、构建、测试并部署一个可工作的多组件软件平台——CLI、API 和 Web——使用 Claude Code 作为主要实现工具。开发者理解和维护的系统，就是开发者自己构建的系统。理解债务（comprehension debt）是真实存在的，本书对此有专门论述。但代码能工作、测试能通过、用户可以注册并使用它。

这就是"发布"对于一个独立开发者的意义。

让这一切成为可能的思维模型，出现在第一行代码之前。第 1 章从一个问题开始，这个问题每一个使用过 AI 编码工具的开发者最终都会问：为什么输出看起来正确，但做的事却是错的？
