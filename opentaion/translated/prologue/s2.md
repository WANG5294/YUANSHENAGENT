## 章节 P.2：OpenTalon 系统概览

在开始构建之前，你需要一张地图。

OpenTalon 由三个组件构成：一个运行在 macOS 终端上的 CLI、一个位于 CLI 和语言模型之间的 API、以及一个供用户注册和管理访问权限的 Web 平台。每个组件都足够独立，可以单独构建——我们也将在第 11 章中并行构建它们。但它们通过一个单一流程连接在一起，你应该在编写第一个文件之前理解这个流程。

CLI 是用户实际工作的界面。它接收自然语言提示，调用 OpenTalon API，将模型的响应流式返回终端，并应用结果——编辑文件、运行命令、报告变更内容。这是一个通过 Homebrew 分发的 Python 应用程序，使用 Click 构建命令接口，使用 Rich 构建终端显示。

API 是平台的骨干。它对每个请求进行 API 密钥认证，将 Token 用量记录到数据库，并将调用代理到 OpenRouter。从 CLI 的角度看，API 看起来与 OpenAI 的 API 完全一样——它们使用相同的协议。从平台的角度看，API 是计费、速率限制和用量追踪的所在地。CLI 和用户的浏览器都不会直接调用 OpenRouter。

Web 平台是用户注册和管理访问权限的地方。它处理三项任务：魔法链接（magic link）邮箱认证、API 密钥的创建和撤销、以及一个展示 Token 随时间消耗情况的用量仪表盘。这是一个部署在 Vercel 上的 Vite + React 应用程序。仪表盘通过查询 API 的用量端点，用 Recharts 按模型分组的柱状图展示每日 Token 用量。

在 OpenTalon 中，每一个架构决策都源于一个约束：API 必须位于 CLI 和模型之间。仅仅这一个约束，就让用量追踪成为可能，让 API 密钥认证变得实用，也让 CLI 可以保持为一个轻量级 Python 脚本而无需知道任何关于计费的事情。当你在后续章节中看到一个看似不必要的间接层设计时，请回到这张图。

OpenRouter 是 OpenTalon 用户的 LLM 提供商。它将多个开源模型聚合在一个 API 密钥后面，并通过 OpenAI 兼容的接口暴露出来。图中列出的四个模型都可以在 OpenRouter 的免费层使用——无需信用卡。没有 OpenRouter 预算的读者也可以跟进本书的每一章，部署一个可工作的系统，而无需在模型推理上花一分钱。

现在需要提一个先决条件：Python 组件（CLI 和 API）需要 Python 3.12+，并使用 `uv` 管理。Web 平台和我们在第 13 章安装的 BMAD 方法论工具都需要 Node.js。如果你的机器上已经安装了 Python 和 Node.js，你就拥有了一切所需。

下一节将从用户的角度描述 OpenTalon——不是我们要构建什么，而是使用我们构建的东西是什么感觉。
