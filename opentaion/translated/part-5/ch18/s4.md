## 18.4 节：监控生产环境中的 AI 生成代码

测试套件在部署时验证行为。监控在运行时验证行为——在真实用户发送真实请求、带来测试未曾预料到的真实边缘情况之后。两者都是必需的，谁也替代不了谁。

AI 生成的代码在生产环境中具有与手写代码不同的特定失败模式。理解这些失败模式有助于确定监控什么。

### AI 生成代码的失败模式

**过度抽象。** 技术上正确但过于通用导致速度慢的代码。Amelia 有时会生成一个查询，返回所有记录然后在 Python 中过滤，而不是在 SQL 中过滤——因为 Python 方法更通用，而故事没有指定性能约束。这会产生正确的输出，但速度比 SQL 版本慢 10 倍。

**边缘情况缺少错误处理。** 快乐路径测试充分；边缘情况则不然。Amelia 实现了故事指定的内容。如果故事没有指定"当 OpenRouter 返回 504 时会发生什么"，那么该情况的错误处理很可能要么不存在，要么不正确。

**关于输入格式的错误假设。** 一个期望特定 JSON 结构的端点，当 CLI 发送了略有不同的结构时，会静默失败或产生令人困惑的错误。故事指定了一种格式；CLI 实现假设了另一种。

### OpenTalon 需要监控什么

独立开发者的最低监控设置：

```
1. 各端点的 API 错误率
   - 告警：任何端点在 5 分钟内错误率 > 5%
   - 原因：捕获静默失败和边缘情况异常

2. 代理端点响应时间
   - 告警：10 分钟内 p95 响应时间 > 3 秒
   - 原因：捕获过度抽象的查询和连接池问题

3. Token 数量异常
   - 告警：任何单次请求记录超过 500K Token
   - 原因：捕获 Token 数量膨胀的计量 bug

4. 失败的认证尝试
   - 告警：5 分钟内同一 IP 超过 20 次失败认证
   - 原因：捕获凭证填充和暴力破解尝试
```

### 监控实现

Railway 为已部署的服务提供内置日志访问和基本指标。对于 V1 版本，这已经足够：

```python
# In the FastAPI middleware — add structured logging for monitoring
import structlog

logger = structlog.get_logger()

@app.middleware("http")
async def log_requests(request: Request, call_next):
    start = time.time()
    response = await call_next(request)
    duration = time.time() - start

    logger.info(
        "request",
        method=request.method,
        path=request.url.path,
        status=response.status_code,
        duration_ms=round(duration * 1000)
    )
    return response
```

Railway 会捕获这些结构化日志。每日的 Supabase 查询用于检查异常：

```sql
-- Check for token count anomalies in the last 24 hours
SELECT user_id, model, input_tokens + output_tokens as total_tokens
FROM usage_logs
WHERE created_at > now() - interval '24 hours'
  AND input_tokens + output_tokens > 500000
ORDER BY total_tokens DESC;
```

如果此查询返回了行数据，说明计量代码出了问题。

### 调试工作流

当生产中出了问题时：

1. **本地复现。** 失败发生在真实输入上；在修改任何内容之前，先在本地用相同的输入复现。

2. **让 Claude Code 解释相关代码。** 不是修复它——而是解释它。理解代码试图做什么，是理解它做错了什么的前提。

3. **编写一个能复现 bug 的测试。** 测试是修复的规范。当测试通过时，bug 就修复了。

4. **修复实现。** 有了一个失败的测试和对代码的理解，修复方案通常就很明确了。

这个工作流将理解债务减少嵌入了 bug 修复过程。每个被修复的 bug 都产生了一个记录边缘情况的测试，以及一个比之前更理解相关代码的开发者。

### 在 OpenTalon 中

在 OpenTalon 中，生产监控在系统上线之前就已配置好（第 19 章）。上述四个指标是初始集。随着真实用量的积累，会根据实际发生的问题——而不是理论可能发生的问题——添加额外的监控。Railway 日志提供了原材料；structlog 中间件提供了结构；Supabase 异常查询提供了每日检查。企业级监控工具增加了成本和复杂性，但为拥有小用户群的 V1 系统并没有提供相应比例的价值。
