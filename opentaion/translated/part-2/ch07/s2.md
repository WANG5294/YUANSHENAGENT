## 第 7.2 节：用于仓库管理的 GitHub MCP

GitHub MCP 服务器将 Claude Code 的触角延伸到仓库托管层。代码审查会在 pull request 上留下 Claude Code 无法看到的评论。CI 检查报告的状态存在于 GitHub 的界面中，而非终端中。Issue 跟踪的 bug 和功能只存在于 GitHub 的数据库中。没有 GitHub MCP，Claude Code 对所有这些都视而不见。有了它，存在于 GitHub 中的反馈循环就变得和本地文件一样可访问。

### GitHub MCP 提供的能力

GitHub MCP 服务器由 GitHub 维护，涵盖了完整的仓库管理面：

- **Issues**：创建、读取、更新、列出、搜索、关闭 issues
- **Pull requests**：创建 PR、读取 PR diff、阅读和回复审查评论、合并、关闭
- **分支**：创建、删除、列出分支；比较引用
- **工作流**：触发工作流运行、检查运行状态、读取工作流日志
- **仓库**：读取仓库元数据、列出贡献者、访问提交历史

完整的工具列表很长，并且随着 GitHub API 的更新而演变。对于 OpenTalon 开发来说，最重要的三个工具是：

**`get_pull_request_comments`** —— 读取 pull request 上的所有审查评论。当代码审查返回十个具体建议时，Claude Code 可以直接读取它们并按顺序处理每一个。这就是 `/pr_comments` 内置命令在底层使用的方法。

**`get_issue`** 和 **`create_issue`** —— 按编号读取 issue 并创建新 issue。当 Claude Code 遇到无法在当前上下文中修复的 bug（范围不对、打开的文件太多）时，它可以创建一个已追踪的 issue，而不是只在对话中记录——在那里它将被遗忘。

**`get_check_runs`** —— 读取提交或分支的 CI 状态。当推送触发 CI 失败时，Claude Code 可以直接通过 MCP 服务器读取失败日志，而不需要开发者复制粘贴错误信息。

### 安装与配置

GitHub MCP 服务器需要一个 GitHub 个人访问令牌。在用户级别（而非项目级别）配置，以将令牌排除在 git 之外：

```bash
# Install the GitHub MCP server (requires Node.js)
npm install -g @github/mcp

# Add to Claude Code as a user-level MCP server
claude mcp add github \
  --scope user \
  --transport stdio \
  --command "npx @github/mcp" \
  --env GITHUB_PERSONAL_ACCESS_TOKEN=ghp_yourtoken
```

私有仓库的令牌需要 `repo` 范围，公共仓库需要 `public_repo` 范围。在 GitHub 设置 → 开发者设置 → 个人访问令牌 → 细粒度令牌中生成。

在 Claude Code 会话中验证连接：

```
/mcp
→ github: connected (42 tools available)
```

### /pr_comments 工作流

有了 GitHub MCP 连接，`/pr_comments` 命令就变得有用了。当你有一个带有审查者反馈的开放 pull request 时：

```
/pr_comments
```

Claude Code 读取当前分支，识别开放的 PR，通过 GitHub MCP 服务器获取所有审查评论，并按顺序处理每一个。它使用评论文本作为每个变更的规范，编辑相关文件，并报告更改了什么。

这个工作流替代了循环：在浏览器中阅读评论，切换到终端，向 Claude Code 描述变更，验证变更，重复。Claude Code 直接进行读取，并在自己的上下文中维护待处理评论的列表。

### 与 OpenTalon 的关联

在 OpenTalon 中，GitHub MCP 从第 7 章开始配置。CLI、Web 和 API 仓库都在 GitHub 上。当第 17 章的 CI 流水线运行时，Claude Code 将通过 GitHub MCP 读取 CI 失败日志，而不是需要手动复制粘贴。当第 16 章代码审查反馈到来时，Claude Code 将通过 `/pr_comments` 处理。该连接配置一次，在用户级别，并持久存在于所有会话和项目中。
