## 章节 5.4：编写 /api-endpoint 技能

`/opentaion-component` 技能处理 CLI 脚手架。`/api-endpoint` 技能处理另一侧：给定一个路由器名称和一个 HTTP 方法，它搭建一个带有 Pydantic 请求和响应模型、一个异步处理器和一个 pytest 测试的 FastAPI 路由。模式与上一节相同——先读后写，生成脚手架但不生成实现，让用户处于清晰的起点——应用于不同的领域。

一个设计决策使这个技能与组件技能区分开来：它没有 Bash 访问权限。组件技能需要 Bash 来运行测试并确认导入成功。API 端点技能只读写文件。运行数据库迁移、重启服务器或运行集成测试是有副作用的操作，应该有意为之，而不是在脚手架期间自动完成。`allowed-tools` 列表使这个约束显式化。

### 技能做什么

给定一个像 `/api-endpoint usage GET` 这样的调用，该技能会：

1. 读取 `api/src/opentaion_api/routers/` 中现有的路由器结构，以理解当前的模式
2. 创建或追加到适当的路由器文件中，添加遵循已建立 FastAPI 约定的新路由
3. 为新端点创建 Pydantic 请求和响应模型
4. 在 `api/tests/` 中创建测试存根，导入路由器并测试路由存在
5. 报告创建了什么，不运行任何东西

该技能不会启动服务器、运行迁移或执行测试。这些是开发者的下一步。

### 完整的技能文件

在 `.claude/skills/api-endpoint/SKILL.md` 创建文件：

```markdown
---
name: api-endpoint
description: 用 Pydantic 模型和测试存根搭建 FastAPI 路由
argument-hint: <router-name> <HTTP-method>
allowed-tools:
  - Read
  - Glob
  - Write
---

# Skill: api-endpoint

为 OpenTalon API 搭建一个新的 FastAPI 路由。

## 参数

`$ARGUMENTS` — 两个词：路由器名称和 HTTP 方法
  示例：`usage GET`、`keys POST`、`proxy POST`

## 过程

1. **读取现有路由器结构**
   - 使用 Glob 查看 `api/src/opentaion_api/routers/*.py` 以了解现有路由器
   - 读取与第一个参数匹配的路由器文件（如果存在）
   - 如果不存在，读取一个现有路由器以理解模式

2. **解析参数**
   - 路由器名称：`$ARGUMENTS` 的第一个词（例如 `usage`）
   - HTTP 方法：`$ARGUMENTS` 的第二个词（例如 `GET`）
   - 推导端点路径：`/{router-name}`（例如 `/usage`）

3. **创建或更新路由器文件**
   写入或追加到 `api/src/opentaion_api/routers/{router_name}.py`：

   ```python
   # api/src/opentaion_api/routers/{router_name}.py
   from fastapi import APIRouter, Depends
   from pydantic import BaseModel
   from ..auth import get_current_user

   router = APIRouter(prefix="/{router_name}", tags=["{router_name}"])

   class {RouterName}Response(BaseModel):
       # TODO: 定义响应字段
       pass

   @router.{method}("/")
   async def handle_{router_name}_{method_lower}(
       current_user = Depends(get_current_user),
   ) -> {RouterName}Response:
       """[此端点的简短描述]"""
       raise NotImplementedError("Endpoint not yet implemented")
   ```

4. **创建测试存根**
   写入 `api/tests/test_{router_name}.py`：

   ```python
   # api/tests/test_{router_name}.py
   from fastapi.testclient import TestClient
   from opentaion_api.main import app

   client = TestClient(app)

   def test_{router_name}_{method_lower}_exists():
       # 此测试验证路由已注册，而非已实现
       response = client.{method_lower}("/{router_name}/")
       # 401 表示路由存在但需要认证——在此阶段是预期的
       assert response.status_code in [200, 401, 422]
   ```

5. **报告**
   告诉用户：
   - 创建或修改了哪些文件
   - 脚手架构建的路由：`{METHOD} /{router_name}/`
   - 下一步：实现处理器、定义 Pydantic 模型、必要时运行迁移

## 备注
- `{RouterName}` 是路由器名称的骆驼命名法（CamelCase）版本
- `{method_lower}` 是 HTTP 方法的小写形式（get、post、delete）
- 不要运行服务器、测试或任何 Bash 命令——由开发者运行
- 如果路由器文件已经存在，添加新路由而不触碰现有路由
```

### 关键设计决策

`allowed-tools` 限制为 `[Read, Glob, Write]` 是一个有意的约束。一个也有 Bash 访问权限的脚手架技能可能会决定运行 `alembic upgrade head` 或 `uvicorn main:app` 作为其过程的一部分。这些是数据库迁移和服务器重启——有真实后果的操作，可能因与所搭建路由无关的原因而失败，而且开发者需要决定何时运行。通过明确排除 Bash，该技能的影响范围仅限于文件创建。开发者保留对涉及运行系统的所有内容的控制。

这是来自第 3 章的原则应用于技能设计：风险层级从 Read（无副作用）经过 Write（创建文件）到 Bash（执行任意命令）。使用任务所需的最小访问级别。

### OpenTalon 连接

在 OpenTalon 中，`/api-endpoint` 是加速第 15 和 16 章（API 逐条路由构建）的工具。代理端点、用量端点和 API 密钥管理端点都是一行调用后加实现的。没有这个技能，Claude Code 每次添加路由时都需要重新发明路由结构。有了这个技能，路由结构由脚手架固定，实现是唯一的变量。该技能尚未使用——API 目录还不存在——但它已经就位在 `.claude/skills/api-endpoint/SKILL.md`，等待构建开始。
