## 章节 5.5：/effort 命令：调整思考深度

前两节中的两个技能现在已经就位：`/opentaion-component` 和 `/api-endpoint` 存在于 `.claude/skills/` 中，可供第三部分开始的构建使用。在结束第 5 章之前，还有第三个工具对每个会话都很重要：`/effort` 命令，它控制 Claude Code 应用于每个任务的推理量。

扩展思考（extended thinking）不是一个二进制的开关。Claude Code 支持的模型可以在生成响应之前执行不同数量的内部推理——这个推理对用户是不可见的，但它的效果不是。以最小思考完成的任务和以密集思考完成的任务可以产生实质上不同的输出。`/effort` 命令让你有意识地将每个任务放置在这个频谱上。

### /effort 控制什么

`/effort` 命令设置思考 Token 预算：Claude Code 在生成响应之前可以用于内部推理的最大 Token 数。预算在推理步骤中被消耗；输出 Token（你看到的）在它之外。

四个级别：

**`/effort low`**——大约 4,000 个思考 Token。快速。适用于样板生成、格式化修复、重命名以及其他输出由机械方式决定的任务。一个有一个明确正确答案的任务不需要扩展推理。

**`/effort medium`**——大约 10,000 个思考 Token。大多数实现工作的平衡默认值。适用于根据规格说明实现功能、为现有代码编写测试、或进行有明确诊断的针对性 Bug 修复。

**`/effort high`**——大约 20,000 个思考 Token。适用于根因不明显的调试任务、涉及多个相互关联文件的重构、或智能体需要考虑几种替代方案然后才决定采用哪一种的任务。

**`/effort max`**——大约 31,000 个思考 Token。保留用于难以逆转且错误后果严重的设计决策：编写 SPEC.md、设计 API 模式、做出将影响数十个后续决策的架构选择。成本是同一任务的低思考水平的 3–5 倍。

### 默认行为

Claude Code 在支持扩展思考的模型上默认使用扩展思考。默认值大约为中等——足够的推理来处理大多数任务，又不至于每个会话都在日常工作上耗尽预算。`/effort` 从该默认值向上或向下调整预算。你不需要在每个会话上设置它；你在知道特定任务需要不同级别时才设置。

### 在 OpenTalon 中何时使用每个级别

在 OpenTalon 开发中，每个努力级别在构建序列中都有其自然位置：

**低**——运行 `/opentaion-component display` 来搭建新文件。结构由技能定义。不需要推理来改变输出。在脚手架任务之前设置 `/effort low`。

**中**——在第 9 章实现 `run_loop()` 函数。规格说明存在（来自第 9.5 节的 SPEC.md），测试存在（来自第 10 章），任务是将测试变绿。从清晰的规格说明进行实现是中等工作。

**高**——在第 12 章调试上下文溢出 Bug，其中智能体在 40 轮后失去连贯性。根因不是立即明显的，需要测试几个假设，修复可能不平凡。高努力是合适的。

**最高**——在第 9.3 节编写 OpenTalon SPEC.md，或在第 15.4 节设计代理端点模式。这些是将在本书其余部分被引用的决策。第一次就做对的成本低于在其上构建了二十个下游章节后进行修改的成本。

### 成本含义

最高思考的 3–5 倍成本乘数不是回避它的理由——而是有意使用它的理由。一个在架构决策时以最高努力运行、然后为结果脚手架任务切换到低努力的会话，将推理预算花在回报最高的地方。一个在一切任务上都保持最高努力的会话，是在为常规实现支付架构推理的价格，这是浪费。

`/cost` 命令实时显示当前会话的 Token 消耗。在使用 `/effort max` 进行 SPEC.md 会话之后，检查 `/cost`。这个数字会很有启发性。它还将帮助你校准什么时候最高思考是合理的，而什么时候高级思考就足够了。
