## 章节 5.3：编写 /opentaion-component 技能

前一节描述了技能是什么以及 Claude Code 如何发现它们。本节构建一个。`/opentaion-component` 技能解决了 CLI 开发中一个特定的、重复出现的问题：当你向 CLI 添加新命令时，每次都发生相同的操作——在正确的目录中创建新文件、给出正确的导入、用正确的 Click 装饰器定义命令、创建匹配的测试存根。没有技能，Claude Code 每次都会以不同方式执行此操作，产生略微不一致的结构，累积成维护问题。有了技能，过程被编码一次并一致地应用。

### 技能做什么

给定一个组件名称——比如 `/opentaion-component config`——该技能会：

1. 读取 `cli/src/opentaion/` 中现有的组件结构，以理解当前的模式
2. 创建 `cli/src/opentaion/config.py`，包含正确的导入、一个 Click 命令定义和一个存根实现
3. 创建 `cli/tests/test_config.py`，包含一个最小的 pytest 测试，导入新命令并验证它存在
4. 运行测试以确认它以预期的方式失败（导入成功，行为尚未实现——TDD 的 RED 状态）
5. 报告创建了什么以及下一步是什么

该技能不实现功能。它创建脚手架，使得实现该功能成为下一个自然的任务。

### 完整的技能文件

在 `.claude/skills/opentaion-component/SKILL.md` 创建文件：

```markdown
---
name: opentaion-component
description: 用测试存根搭建新的 OpenTalon CLI 命令
argument-hint: <component-name>
allowed-tools:
  - Read
  - Glob
  - Write
  - Bash
---

# Skill: opentaion-component

为 OpenTalon 智能体搭建一个新的 CLI 命令。

## 参数

`$ARGUMENTS` — 蛇形命名法（snake_case）的组件名（例如 `config`、`usage_report`）

## 过程

1. **读取现有结构**
   - 使用 Glob 查看 `cli/src/opentaion/*.py` 以了解现有命令文件
   - 读取一个现有命令文件以理解导入和装饰器模式

2. **创建命令文件**
   写入 `cli/src/opentaion/{$ARGUMENTS}.py`：

   ```python
   # cli/src/opentaion/{component_name}.py
   import click
   from rich.console import Console

   console = Console()

   @click.command()
   @click.argument("input", required=False)
   def {component_name}(input: str | None) -> None:
       """[此命令功能的简短描述]"""
       console.print(f"[yellow]{component_name}:[/yellow] not yet implemented")
   ```

3. **创建测试存根**
   写入 `cli/tests/test_{$ARGUMENTS}.py`：

   ```python
   # cli/tests/test_{component_name}.py
   from opentaion.{component_name} import {component_name}
   from click.testing import CliRunner

   def test_{component_name}_exists():
       runner = CliRunner()
       result = runner.invoke({component_name}, [])
       assert result.exit_code == 0
   ```

4. **运行测试**
   ```bash
   cd cli && uv run pytest tests/test_{$ARGUMENTS}.py -v
   ```
   确认：导入成功且测试通过（存根是功能性的）。

5. **报告**
   告诉用户：
   - 创建了哪些文件
   - 测试结果
   - 下一步：实现 `{component_name}()` 以替换存根

## 备注
- 将全文中的 `{component_name}` 替换为实际的 `$ARGUMENTS` 值
- 不要实现功能——只搭建结构
- 如果 `$ARGUMENTS` 包含连字符，转换为下划线以作为 Python 标识符
```

### 为什么这个技能存在

考虑没有这个技能会发生什么：每次你让 Claude Code"向 CLI 添加一个 config 命令"，它都会重新决定如何结构化文件、使用什么导入、是否创建测试、测试放在哪里以及测试应该验证什么。输出各不相同。十个命令之后，代码库有十种略微不同的风格——有些有测试，有些没有，有些导入了 Rich，有些没有。

该技能一次性编码了这些决策。读取步骤（步骤 1）意味着该技能始终匹配代码库中已有的任何模式，即使该模式有演变。如果你在第 9 章更改了命令的结构方式，该技能在下次调用时会自动采用新结构，因为它在写入之前先读取。

### `$ARGUMENTS` 变量

当用户调用 `/opentaion-component config` 时，字符串 `config` 在技能内部作为 `$ARGUMENTS` 可用。该技能可以在对 Claude Code 的散文指令中引用这个变量，Claude Code 会解释它并替换为实际值。这就是技能如何变得参数化：YAML 的 `argument-hint` 声明期望的形式，而技能主体中的 `$ARGUMENTS` 接收实际的调用值。

### OpenTalon 连接

在 OpenTalon 中，这个技能是从第 9 章开始实现 CLI 命令时将使用的组件创建工作流的第一个部分。它还没有做任何可见的事情——CLI 目录结构还不存在。但技能文件本身存在于 `.claude/skills/opentaion-component/SKILL.md`，并且当 CLI 实现开始时它会在那里。这是来自第 2 章的原则的应用：工具先于工作。现在编写该技能意味着当我们到达实现章节时，脚手架过程已经定义好了且保持一致。
