## 章节 5.2：技能系统：YAML 前置元数据和作用域

内置命令涵盖了会话管理、模型控制和工作流原语。它们不能被扩展。当你需要一个对你的项目做特定事情的命令——在正确的位置搭建新的 CLI 命令、遵循正确的模式、运行正确的测试——那就是一个技能（skill）。

技能是具有 YAML 前置元数据的 Markdown 文件，Claude Code 发现它并作为斜杠命令提供。其机制并不复杂。唯一最重要的实现细节是目录结构——在写一行代码之前要精确地理解它。

### 能工作的结构

一个名为 `my-skill` 的技能必须创建在以下路径：

```
.claude/skills/my-skill/SKILL.md
```

也就是说：在 `.claude/skills/` 内部一个以技能命名的子目录，其中包含一个精确命名为 `SKILL.md` 的文件。当 Claude Code 在启动时扫描技能时，它专门寻找这种模式。该技能以 `/my-skill` 的形式可用。

### 静默失败的结构

以下路径看起来合理：

```
.claude/skills/my-skill.md
```

它在调用时产生"Unknown skill: my-skill"错误，没有任何额外的解释。启动时没有错误，没有警告说文件被发现但格式错误。文件存在，Claude Code 忽略它，命令什么也不做。这是最常见的技能设置错误，而且正因为它直到调用时才无声失败，所以极难诊断。

如果你遇到一个"Unknown skill"错误，而你确信已经创建了该技能，请先检查目录结构。带有内部 `SKILL.md` 的子目录不是可选的。

支持性文件的好处是接受这种结构的一个理由：因为技能存在于一个目录中而不是单个文件中，你可以在 `SKILL.md` 旁边包含模板、参考文档、示例输入和脚本。技能在执行期间可以使用相对路径读取它们。一个搭建多个文件的脚手架技能可以将那些文件模板包含在自己的目录中，而不是将它们嵌入为 SKILL.md 散文中 heredoc 字符串。

### YAML 前置元数据

一个 `SKILL.md` 文件以 `---` 分隔符之间的 YAML 前置元数据开始：

```yaml
---
name: opentaion-component
description: 按照 OpenTalon 约定搭建一个新的 CLI 命令及测试
argument-hint: <component-name>
allowed-tools:
  - Read
  - Write
  - Bash
---
```

**`name`** — 技能的标识符。必须与目录名匹配。这是 Tab 补全中显示的内容以及你在 `/` 后输入的内容。

**`description`** — 在帮助显示和 Tab 补全预览中显示。这个字符串也会消耗上下文预算——所有已发现技能的描述大约占用上下文窗口的 2%。保持描述精确在二十字以内。"搭建一个新的 CLI 命令及测试"是好的。"这个技能帮助你按照所有已建立的模式和约定，在 OpenTalon 项目中搭建新的 CLI 组件，包括测试文件"则不是。

**`argument-hint`** — 在 Tab 补全中显示的参数语法。尖括号表示必需参数（`<component-name>`）。方括号表示可选（`[--dry-run]`）。这只是文档——技能主体决定对参数做什么。

**`allowed-tools`** — 明确限制技能可以使用的工具。如果省略，技能继承会话的完整工具集。这是一个安全性和可预测性控制：一个只应该读写文件的脚手架技能应该声明 `[Read, Write]` 而不是 `[Bash]`，这样它就不能意外地运行 Shell 命令。第 8 章从安全角度再次讨论这个问题。

**`model`** — 可选。为此技能的执行覆盖会话模型。当一个技能需要比默认会话模型更重的推理（设置为 `claude-opus-4-6`）或更快的执行（设置为 `claude-haiku-4-5-20251001`）时使用。大多数技能不应该设置这个，而是继承会话的设置。

### 三个作用域级别

技能从三个位置被发现，按顺序：

**项目作用域** — 项目根目录下的 `.claude/skills/`。这些技能提交到仓库中并在团队中共享（或在 OpenTalon 的情况下跨越所有写书会话）。你现在正在使用的 `write-section`、`session-wrap` 和 `autonomous-write` 技能就是项目作用域的。

**用户作用域** — 用户主目录中的 `~/.claude/skills/`。这些技能是个人化的——在你打开的每个项目中可用，但不通过 Git 共享。如果你有一个适用于所有项目的通用脚手架技能，它放在这里。

**子目录作用域** — 单仓库子目录内的 `.claude/skills/` 目录（例如 `cli/.claude/skills/`）。这些技能仅当 Claude Code 在该子目录内工作时可用。对于那些过于组件特定、在处理不同组件时甚至不应出现在 Tab 补全中的技能，这很有用。

Claude Code 在启动时扫描所有三个级别。如果同一个技能名称存在于多个级别，最具体的级别获胜（子目录 > 项目 > 用户）。

### 旧版格式

Claude Code 也支持在 `.claude/commands/` 中定义为扁平的 Markdown 文件的技能。这是原始格式。两种格式都可用。如果你为 OpenTalon 使用 BMAD V6（在第 4 部分中介绍），其智能体定义使用 `.claude/commands/` 格式——这没问题，也是有意的。`.claude/skills/<name>/SKILL.md` 子目录格式是新技能使用的现代方法。

### OpenTalon 连接

在 OpenTalon 中，三个现有技能（`write-section`、`session-wrap`、`autonomous-write`）展示了上述所有原则。它们位于 `.claude/skills/<name>/SKILL.md`，有精确的描述，而 `autonomous-write` 技能故意不包含 `allowed-tools` 限制，因为它需要完整的工具集来编写章节、读取文件和更新进度。我们接下来要编写的两个技能——`/opentaion-component` 和 `/api-endpoint`——将使用显式的 `allowed-tools` 限制，因为它们的范围更窄。这种设计是有意的：为任务使用最小的工具集，并明确声明它，使约束可见。

接下来两节从零开始构建这些技能。这里建立的模式——目录结构、精确的前置元数据、最小的工具作用域——适用于两者。
