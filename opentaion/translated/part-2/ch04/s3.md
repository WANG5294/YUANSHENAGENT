## 章节 4.3：单仓库策略：一个项目三个 CLAUDE.md 文件

层次化加载系统的存在是为了解决一个特定问题：一个项目不是统一的。适用于 Python 开发的规则不适用于 TypeScript 开发。FastAPI 路由处理器的约定与 Click 命令定义毫无关系。当一个单一的 CLAUDE.md 试图覆盖所有领域时，它要么覆盖得很差（对每个领域的需求来说都太单薄），要么产生一个不可读的文档，在第一次工具调用之前就消耗完了上下文预算。

OpenTalon 有三个组件：一个 Python CLI、一个 React Web 仪表盘和一个 FastAPI 后端。它们共享一个项目标识、一个部署故事和彼此之间的关系——但它们的实现约定是完全独立的。正确的结构是三个 CLAUDE.md 文件，处于一个有意的层级中。

### 根 CLAUDE.md：什么是通用的

根 `CLAUDE.md`（你在第 2 章编写的）处理对项目整体成立的一切，无论你在处理哪个组件：

- **项目标识**：OpenTalon 是什么、它解决什么问题、书和软件如何交织在一起
- **共享技术栈**：已批准库的完整列表以及不能偏离的规则
- **会话前检查清单**：Claude Code 在编写任何东西之前运行的五个步骤协议
- **通用规则**："不要发明尚未构建的代码"；"不要跳过写作前检查清单"
- **里程碑地图**：哪一章产出什么可交付成果，以便上下文始终保持接地

根 CLAUDE.md 不做的，是指定 Python 格式化偏好、TypeScript 严格性设置或 FastAPI 端点命名约定。那些放在组件文件中——因为它们只有在处理该组件时才相关。

### cli/CLAUDE.md：Python 领域

当 Claude Code 打开 `cli/` 子目录时，它加载根 CLAUDE.md 并在之上添加 CLI 规则。`cli/CLAUDE.md` 覆盖：

```markdown
# CLI — 组件规则

## 环境
- Python 3.12+。包管理器：uv。绝不要直接使用 pip。
- 运行命令：智能体循环使用 `uv run python -m opentaion`
- 运行测试：`uv run pytest tests/` —— 始终从 cli/ 目录执行

## Click 约定
- 命令使用 @click.command() 和 @click.option() 装饰器
- 命令分组在 `cli = click.group()` 下
- 通过 Click 上下文（ctx.obj）传递配置，而非全局状态

## Rich 显示
- 控制台输出通过 `rich.console.Console()` 进行
- 状态微调器：`with console.status("[bold]Working..."):`
- 错误输出：`console.print("[red]Error:[/red] ...", err=True)`

## 反模式
- 绝不从 web/ 或 api/ 导入——CLI 是独立工具
- 绝不使用同步 httpx——所有 HTTP 调用使用 async/await
- 绝不硬编码 API 端点——从配置读取
```

这个文件简短、具体且可操作。每条规则都有一个直接的行为后果。当 Claude Code 编写 Click 命令时，它拥有所需的 Python 约定，同时不会看到它不需要的 Vite 构建配置。

### web/CLAUDE.md：TypeScript 领域

`web/CLAUDE.md` 承载前端规则：

```markdown
# Web — 组件规则

## 环境
- Vite 5.x + React 18 + TypeScript（严格模式）
- 运行：`npm run dev` —— 开发服务器
- 构建：`npm run build` —— 生产构建
- 需要 Node.js 20+

## 组件约定
- 函数组件配合 TypeScript 类型——无类组件
- 状态：useState 用于局部状态，无 Redux 或 Zustand
- 样式：仅使用 Tailwind 工具类——无自定义 CSS 文件
- 图表：所有数据可视化使用 Recharts 库

## 文件结构
- src/components/ —— 可复用组件
- src/pages/ —— 顶层路由组件（Login、Dashboard）
- 类型在使用它们的文件中定义，不在 types/ 目录中

## 反模式
- 绝不使用内联样式——仅使用 Tailwind
- 绝不在 web 代码中访问 api/ 或 cli/ 目录
- 绝不添加 npm 包而不先检查 Tailwind 或 Recharts 是否已满足需求
```

同样简短、具体、可操作。没有 TypeScript 教程。没有解释 Tailwind 是什么。

### api/CLAUDE.md：FastAPI 领域

```markdown
# API — 组件规则

## 环境
- Python 3.12+ / FastAPI 0.110+
- 运行：`uv run uvicorn main:app --reload`
- 数据库：Supabase PostgreSQL（异步 SQLAlchemy）

## 路由约定
- 所有路由返回 Pydantic 模型，而非字典
- 端点名称：名词优先、连字符连接（`/api/usage-records`）
- 认证：通过 `Depends(get_current_user)` 提取 Token——不手动读取头部

## SQLAlchemy 模式
- 所有数据库操作使用异步会话：`async with AsyncSession() as session:`
- 模型放在 models/ 目录中，每个资源一个文件

## 反模式
- 绝不提交数据库连接字符串——使用环境变量
- 绝不编写原生 SQL——使用 SQLAlchemy 查询 API
- 绝不返回包含错误体的 200 状态码——正确使用 HTTP 状态码
```

### 实践中的继承模型

当 Claude Code 正在修复 `api/routes/auth.py` 中的一个 Bug 时，其有效上下文包括：

1. 根 CLAUDE.md —— 项目标识、共享技术栈、通用规则
2. `api/CLAUDE.md` —— FastAPI 约定、SQLAlchemy 模式、反模式

它不会加载 `cli/CLAUDE.md` 或 `web/CLAUDE.md`。它们是不相关的，会是噪音。加载层级将上下文过滤为与当前工作位置相关的内容。

这不是一个次要的便利。不相关的上下文比缺失的上下文更糟糕，因为它可能产生不相关的建议。一个在编写 TypeScript 时加载 Python 约定的 CLAUDE.md，是一个可能产生 Python 风格 TypeScript 的 CLAUDE.md。子目录分离防止了交叉污染。

实用规则：当一个子目录拥有根文件无法在不变得臃肿的情况下覆盖的真实、领域特定的约束时，创建一个组件 CLAUDE.md。不要为每个子目录都创建 CLAUDE.md——`api/` 内部的 `models/CLAUDE.md` 几乎肯定是不必要的。OpenTalon 的三个组件（CLI、Web、API）是恰当的粒度。

### OpenTalon 连接

在 OpenTalon 中，这三个组件 CLAUDE.md 文件尚不存在——代码库本身也还不存在。它们将在第 4.5 节的里程碑 M3 工作中被创建，届时我们将建立完整的生产级 CLAUDE.md 结构。在编写这些文件之前，接下来两节将涵盖包含什么和排除什么，以及使 CLAUDE.md 文件失败的反模式。这些原则在我们实际编写文件时指导其内容——所以顺序很重要。
