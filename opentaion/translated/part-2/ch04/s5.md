## 章节 4.5：反模式：不可维护的 CLAUDE.md

前三节构建了一幅好的 CLAUDE.md 看起来是什么样子的图景：层次结构化、简短、具体、可操作。本节从另一个方向——从失败模式——来探讨同一个问题。四种反模式占据了实践者遇到的绝大多数 CLAUDE.md 功能失调。了解它们可以保护你不至于重蹈覆辙。

### 反模式 1：垃圾场

一个从 80 行开始、在三个月内不受管理地增长到 600 行的 CLAUDE.md 是一个垃圾场。每次出问题时——Claude Code 建议错误的测试命令、推荐不在技术栈中的库、忘记命名约定——就有人添加一条规则。添加这条规则的感觉像是在改进。整体效果则不是。

问题不在于个别的添加。而在于缺乏删减。一个 CLAUDE.md 应该通过有意的决策来成长，而不是通过积累每次会话中的每一个教训。当你添加一条规则时，问问是否已有的规则已经覆盖了这种情况只是不够具体，或者根本问题在于规则没有被读到。一个超过 200 行而没有强有力理由的文件已经变成了垃圾场。修复方法不是添加另一条规则——是进行一次清理，删除所有代码本身已经传达的内容。

### 反模式 2：矛盾工厂

在不同时间由不同冲动添加的规则互相矛盾。六个月前的 CLAUDE.md 说"使用基于类的视图"。上个月添加的规则说"优先使用函数组件"。Claude Code 遵循哪一个？答案是不确定的——而智能体中的不确定行为意味着不可预测的输出。

矛盾之所以隐蔽，是因为它们看起来像全面性。一个从两面覆盖某个话题的文件（"所有 I/O 使用 async"和"同步 httpx 对于简单脚本是可以接受的"）看起来详尽。实际上，它给了 Claude Code 做任何一个的许可，这意味着它会在当下做上下文最方便的那个。

修复需要定期审查——不仅仅是添加规则，而是将文件作为一个连贯的文档来阅读。任何对另一条规则创建例外的规则都是删除或合并的候选。引导性问题：如果 Claude Code 只能遵循这两条规则中的一条，它应该遵循哪一条？删除另一条。

### 反模式 3：过时引用

CLAUDE.md 说"对于 API 设计，参见 @docs/api-contract.md。"但 `docs/api-contract.md` 在六周前的重构中被移到了 `api/specs/contract.md`。Claude Code 加载 CLAUDE.md，找到 @-引用，尝试加载文件，然后静默失败或什么都不加载。

过时引用比矛盾更难捕捉，因为它们需要检查文件系统，而不仅仅是阅读 CLAUDE.md。预防措施是将 CLAUDE.md 当作代码来对待：它需要定期通过"这个引用存在吗？"的检查。补救方法是定期运行 `grep "@" CLAUDE.md` 并验证每个被引用的文件仍然存在于指定的路径。这也是为什么第 6 节的钩子包括一个会话前检查——在会话开始前、而非任务中捕捉过时引用。

### 反模式 4：一厢情愿的指令

"编写安全、生产质量的代码。""遵循最佳实践。""在做出更改之前仔细思考。"这些指令感觉重要。它们没有行为后果。

如果 Claude Code 可以基于一条指令做出具体决策，那么它就是可操作化的。"不要对用户输入使用 `eval()` 或 `exec()`"是可操作化的。"编写安全的代码"则不是——它要求 Claude Code 推断"安全"是什么意思，而这种推断会因上下文而异。"在提交前运行 `uv run pytest tests/`"是可操作化的。"测试你的更改"则不是。

测试很简单：对于你 CLAUDE.md 中的任何规则，问"什么具体行动会违反这条规则？"如果你不能具体地回答这个问题，这条规则就是一厢情愿。要么让它具体，要么删除它。

### 维护协议

CLAUDE.md 是活文档，不是一次设置就永忘的配置文件。每当一次会话揭示智能体做出了你不会做的决策时，都应该对它进行审查——不是为了添加一条责怪智能体的规则，而是为了问现有的规则是否足够清晰、或者是否有规则已变得过时。将 CLAUDE.md 视为回归目标：如果 Claude Code 在新会话中的行为与 CLAUDE.md 指定的不匹配，要么是 CLAUDE.md 错了，要么是它不够具体。

在 OpenTalon 中，你在第 2 章编写的根 CLAUDE.md 被设计为一开始就是干净的。在添加组件文件之前，现在用反模式来审视它是值得的。是否有任何规则看起来像垃圾场式的添加？是否有任何规则与其他规则矛盾？是否有任何引用指向尚不存在的文件？你对 CLAUDE.md 做的第一次编辑，就是把它当作一个受维护的文档的开始。

### 里程碑 M3：生产级 CLAUDE.md 和单仓库结构

带着反模式在脑海中，里程碑是这样的：创建三个组件 CLAUDE.md 文件，保持每个简短，并验证它们一起构成了一个连贯、无矛盾的系统。

在你的 OpenTalon 仓库中创建以下三个文件：

```
# cli/CLAUDE.md
```

```markdown
# CLI — 组件规则

## 环境
- 包管理器：uv（绝不用 pip，绝不用 poetry）
- 运行智能体：`uv run python -m opentaion "你的提示"`
- 运行测试：`uv run pytest tests/ -v`
- Lint：`uv run ruff check . && uv run black --check .`

## Click 约定
- 命令使用 @click.command() 和 @click.option() 装饰器定义
- 分组在 `cli = click.group()` 下
- 通过 Click 上下文（ctx.obj）传递配置——而不是全局变量

## Rich 显示
- 所有控制台输出通过 `rich.console.Console()`
- 进度：`with console.status("[bold]Working..."):`
- 错误：`console.print("[red]Error:[/red] message", err=True)`

## 反模式（IMPORTANT）
- 绝不从 web/ 或 api/ 导入——CLI 是独立工具
- 绝不使用同步 httpx——所有 HTTP 调用使用 async/await
- 绝不要硬编码 API 基础 URL——从配置读取
```

```
# web/CLAUDE.md
```

```markdown
# Web — 组件规则

## 环境
- Node.js 20+。框架：Vite 5 + React 18 + TypeScript（严格模式）
- 开发服务器：`npm run dev`
- 构建：`npm run build`
- 测试：`npm run test`（vitest）

## 组件约定
- 函数组件配合显式 TypeScript prop 类型
- 局部状态：仅使用 useState——无外部状态管理
- 样式：仅使用 Tailwind 工具类——无独立 CSS 文件
- 图表：所有数据可视化使用 Recharts 库
- 无路由库——基于 Supabase 认证状态的条件渲染

## 反模式（IMPORTANT）
- 绝不使用内联样式——仅使用 Tailwind 类
- 绝不在 web 代码中访问 api/ 或 cli/
- 如果 Tailwind 或 Recharts 已满足需求，绝不添加 npm 包
```

```
# api/CLAUDE.md
```

```markdown
# API — 组件规则

## 环境
- 包管理器：uv
- 运行：`uv run uvicorn opentaion_api.main:app --reload`
- 运行测试：`uv run pytest tests/ -v`

## FastAPI 约定
- 所有路由返回带类型的 Pydantic 响应模型——绝不返回原生的字典
- 端点路径：连字符连接、名词优先（`/usage-records`、`/api-keys`）
- 认证：通过 `Depends(get_current_user)` 提取当前用户——不手动读取头部

## SQLAlchemy 约定
- 所有数据库操作使用异步会话
- 每个资源一个模型文件，位于 models/ 中
- 通过 Alembic 进行迁移——绝不手动修改数据库模式

## 反模式（IMPORTANT）
- 绝不提交连接字符串——使用环境变量
- 绝不编写原生 SQL——使用 SQLAlchemy 查询 API
- 绝不返回带有错误体的 HTTP 200——使用正确的状态码
```

有了这三个文件与根 CLAUDE.md 并存，单仓库结构就完整了。每个组件添加了自己的领域特定规则，而不重复根文件已经覆盖的内容。根文件指定技术栈；组件文件指定如何使用它。当 Claude Code 打开 `cli/` 时，它知道 Python/uv/Click 约定。当它打开 `web/` 时，它知道 React/TypeScript/Tailwind 约定。两个组件文件互不知晓对方。

现在通读全部四个 CLAUDE.md 文件——根文件加上三个组件文件——并根据你的开发环境进行任何调整。测试命令可能不同。Python 版本可能不同。这是你对 CLAUDE.md 作为受维护文档的第一次有意编辑。本节中的反模式是你添加之前要删除什么的指南。

### 刚刚发生了什么

本章以一个加载机制开始，以一个完整的四文件 CLAUDE.md 系统结束。机制解释了为什么结构有效。反模式解释了为什么维护不良的结构会失败。组件文件将原则付诸实践。

里程碑 M3 是生产级 CLAUDE.md：第 2 章中经过优化的根文件，加上覆盖完整单仓库的三个组件文件。从这一刻起，每一个打开 OpenTalon 的会话都拥有在当前相关组件中工作所需的上下文——而不加载不相关的规则、没有矛盾、没有过时引用。

下一章从被动的上下文转向主动的工具：技能系统，它让你直接将可复用程序嵌入到 Claude Code 的命令集中。
