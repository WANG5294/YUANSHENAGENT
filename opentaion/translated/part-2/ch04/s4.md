## 章节 4.4：包含什么和排除什么

每个与 CLAUDE.md 文件打过交道的经验丰富的开发者，都至少犯过同一个错误一次：添加一些东西是因为它感觉重要，而不是因为它可操作。文件变大了。会话感觉变慢了。智能体开始提出在技术上符合规则但在实践中错误的建议——因为它试图同时满足二十条指令，而最新的那条被其他十九条的噪音稀释了。

本节是纠正。CLAUDE.md 文件有大小纪律（size discipline）——200 行是软上限，保持低于这个上限的原因不是美学，而是性能。一个超过 200 行的 CLAUDE.md 是一个试图做太多事情的 CLAUDE.md。

### 200 行纪律

上下文是有限的。CLAUDE.md 中的每个 Token 都是一个不能用于代码、测试输出或工具调用结果的 Token。一个 400 行的 CLAUDE.md 在会话开始做任何工作之前就消耗了大约 2,000 Token。在会话尾声，当上下文已经积累到很复杂时，这 2,000 Token 可能成为智能体保持连贯性与丢失对早期决策的跟踪之间的分水岭。

200 行限制强制了一种纪律：每一行必须证明自己的地位。如果一条规则被删除而智能体仍然在 90% 的情况下正确行为，就删除它。10% 的情况不值得花费 20 行上下文成本。

### 什么属于 CLAUDE.md

**确切的命令。** 你能放进 CLAUDE.md 的最可操作的东西，就是构建、测试和运行项目所需的精确 Shell 命令。不是描述——是实际的命令。

```
Build: uv run python -m build
Test: uv run pytest tests/ -v
Lint: uv run ruff check . && uv run black --check .
Run: uv run python -m opentaion "在此输入提示"
```

这些不会很快过时。当它们确实变化时，文件很容易更新。它们还防止了最常见的失败模式：Claude Code 猜测错误的测试命令并运行错误的东西。

**工程原则。** 具有明确行为后果的规则。"函数不得超过 40 行——如果超出了就提取辅助函数。""所有 HTTP 调用都是异步的。""CLI 中无全局状态。"这些不是样式偏好——它们是影响项目中每个代码决策的架构约束。

**文件夹结构。** 简要描述什么东西在哪里。三到五行。足以让 Claude Code 将新文件放在正确的位置而无需猜测。不是每个文件的完整清单。

**命名约定。** 当名称遵循非自明的模式时。"API 路由处理器命名为 `handle_{resource}_{action}`。""测试文件镜像源文件结构：`tests/core/test_agent.py` 对应 `core/agent.py`。"那些如果被违反就会造成人类需要修复的不一致性的约定。

**关键规则标记为 IMPORTANT。** 当一条规则是关键的——其违反会导致安全问题、生产事故或根本性架构错误——明确标记它：

```
IMPORTANT: 绝不要将环境变量提交到仓库中。
所有秘密放在 .env（已 gitignore）中。API 密钥验证钩子
将阻止包含裸 API 密钥字符串的提交。
```

`IMPORTANT:` 标签不是装饰。Claude Code 会权衡它。标记为 IMPORTANT 的规则比埋在散文中规则获得更强的关注。

### 什么不属于 CLAUDE.md

**linter 已强制执行的代码风格规则。** 如果 `ruff` 被配置为强制执行 88 字符行限制，你不需要再说"保持行数在 88 字符以下"。工具自动强制执行规则。将其写在 CLAUDE.md 中会造成重复，并且最终会出现分歧——linter 被更新了，CLAUDE.md 没有，现在它们互相矛盾。

**关于技术如何工作的解释。** CLAUDE.md 不是给读者看的文档。它是给智能体的指令。"FastAPI 使用依赖注入"是 Claude Code 已经拥有的背景知识。"使用 `Depends(get_current_user)` 进行认证——绝不要手动读取 Authorization 头部"是一条指令。

**个性指令。** "在你的回答中保持简洁和专业。"Claude Code 已经有自己的语调。关于听起来怎么样的指令只是噪音，稀释了关于如何编码的指令的空间。

**每次会话都加载的全包式 @-引用。** "开始工作时加载 @docs/all-design-notes.md。"如果文档始终相关，其关键点应该属于 CLAUDE.md 本身。如果它不是始终相关，使用条件散文（参见第 4.2 节）。无条件地加载一个大型文档而它只是有时被需要，是一种没有保证回报的上下文税。

**一厢情愿的指令。** "始终编写生产质量、安全、经过良好测试的代码。"这不是指令。这是一种愿望。Claude Code 无法将其操作化。像"所有接受用户输入的函数必须在边界处进行验证——没有原始输入到达数据库查询"这样的具体规则是可操作化的。模糊的质量愿望则不是。

### 基准测试

在将任何东西添加到 CLAUDE.md 之前，问：一个从未见过这个项目的有能力的工程师是否需要知道这一点才能正确工作？如果是，它属于。如果一个有能力的工程师会通过阅读代码弄明白，它不属于。CLAUDE.md 填补了代码本身无法传达的空白——架构约束、工具选择、禁止模式以及操作项目所需的命令。

### OpenTalon 连接

在 OpenTalon 中，你在第 2 章编写的根 CLAUDE.md 就是按照这些原则设计的。会话前检查清单是可操作的。技术栈规范是具体的。"Claude Code 绝不能做什么"清单是一组行为约束，而不是愿望。当我们在第 4.5 节编写三个组件 CLAUDE.md 文件时，同样的纪律适用：每条规则应该通过基准测试。组件文件会很短——可能每个 40-60 行——因为它们只覆盖组件在根文件已提供的基础之上真正需要的东西。简短不是关注不足的迹象。简短就是目标。
