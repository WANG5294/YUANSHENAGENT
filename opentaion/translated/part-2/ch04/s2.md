## 章节 4.2：@-引用系统与条件加载

前一节描述了 Claude Code 如何基于目录层次自动加载 CLAUDE.md 文件。这种自动加载处理了最常见的情况：当你处理某个组件时始终适用的规则。但还有第二种加载机制——你从 CLAUDE.md 文件内部刻意触发的机制——它处理另一种情况：那些大型、专业且仅在某些时候相关的参考文档。

`@`-引用语法是你如何将一个文件包含到另一个文件中的方式。它是带有特定分隔符的文件包含。仅此而已。

### 基本 @-引用

在任何 CLAUDE.md 内部，像下面这样的一行会使该文件在 CLAUDE.md 被处理时被包含：

```
@docs/architecture.md
```

路径相对于仓库根目录。当 Claude Code 加载包含这一行的 CLAUDE.md 时，它会读取 `docs/architecture.md` 并将其内容插入到 `@` 引用的位置。结果与将 `architecture.md` 的内容直接粘贴到 CLAUDE.md 中完全相同——从上下文窗口的角度来看，它们是一个文件。

这有一个直接用途：将规则与参考材料分离开来。规则是简短、命令式的语句（"使用 uv，不要用 pip"）。参考材料是冗长、描述性的内容（一个有二十张表的数据模型、一个有四十个端点的 API 规范）。将它们混在一个 CLAUDE.md 中会使文件难以维护和阅读。@-引用让你可以将规则保留在 CLAUDE.md 中，将参考材料放在专用文件中，然后在相关的地方引用参考材料。

@-引用不是有条件的。如果该行存在，文件就会加载。总是、每个会话、从该 CLAUDE.md 被处理的那一刻起。这对于第 3.4 节的成本分析很重要：一个 @-引用的文件在每次会话中都会计入上下文预算，就像其内容被直接写入 CLAUDE.md 一样。导入一个 4,000 Token 的架构文档意味着每个会话开始时短了 4,000 Token。

### 条件加载：散文模式

替代方案是实践者称之为条件加载（conditional loading）的方式，尽管 Claude Code 没有内置的条件语法。其机制是散文指令，而不是代码。

在 CLAUDE.md 中，你可以写这样的东西：

```
## 当处理 API 时

当你在 api/ 中编辑文件时，在对任何路由处理器进行更改之前，
请先阅读 @api/specs/endpoint-reference.md。该文件包含当前的
API 契约。不要以会破坏它的方式修改端点。
```

`@api/specs/endpoint-reference.md` 引用位于散文中，而不是在顶层独立的行上。Claude Code 会看到这个指令，当处理 API 代码时，在被描述的条件触发时加载引用的文件。当处理 CLI 代码时，Claude Code 没有遇到要加载这个文件的指令，所以它不会。

这就是关键的区分：始终加载 vs. 按需加载。CLAUDE.md 顶层的一个 @-引用是始终加载的。嵌入在条件散文指令中的 @-引用是按需加载的——只有当描述的条件被满足时才加载。

成本差异迅速累积。假设 OpenTalon 的架构文档是 3,000 Token。如果它通过顶层的 @-引用始终加载，每个处理 CLI 的会话（与架构文档完全无关）从一开始就支付 3,000 Token。十个会话就是 30,000 Token 花在一个从未被需要的文档上。而有了条件加载——"当你即将做出架构更改时，请阅读 @docs/architecture.md"——这些 Token 只在相关时才被花费。

### 何时使用 @-引用

实用规则：对大型、稳定且始终相关的文档使用 @-引用。对大型、稳定但有时相关的文档使用条件散文。

"始终相关"包括项目技术栈规范、数据模型模式、以及如果你正在构建客户端时的 API 契约。这些文档被频繁引用——始终加载它们的成本低于重复按需加载它们的精力。

"有时相关"包括完整的测试策略文档、部署运行手册、第三方服务的 API 参考。这些在需要时至关重要，其他时候则无关紧要。条件散文让它们保持在上下文之外，直到工作特别需要它们。

要避免的一个模式：将一切作为风险对冲导入。一个 @-引用了十二个文档"以便 Claude Code 总是有上下文"的 CLAUDE.md，是一个牺牲了上下文预算而不确定获得了什么的 CLAUDE.md。导入要有目的性。如果你不能说出哪个特定的工具调用需要这个文档，就不要导入它。

### OpenTalon 连接

在 OpenTalon 中，`opentaion-state.md` 是不应该被 @-引用的最清晰的例子。它描述了代码库当前的确切状态——存在哪些文件、实现了哪些功能、上一个里程碑产出了什么。在编写涉及代码的章节时它是必不可少的。在编写没有里程碑的概念性章节时它是不相关的。`write-section` 技能在其程序的七个步骤中明确地读取它，正是因为这个明确的读取应该在技能运行时发生，而不是在会话开始时。如果 `opentaion-state.md` 被 @-引用到根 CLAUDE.md 中，它那 400+ Token 的内容会在每个会话中加载——包括那些没有编写任何代码的会话。条件模式（技能在需要时读取它）更便宜且同样可靠。
