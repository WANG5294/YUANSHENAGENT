## 章节 4.1：层次化加载：Claude Code 如何读取记忆

第一部分确认了 Claude Code 没有会话记忆——每个会话都是从空白开始的。它所拥有的是文件。你在第 2 章编写的 CLAUDE.md 文件是系统提供的最接近持久化记忆的东西。第二部分将精确考察这种记忆如何加载、它可以采用什么结构、以及制约它的约束是什么。本章从机制本身开始：Claude Code 如何读取 CLAUDE.md 文件、加载哪些文件、以及按什么顺序加载。

"层次化（hierarchical）"这个词是精确的。Claude Code 不是简单地找到并加载仓库中的每一个 CLAUDE.md。它按照定义的优先级顺序加载它们，并且子目录文件是附加式的（additive）——它们扩展而不是替换上层的文件。理解这个顺序是设计 OpenTalon 将使用的三文件 CLAUDE.md 结构的先决条件。

### 五个层级，一个顺序

Claude Code 在五个加载级别处理 CLAUDE.md 文件，从最高优先级到最低优先级：

**1. 托管策略（组织级别）。** 如果 Claude Code 在 Anthropic 组织账户内部署，管理员可以设置适用于所有用户的策略文件。这些无法被项目设置排除。使用个人订阅的独立开发者永远不会遇到这一级别。它的存在是为了支持需要安全或合规团队保证约束的企业部署。

**2. 用户级别（`~/.claude/CLAUDE.md`）。** 用户主目录中的单个文件。这是放置适用于所有项目的个人开发偏好设置的正确位置：偏好的编程语言习惯、提醒不要在代码注释中使用表情符号、关于本地环境特殊性的说明。它在每个项目、每个会话中加载。

**3. 项目根目录（`./CLAUDE.md`）。** 你在第 2 章编写的文件。这是主要的项目宪法——它描述项目是什么、技术栈是什么、以及制约它的规则是什么。它在 Claude Code 打开项目目录时加载。

**4. 子目录 CLAUDE.md 文件。** 如果 Claude Code 在 `cli/` 中工作，它会加载 `cli/CLAUDE.md` 作为根 CLAUDE.md 的补充。如果它移动到 `api/`，它会加载 `api/CLAUDE.md`。这些文件是领域特定的：它们携带仅在处理该组件时才适用的规则。它们不替换根文件。两者同时处于活动状态。

**5. `.claude/rules/` 拆分文件。** 对于根 CLAUDE.md 可能会变得难以控制地长的项目，Claude Code 支持将文件拆分为 `.claude/rules/` 中的多个文件。该目录中的文件与根 CLAUDE.md 一起加载。这是一个组织模式，而不是一个优先级级别——所有 `.claude/rules/` 文件都被视为项目根级别的一部分。

当文件冲突时，优先级顺序很重要。用户级别文件中的规则可以被项目根文件覆盖。项目根文件中的规则可以被子目录文件覆盖。最具体的文件获胜。

### "附加式"在实践中意味着什么

"附加式"这个词容易被误解为"累积到膨胀"。更好的思维模型是分层：每个 CLAUDE.md 层添加了领域特定的规则，而不使下面的层失效。

当 Claude Code 打开 OpenTalon 的 `cli/` 子目录时，其有效指令集是三个文件的并集：用户级别的 `~/.claude/CLAUDE.md`（如果你有的话）、项目根目录的 `CLAUDE.md` 和 `cli/CLAUDE.md`。所有三个都是活动的。cli/ 文件添加了 Python 特定的规则——"使用 uv，不要用 pip；偏好使用 Click 装饰器定义命令；使用 `uv run pytest` 运行测试"——而不需要重复根文件中的任何内容。根文件仍然说"技术栈是固定的"。cli/ 文件说"这是该技术栈的 CLI 组件需要的东西"。各层互相补充。

这对于像 OpenTalon 这样的单仓库（monorepo）很重要，其中三个组件（CLI、Web、API）共享一个项目标识，但有完全不同的工具约定。一个试图覆盖所有三个领域的单一根 CLAUDE.md 会冗长到不可读——而且更糟糕的是，即使你只处理 CLI，它也会加载所有三套规则，使上下文充满不相关的指令。子目录模式解决了这个问题：只加载与你当前工作位置相关的内容。

### @-引用语法

与层次化加载系统分开的是，Claude Code 支持使用 `@` 语法的显式文件引用。在任何 CLAUDE.md 中，像这样的一行：

```
@docs/architecture.md
```

会使 Claude Code 在处理该 CLAUDE.md 时将该文件的内容加载到上下文中。这是内联引用——被引用文件的内容出现在 `@` 引用的位置，就像它被写在那里一样。

与子目录加载的区别在于加载的时间点。子目录 CLAUDE.md 文件在你导航到该目录时自动加载。@-引用的文件在处理包含该引用的 CLAUDE.md 时加载。两者都在会话开始时发生，但触发条件不同。第 4.2 节将全面涵盖 @-引用系统，包括条件加载（conditional loading）的模式——告诉 Claude Code 仅在特定条件满足时才加载某个文件。

### claudeMdExcludes 设置

一个额外的控制项：项目 Claude Code 设置中的 `claudeMdExcludes` 字段接受 Glob 模式。匹配这些模式的文件在加载过程中会被跳过。当某个子目录的 CLAUDE.md 不应在特定会话中加载时，或者当外部包包含你希望抑制的 CLAUDE.md 文件时，这项设置很有用。该设置位于项目级别的 `.claude/settings.json` 中。

### OpenTalon 连接

在 OpenTalon 中，层次化加载系统是我们将在本章其余部分逐步构建的架构。你在第 2 章编写的根 `CLAUDE.md` 处理项目标识、共享技术栈约束和会话前检查清单。在第 4 章（当前章节）中，我们将添加 `cli/CLAUDE.md`、`web/CLAUDE.md` 和 `api/CLAUDE.md`——每个都包含其子目录所需的组件特定规则。当 Claude Code 构建 CLI 智能体时，它将拥有根规则加上 Python 特定规则。当它构建 Web 仪表盘时，它将拥有根规则加上 Vite/React/Tailwind 规则。没有组件的 CLAUDE.md 需要重复根文件已经说过的话。加载层级会自动完成组合。

每个文件应该包含什么——以及应该排除什么——是接下来几节要回答的问题。在这之前，第 4.2 节中的机制解释了如何从 CLAUDE.md 内部引用文件，而不是仅仅依赖基于目录的加载。
