## 第 6.3 节：PostToolUse 钩子：自动化代码质量

上一节介绍了强制执行：在事情发生之前将其阻止。PostToolUse 钩子在工具调用成功后运行。它们不阻止任何事情——它们做出反应。典型的用例是代码质量自动化：在任何 Python 文件被写入或编辑之后，自动运行格式化器和代码检查器（linter）。文件已经改变了；钩子在 Claude Code 继续下一步之前把它清理干净。

没有钩子时，会话依赖于 Claude Code 记得在文件编辑后运行格式化。有时它会记得。有时上下文压力导致它跳过这一步。有时它格式化了但使用了错误的工具。PostToolUse 钩子使格式化成为结构性的——它总是发生，通过钩子，无论 Claude Code 决定做什么。

### 保存时格式化模式

我们将要构建的钩子会在任何被写入或编辑的 Python 文件上运行 `ruff check --fix` 和 `black`。`ruff` 是 OpenTalon 的代码检查器（快速，与 pyproject.toml 配置兼容）。`black` 是格式化器。两者按顺序运行：ruff 先修复 lint 问题，然后 black 格式化结果。

钩子脚本通过 stdin 以 JSON 形式接收工具调用的输出，提取文件路径，并在该文件上运行质量工具：

```bash
#!/bin/bash
# .claude/hooks/format-python.sh
# Runs after any Write or Edit tool call on a .py file

INPUT=$(cat)
FILE_PATH=$(echo "$INPUT" | python3 -c "
import json, sys
d = json.load(sys.stdin)
# Write tool output has 'filePath', Edit tool has 'path'
print(d.get('filePath') or d.get('path') or '')
")

# Only run on Python files
if [[ "$FILE_PATH" != *.py ]]; then
    exit 0
fi

# Run from the project root (file paths are relative to project)
cd "$(git rev-parse --show-toplevel 2>/dev/null || pwd)"

# Run ruff and black; suppress output unless they fail
uv run ruff check --fix "$FILE_PATH" 2>&1 | grep -v "^Found\|^All checks"
uv run black "$FILE_PATH" --quiet 2>&1

exit 0
```

settings.json 配置：

```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Write|Edit",
        "hooks": [
          {
            "type": "command",
            "command": "bash .claude/hooks/format-python.sh"
          }
        ]
      }
    ]
  }
}
```

### 钩子接收的内容

PostToolUse 钩子接收一个 JSON 负载，包含工具名称、工具输入（传递给工具的内容）和工具输出（结果）。对于 Write 工具，相关字段是 `filePath`——写入文件的路径。对于 Edit 工具，是 `path`。钩子提取存在的那个字段。

路径提取使用一个小的 Python 一行命令来解析 stdin 中的 JSON。这是钩子脚本中的常见模式：负载是 JSON，Python 能可靠地解析它，然后脚本用提取的值继续执行。用 Bash 字符串解析 JSON 是脆弱的；对于任何非平凡的提取，请使用 Python。

### PostToolUseFailure 变体

`PostToolUseFailure` 在工具调用失败时触发——当 Bash 命令返回非零退出码时，当文件写入因权限错误而失败时，或当任何其他工具调用遇到错误时。钩子接收错误信息。

在 OpenTalon 中的实际用例是日志记录：当测试运行失败时，记录哪个测试失败了、何时失败以及错误是什么。这会创建一个持久的错误日志，超越会话的上下文而存在。Claude Code 可以在调试会话开始时读取该日志，以了解什么一直在失败以及失败的时间。具体实现属于第 10 章，届时测试套件已存在以供失败。

### 为什么这比要求 Claude Code 格式化更好

有一种方案是在每个 CLAUDE.md 中写上"在编辑 Python 文件后总是运行 black 和 ruff"。Claude Code 以中等的稳定性遵循这个指令。在 80% 的准确率下，20% 的文件编辑会留下未格式化的代码。这些编辑累积起来。任何代码审查的 diff 都会包含无关的格式化噪音。CI 的代码检查会失败。开发者手动清理。

而在 100% 的准确率下——即钩子提供的——这些都不会发生。钩子不会被复杂的 bug 修复分心而忘记格式化。它在每次匹配的工具调用上运行，每次。代价是每次文件写入花费不到一秒的时间和零开发者的注意力。

### 与 OpenTalon 的关联

在 OpenTalon 中，这个钩子是 CLI 和 API 组件的自动质量层。当 Claude Code 在 `api/src/opentaion_api/routers/proxy.py` 中编写一个新的路由处理器时，钩子立即触发，ruff 捕获任何导入顺序或未使用变量的问题，black 规范化格式。仓库中累积的代码从第一次提交开始就是一致格式化的。在第 17 章，当 CI 流水线运行自己的代码检查时，它将通过——不是因为我们记得去格式化，而是因为钩子让忘记变得不可能。
