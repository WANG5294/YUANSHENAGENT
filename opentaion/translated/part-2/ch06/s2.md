## 第 6.2 节：PreToolUse 钩子：强制执行约束

PreToolUse 钩子是一道闸门，立于工具调用请求与其执行之间。当 Claude Code 决定运行一个 Bash 命令时，钩子在命令运行之前触发。钩子检查该命令。它返回一个决策。如果决策是"阻止"，该命令永远不会执行。这是结构性的强制执行——不是 Claude Code 可能会忘记的指南，而是一道它无法通过的闸门。

这种区别至关重要。CLAUDE.md 中的规则是 Claude Code 读取并以不同精确度遵循的指令。而 PreToolUse 钩子是一种检查，无论 CLAUDE.md 中写了什么，无论会话的对话上下文是什么，无论自上次读取 CLAUDE.md 以来累积了多少次工具调用，它都会运行。这道闸门不会疲倦。

### 三种钩子响应

PreToolUse 钩子通过 stdin 以 JSON 格式接收工具名称及其输入。它返回一个 JSON 响应，有三种形态：

**批准（Approve）** —— 工具调用不变地继续执行：
```json
{"decision": "approve"}
```

**阻止（Block）** —— 工具调用被取消，并向 Claude Code 返回错误：
```json
{"decision": "block", "reason": "Bash command contains rm -rf outside allowed directories"}
```

Claude Code 收到原因，将其纳入推理过程，并必须决定如何继续——通常会尝试以其他方式完成任务或询问用户。

**修改（Modify）** —— 工具调用继续执行，但输入已被修改：
```json
{"updatedInput": {"command": "ls -la /safe/directory"}}
```

Modify 是最灵活的响应，也是最容易因粗心使用而带来危险的响应。一个静默重写 Bash 命令的钩子，是一个会产生意外行为的钩子。请谨慎使用 modify，并且只在转换是明确无歧义的情况下使用。

### 匹配器语法

钩子配置中的 `matcher` 字段决定哪些工具调用会触发该钩子。常见模式：

- `"Bash"` —— 匹配所有 Bash 工具调用
- `"Write|Edit"` —— 匹配 Write 和 Edit 工具调用（竖线分隔符是正则表达式中的"或"）
- `"Bash(rm -rf*)"` —— 匹配命令匹配模式 `rm -rf*` 的 Bash 调用
- `".*"` —— 匹配所有工具调用（很少有用，钩子开销很大）

匹配器针对工具名称进行评估，对于 Bash，则针对括号中的命令字符串进行评估。使用覆盖你约束条件的最具体的匹配器。

### 一个具体示例：阻止危险的 Shell 命令

以下钩子阻止任何包含 `rm -rf` 的 Bash 命令，当它会影响已定义安全区域之外的目录时：

```bash
#!/bin/bash
# .claude/hooks/block-dangerous-rm.sh
# Receives tool call JSON via stdin, returns decision

INPUT=$(cat)
COMMAND=$(echo "$INPUT" | python3 -c "import json,sys; d=json.load(sys.stdin); print(d.get('command',''))")

# Block rm -rf on project root or home directory
if echo "$COMMAND" | grep -qE "rm -rf (/|~|\.\./)"; then
    echo '{"decision":"block","reason":"rm -rf targeting root, home, or parent directory is blocked. Use specific paths within the project."}'
    exit 0
fi

# Allow everything else
echo '{"decision":"approve"}'
```

对应的 settings.json 配置项：

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "bash .claude/hooks/block-dangerous-rm.sh"
          }
        ]
      }
    ]
  }
}
```

当 Claude Code 尝试一个匹配危险 rm 模式的 Bash 调用时，钩子触发，返回带有原因的阻止决策，命令不会执行。Claude Code 看到这个原因，必须用其他方式解决问题。

### Claude Code 如何处理阻止事件

当 PreToolUse 钩子阻止了一个工具调用时，Claude Code 不会停止。它会收到阻止原因，将其纳入上下文，并且通常会：

1. 重新考虑方法，尝试替代方案
2. 询问用户不同的方法是否可以接受
3. 报告无法在给定的约束条件下完成任务

这是正确的行为。钩子不是崩溃器——它是重定向器。一个设计良好的钩子会在原因字符串中提供足够的信息，帮助 Claude Code 找到替代路径。

### 与 OpenTalon 的关联

在 OpenTalon 中，PreToolUse 钩子强制执行开发过程中最重要的两个约束：危险的文件系统操作（在此处介绍）和写入代码中的秘密信息（在第 6.5 节中介绍）。这两个约束都可以作为规则包含在 CLAUDE.md 中——但依赖于 Claude Code 注意力的规则，弱于依赖于结构性执行的钩子。钩子在每次匹配的工具调用上触发，每次，无一例外。当我们在第 6.5 节配置完整的钩子系统时，两个钩子将同时就位，每个会话从第一个工具调用开始就受到保护。
