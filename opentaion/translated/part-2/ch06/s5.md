## 第 6.5 节："代码中禁止 API 密钥"钩子

保存时格式化钩子和通知钩子改善了工作流程质量。本节构建一个保护安全的钩子。Claude Code 有时会编写包含 API 密钥、令牌或凭据的代码直接放在源文件中——不是出于恶意，而是因为它试图生成可工作的代码，而它使用的配置值可能是来自对话上下文的字面量。一个被提交的秘密可以使 API 密钥失效、导致意外费用，或使服务暴露于未授权访问。

这个钩子在发生之前进行拦截。

### 为什么仅有规则是不够的

OpenTalon 的 CLAUDE.md 中说："永远不要提交数据库连接字符串——请使用环境变量。"这条规则在 `api/CLAUDE.md` 中，被标记为重要，Claude Code 在大多数情况下都会遵循。但"大多数情况下"对于秘密来说是不够的。在会话的第三小时，上下文压力很大，一个新的集成测试可能为了方便而包含一个硬编码的测试令牌。开发者没有注意到就提交了。秘密进入了 git 历史。

一个在 Write 和 Edit 上的 PreToolUse 钩子可以在文件被写入之前捕获这个问题。检查针对的是 Claude Code 即将写入的内容，而不是已经存在的文件。阻止操作在 git 看到内容之前就发生了。

### 完整的钩子脚本

```bash
#!/bin/bash
# .claude/hooks/no-api-keys.sh
# PreToolUse hook on Write and Edit
# Blocks file writes that contain patterns matching API keys or secrets

INPUT=$(cat)

# Extract the content being written
# For Write tool: 'content' field
# For Edit tool: 'new_string' field
CONTENT=$(echo "$INPUT" | python3 -c "
import json, sys
d = json.load(sys.stdin)
content = d.get('content') or d.get('new_string') or ''
print(content)
" 2>/dev/null)

# Check for common secret patterns
# These patterns are deliberately conservative to minimize false positives

# OpenRouter / OpenAI style keys: sk- followed by 40+ alphanumeric chars
if echo "$CONTENT" | grep -qE 'sk-[A-Za-z0-9]{40,}'; then
    echo '{"decision":"block","reason":"File contains what appears to be an OpenAI/OpenRouter API key (sk-...). Use environment variables instead: os.environ[\"OPENROUTER_API_KEY\"]"}'
    exit 0
fi

# Bearer tokens: Bearer followed by 30+ alphanumeric/special chars
if echo "$CONTENT" | grep -qE '"Bearer [A-Za-z0-9+/=_-]{30,}"'; then
    echo '{"decision":"block","reason":"File contains a hard-coded Bearer token. Use environment variables instead."}'
    exit 0
fi

# Supabase keys: typically eyJ... (base64-encoded JWT)
if echo "$CONTENT" | grep -qE '"eyJ[A-Za-z0-9+/=]{50,}"'; then
    echo '{"decision":"block","reason":"File contains what appears to be a Supabase JWT key. Use environment variables: os.environ[\"SUPABASE_ANON_KEY\"]"}'
    exit 0
fi

# Generic: long alphanumeric strings assigned to key-like variable names
if echo "$CONTENT" | grep -qiE '(api_key|secret_key|auth_token|access_token)\s*=\s*"[A-Za-z0-9+/=_-]{32,}"'; then
    echo '{"decision":"block","reason":"File contains a hard-coded credential (api_key, secret_key, or token assigned a literal value). Use environment variables instead."}'
    exit 0
fi

echo '{"decision":"approve"}'
```

### 刚才发生了什么

三个钩子脚本现在协同工作。在每次文件写入时：

1. no-api-keys PreToolUse 钩子首先触发，扫描正在写入的内容
2. 如果安全，写入继续
3. format-python PostToolUse 钩子触发，格式化任何被写入的 Python 文件

如果 no-api-keys 钩子触发，写入被阻止，Claude Code 会收到一条包含正确替代方案的具体错误消息。开发者永远不需要考虑 Claude Code 是否记得"禁止秘密"的规则。

### 模式校准

钩子中的模式是故意保守的。过于激进的模式会阻止合法的写入——看起来像令牌的测试夹具占位符、注释中的示例代码、超过 32 个字符的生成 UUID。误报比它们要解决的问题更糟糕，因为它们会中断合法工作，并训练开发者绕过钩子。

如果钩子在合法内容上触发，添加一个例外：用 `# SAFE: not a real key` 注释检测到的行，并更新检测模式以跳过该模式。像维护代码一样维护钩子——当模式变化时需要更新。

### 与 OpenTalon 的关联

在 OpenTalon 中，这个钩子是秘密到达 git 历史之前的最后一道防线。CLAUDE.md 规则是第一道防线。`.env` 文件模式和 `.gitignore` 配置是第二道。这个钩子是第三道——即使在开发者匆忙时、即使在 Claude Code 的上下文已饱和时、即使在会话已运行了三小时且注意力已下降时，它也会运行。它在结构上运行，每次都是。

### 里程碑 M5：自动化质量门禁已激活

随着完整的钩子系统就位——危险命令阻止、保存时格式化、操作系统通知和 API 密钥检测——里程碑 M5 已完成。对项目的每次写入现在都被扫描。每次 Python 编辑都自动格式化。每次完成都触发一个通知。开发环境在结构上强制执行质量。
