## 第 8.1 节：四种权限模式

Claude Code 并非在单一的权限状态下运行。它有四种模式，决定了它如何处理具有副作用的工具调用——文件写入、shell 命令、网络操作。理解这些模式不是可选的上下文；它决定了会话是流畅进行，还是每三十秒中断请求批准，或是完全没有刹车地运行。

这些模式不是从安全到危险的谱系。它们是从交互式到自主式的谱系，每种模式在特定的上下文中都是正确的选择。

### 四种模式

**`default`** —— Claude Code 在任何有副作用的工具调用之前提示批准。文件写入、Bash 命令、MCP 服务器操作——所有都需要显式的"是"才能执行。这是新会话的起始模式。适用于使用外部服务时、触碰需要仔细审查才能更改的文件时，或学习 Claude Code 如何推理新任务时。

**`acceptEdits`** —— Claude Code 自动批准文件创建和文件编辑，无需提示。Bash 命令仍需要批准。这是实现冲刺阶段的正确模式，当任务定义明确、范围清晰、工作主要是创建和修改源文件时。一个需要创建二十个文件并再编辑十个的实现会话应该使用 `acceptEdits` 以避免二十次中断。

**`plan`** —— 只读模式。Claude Code 可以使用 Read、Glob 和 Grep。它不能写文件、运行 Bash 命令或调用外部服务。这是架构分析和规划模式。当目标是产生一个计划或文档——而不是修改系统时使用。探索→规划→编码→提交工作流程中的规划阶段在此模式下运行。

**`bypassPermissions`** —— 跳过所有权限检查。每个工具调用立即执行，无需审查。此模式存在于隔离的沙箱环境中，其中任何行动的后果都是受限的——自动化 CI 流水线、没有外部凭据的 Docker 容器、没有访问生产系统的测试环境。它不是开发机器的生产力捷径。一台包含真实 API 密钥的 `~/.env`、生产系统的 ssh 密钥、git 仓库历史访问权限的开发机器——在这台机器上以 `bypassPermissions` 模式运行 Claude Code，意味着承担相当大的风险，即一个错误的提示或注入的指令就可能变成生产事故。

### Shift+Tab 切换

在会话期间，按 Shift+Tab 可以在权限模式之间循环切换，无需重启。这是实用的机制：以 `default` 模式启动会话，评估任务，当你对范围有信心时切换到 `acceptEdits`，如果任务进入需要更多审查的领域，则切换回 `default`。这个切换按键正是为这种流动性设计的。

当前模式显示在 Claude Code 的状态行和 `/status` 中。如果行为与预期不符，请检查——一个在你期望自动批准时却提示批准的会话，可能已经漂移回了 `default` 模式。

### OpenTalon 开发中每种模式的使用时机

OpenTalon 的构建具有不同的任务类型，需要不同的权限模式：

**架构会话**（编写 SPEC.md、设计 API schema、规划重构）——使用 `plan` 模式。输出是分析和文档。在计划被审查之前，不应更改任何文件。

**实现冲刺**（构建 CLI 命令、实现 API 路由、创建 React 组件）——使用 `acceptEdits`。任务是根据规范编写代码。文件的创建和编辑应该没有中断地流畅进行。

**涉及外部服务的会话**（配置 Supabase、设置 Railway、配置 Homebrew tap）——使用 `default`。这些操作在本地文件系统之外有后果。每个操作都值得明确的审查。

**CI 流水线**（第 17 章，自动化测试和审查）——`bypassPermissions` 唯一适用的上下文，而且只有在隔离的容器中运行时才合适。

### 与 OpenTalon 的关联

在 OpenTalon 中，每个会话的正确权限模式将在相关章节中明确指出。第三部分在 SPEC.md 会话中使用 `plan` 模式，在实现会话中使用 `acceptEdits`。第四部分在 BMAD 辅助会话中使用 `default`，因为 BMAD 代理会做出深思熟虑、高价值的决策，每一步都值得审查。模式不是一次设置就忘记的参数；它是随任务而变化的会话参数。

第 8 章将继续介绍规则语法，让你编码具体的权限约束——不仅是使用哪种模式，而是具体允许和拒绝哪些操作。
