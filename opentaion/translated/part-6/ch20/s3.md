## 20.3 节：作为成本质量杠杆的 /effort 命令

第 5 章介绍了 `/effort` 作为控制思考深度的一种方式。本节用成本数据重新审视它，因为思考 Token 并非免费，而且努力与输出质量之间的关系并非线性。

### 思考 Token 的成本

Claude Code 中的扩展思考（Extended Thinking）专门分配 Token 用于在产生输出之前进行推理。Token 分解：

| 努力级别 | 思考 Token | 近似乘数 |
|--------------|-----------------|------------------------|
| low          | ~4,000          | 1×（基准）           |
| medium       | ~10,000         | 2-2.5×                |
| high         | ~20,000         | 3-4×                  |
| max          | ~31,000         | 4-5×                  |

在 Sonnet 定价下，从 low 努力移动到 max 努力大约会使会话成本增加两倍。在 Opus 定价下，绝对数字更大。问题在于质量改进是否值得这个成本。

### 改进真实存在的场景

思考 Token 在具有歧义、权衡或复杂推理链的任务上能改善输出质量。这些是模型在做出承诺之前从多种方法中受益的任务：

- 编写具有不完整需求的 SPEC.md 或架构文档
- 调试有多个潜在原因的 bug
- 编写测试微妙边缘情况的测试
- 评估实现方法之间的权衡

对于这些任务，`low` 和 `max` 努力之间的差异是可观察的：max 努力的输出更全面、捕获更多边缘情况，并且需要的纠正迭代更少。

### 改进微乎其微的场景

在从输入到正确输出的路径是确定性的、定义明确的任务上，思考 Token 不会改善输出质量：

- 实现指定了精确函数签名和测试的故事文件
- 从模板生成样板代码
- 根据风格指南格式化代码
- 根据最近提交更新 CHANGELOG.md

对于这些任务，low 或 medium 努力产生的输出与 max 努力相同。额外的思考 Token 被消耗在了不需要扩展推理的问题上。

### 实用规则

从 medium 努力开始。仅当以下情况时增加到 high 或 max：
- 任务需要设计决策，而非实现决策
- medium 努力的首次尝试不正确，且错误出在推理上而非执行上
- 任务涉及多个需要平衡的相互制约因素

永远不要将 `/effort max` 设为默认值。真正需要最大思考的任务在所有任务中只占少数。

### 反模式

将 `/effort max` 用于所有事情是对 Claude Code 早期经验的常见反应："模型思考更多，它做出更好的决策，所以总是使用 max。"这个逻辑没有错——更多思考确实对某些任务有帮助——但在大多数例行任务上，边际收益为零。在 70% 的任务上将会话成本增加两倍却没有质量改进，这不是一个合理的权衡。

### 在 OpenTalon 中

在 OpenTalon 中，纲要中的按任务分配努力矩阵直接转化为决策规则：架构工作（SPEC.md、BMAD 规划会话、调试多组件问题）使用 `/effort high` 或 `/effort max`；实现工作（故事文件实现、测试编写）使用 `/effort medium`；样板代码和格式化工作使用 `/effort low`。如果在本书的整个开发过程中应用此规则，与始终使用 medium 努力相比，Token 消耗将减少约 25-35%——而在 low 努力就足够的任务上，输出质量没有可观察的下降。
