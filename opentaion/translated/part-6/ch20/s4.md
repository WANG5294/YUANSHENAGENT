## 20.4 节：OpenTalon 用量代理中的提示缓存

智能体系统中输入 Token 成本的最一致来源，是那些在请求之间不发生变化的内容。系统提示词。CLAUDE.md 的内容。智能体的角色定义。这些内容在每一轮都被加载、每次都计为输入 Token 并相应收费——即使模型以前已经"见过"它们。

提示缓存（Prompt Caching）解决了这个问题。当相同的开头前缀在请求中重复出现时，API 可以在第一次出现后缓存处理过的版本。后续以相同缓存前缀开头的请求，只需为这些 Token 支付原始输入成本的一小部分。

### 缓存的工作原理

OpenRouter API（通过 Anthropic API）通过特定消息上的请求头支持提示缓存：

```json
{
  "messages": [
    {
      "role": "user",
      "content": [
        {
          "type": "text",
          "text": "You are a helpful coding assistant...",
          "cache_control": {"type": "ephemeral"}
        }
      ]
    }
  ]
}
```

`cache_control` 头将此内容标记为缓存。如果在缓存窗口（大约 5 分钟）内，相同的内容前缀出现在后续请求中，API 会返回缓存命中——缓存的 Token 成本大约为原始输入成本的 10%。

### OpenTalon 的使用场景

OpenTalon CLI 在每个请求中都发送一个系统提示词。这个系统提示词定义了智能体的行为——它包括工具描述、角色定义以及 CLAUDE.md 中与智能体操作相关的指令。对于一个典型的 20 次请求的会话：

```
无缓存的会话：
  系统提示词：每次请求 2,000 Token
  20 次请求 × 2,000 Token = 40,000 输入 Token 按全价计算

带缓存的会话：
  第一次请求：2,000 Token 按全价（填充缓存）
  后续 19 次请求：2,000 Token × 10% = 每次 200 Token
  19 × 200 = 3,800 Token 按缓存价格计算

  缓存后的总输入：3,800 + 2,000 = 5,800 Token 有效成本
  相比无缓存：40,000
  节省：每次会话约 38,200 Token
```

按典型的 API 定价，这是有意义的。对于每天运行多次会话的开发者来说，节省会累积。

### 代理中的实现

```python
# api/src/opentaion_api/routers/proxy.py (relevant section)
async def proxy_to_openrouter(request_body: dict) -> AsyncGenerator:
    """Proxy request to OpenRouter with prompt caching."""

    messages = request_body.get("messages", [])

    # Add cache_control to the system message if present
    if messages and messages[0].get("role") == "system":
        system_content = messages[0]["content"]
        if isinstance(system_content, str):
            # Convert to content array format with cache_control
            messages[0]["content"] = [
                {
                    "type": "text",
                    "text": system_content,
                    "cache_control": {"type": "ephemeral"}
                }
            ]

    cached_request = {**request_body, "messages": messages}

    async with openrouter_client.stream(cached_request) as response:
        async for chunk in response:
            yield chunk
```

这个实现透明地向系统消息添加了 `cache_control`。CLI 发送标准 OpenAI 格式的请求。代理在转发到 OpenRouter 之前添加缓存头。CLI 对缓存毫不知情。

### 缓存持续时间和失效

临时缓存大约持续 5 分钟。对于用户积极工作的会话——每隔几分钟提问和接收响应——缓存保持有效。对于被长时间休息中断的会话，缓存过期，下一个请求会重建它。

缓存按内容键控，而非按会话。两个具有相同系统提示词的用户共享相同的缓存版本——每个用户的第一个请求构建缓存；来自任一用户的后续请求都会命中它。

### 在 OpenTalon 中

在 OpenTalon 中，提示缓存在第 20 章的优化冲刺期间被添加到代理中。上面的实现就是完整的变更。第 20.5 节里程碑中的前后对比测量了缓存加模型路由的综合效果——这是直接降低代理最常见使用模式中输入 Token 成本的两种优化。
