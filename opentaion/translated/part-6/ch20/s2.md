## 20.2 节：模型选择矩阵

并非每项任务都需要最强大的模型。将 Opus 分配给样板代码生成任务，就像让高级工程师去重命名变量——技术上可行，但不必要地昂贵。

模型选择矩阵将推理能力与任务难度相匹配。各层之间的成本比率从 Haiku 到 Opus 大约为 15 倍，从 Haiku 到 Sonnet 大约为 5 倍。正确的选择可以收回大部分成本。

### 三层层级

**Opus** —— 复杂推理、架构决策、模糊需求

当错误决策的成本超过使用更强大模型的成本时，使用 Opus。在规划阶段做出的架构决策（Winston 的架构文档、SPEC.md 的设计选择）会影响之后每次会话。第一次就做对，值得付出额外的成本。Opus 也擅长调试复杂的多组件问题，其中根本原因并不明显。

| 任务 | 使用 Opus？ |
|------|-----------|
| 编写 SPEC.md 或 architecture.md | 是 |
| 调试复杂的多组件 bug | 是 |
| 评估两种架构方法 | 是 |
| 实现一个说明充分的故事 | 否 |
| 根据验收标准编写测试 | 否 |

**Sonnet** —— 实现、测试、重构

Sonnet 在大多数任务上能正确处理实现工作。它是实现故事文件、编写测试、重构现有代码以及任何预期输出明确指定的任务的正确默认选择。OpenTalon 的大部分开发是在这个层级完成的。

| 任务 | 使用 Sonnet？ |
|------|------------|
| 实现 BMAD 故事文件 | 是 |
| 编写 Quinn 的测试套件 | 是 |
| 重构一个充分理解的模块 | 是 |
| 从模板生成样板代码 | 否（使用 Haiku） |
| 新颖的架构决策 | 否（使用 Opus） |

**Haiku** —— 样板代码、格式化、简单查找

Haiku 处理预期输出是确定性的或接近确定性的任务。根据最近的提交更新 CHANGELOG.md、根据风格指南重新格式化代码、根据标准模板生成 Dockerfile、或在代码库中搜索特定模式。任务不需要推理该做什么——它只需要执行一个明确的操作。

| 任务 | 使用 Haiku？ |
|------|-----------|
| 更新 CHANGELOG.md | 是 |
| 生成 Dockerfile 模板 | 是 |
| 格式化文件 | 是 |
| 简单的 grep 加解释任务 | 是 |
| 编写测试 | 否（使用 Sonnet） |
| 任何多步推理任务 | 否（使用 Sonnet 或 Opus） |

### 应用矩阵

在 Claude Code 中切换开销很低：无头模式下使用 `claude --model claude-haiku-4-5-20251001 -p "..."`，交互式会话中使用 `/model` 命令。决定只需五秒钟。不应用矩阵意味着在数十次会话中以 Opus 价格支付 Haiku 工作——这个数字会累积。

一个有用的启发式方法：如果你能用一句话描述任务且没有歧义，它很可能是一个 Haiku 或 Sonnet 任务。如果你发现自己需要写一段话在描述任务之前设置上下文，它很可能是一个 Sonnet 或 Opus 任务。

### 在 OpenTalon 中

在 OpenTalon 中，模型选择矩阵在两个层级上应用。在开发层级，开发者根据任务为每个 Claude Code 会话选择模型。在产品层级，OpenTalon 代理可以实现模型路由——根据请求的性质选择合适的免费模型层级。一个带有简短清晰提示词的请求路由到更快的模型；一个带有复杂多轮历史的请求路由到能力更强的模型。第 20.5 节涵盖了 CLI 用例中自动化此过程的 claude-code-router 模式。
