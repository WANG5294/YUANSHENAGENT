## 21.2 节：红灯：变通方案在增多

系统存在问题的最清晰信号，不是某个组件损坏了。而是变通方案（Workaround）的积累。一个变通方案证明系统未能正确处理某件事，而开发者选择绕开它而不是修复它。

一个变通方案是个小不便。五个变通方案就是一个在积极对抗其用户的系统。

### OpenTalon 的红灯

**因为 PostToolUse 钩子停止触发而手动格式化代码。** 第 6 章的钩子在每次文件写入后运行 `black` 和 `ruff`。当它停止触发时，开发者要么：在审查时注意到格式不正确的代码并手动格式化，要么交付格式不一致的代码。变通方案是手动格式化。根因通常是钩子路径配置错误、`black` 版本变更，或意外禁用了钩子的 settings.json 修改。

**因为对于明显需要它的任务"耗时太长"而跳过计划模式。** 计划模式（Plan Mode）之所以存在，是因为没有计划的实现会导致需要返工的架构。当计划模式被跳过用于那些显然会从中受益的任务时——添加新的子系统、重新设计现有模块——开发者已经认定短期时间成本超过了长期返工成本。对于任何大于单个函数的工作来说，这几乎总是错误的。

**因为"你们的模式总是错的"而忽略 Claude Code 的审查评论。** PR 审查 Action 对照 CLAUDE.md 进行检查。如果审查始终产生误报——将正确的代码标记为违反约定——要么 CLAUDE.md 中的约定是错误的，要么审查提示需要更具体地说明对于这个代码库来说什么样的代码是"正确的"。忽略审查是错误修复方式。更新 CLAUDE.md 或审查提示才是正确的修复方式。

**因为技能停止可靠更新而手动更新 progress.md。** 会话收尾技能本应在每次会话结束时更新状态文件。当这个功能停止工作时，开发者要么：让状态文件变陈旧（这会导致下次会话的混乱），要么手动维护它们（这恰恰是技能旨在解决的整个问题）。陈旧的状态文件是上下文连贯性失效的开始。

### 为什么变通方案会累积

每个变通方案都很小。在钩子失效后手动格式化一个文件的开发者几乎没有感到不便。同一个已经手动格式化了三周的开发者，已经破坏了钩子维护的习惯，仓库中有格式不一致的文件，并且已经内化了"格式化是我的工作，不是系统的工作"——这是一种认知转变，使得未来的变通方案更可能发生。

变通方案以两种方式累积：它们解决症状而不解决原因（所以原因继续产生新症状），并且它们使绕开问题而不是修复问题的做法常态化。

### 诊断问题

"这在什么时候开始感觉像摩擦了？"答案通常是一个具体事件：一次 Claude Code 升级、一个修改了钩子的新成员、一个引入了矛盾的 CLAUDE.md 变更、一个根据路径引用了文件但文件已被移动的重构。将摩擦追溯到其源头可以揭示出修复方式是一个配置更新、一个 CLAUDE.md 修正，还是需要真正重新设计某个组件。

### 在 OpenTalon 中

在 OpenTalon 中，红灯是需要定期检查的：每个月问一次"我在应用多少个变通方案？"如果答案超过两三个，就来审计系统。具体的变通方案会指向已经偏离轨道的具体组件。修复组件，而不是症状。第 21.3 节讨论了什么时候正确的修复是演进而非修理。
