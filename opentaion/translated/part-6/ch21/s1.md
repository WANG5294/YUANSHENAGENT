## 21.1 节：绿灯：当你的系统不可见时

一个运行良好的智能体开发系统的衡量标准，不是它的配置多么令人印象深刻。而是它有多不可见。

当系统正常工作时，它处理了所有开销——加载上下文、执行约定、格式化代码、防止错误——而开发者完全没有注意到。开发者思考的是产品，而不是系统。这种不可见性就是成功。

### 悖论

在构建智能体开发系统时有一个诱人的失败模式：优化系统而不是使用它。一个花了三十分钟完善 CLAUDE.md 的开发者的行为，实际上只需五分钟的实现就能推动产品前进，这样的开发者让系统本身变成了工作。

正确的关系是倒过来的：产品是工作，系统是工具。一个运行良好的工具会消失。你在钉钉子的时候不会想锤子的事。

### 绿灯

**CLAUDE.md 在会话中不需要被查阅。** 当约定正确时，Claude Code 无需提醒就会遵循它们。当你发现在实现会话中途输入"记住，我们用的是 uv 而不是 pip"时，相关的约定要么不在 CLAUDE.md 中，要么表述方式没有被遵循。两者都提示需要修复。

**技能运行时不需回忆如何调用。** 当 `/opentaion-component` 被不假思索地使用——你下意识地输入它——它就是有效的。当你发现在使用技能之前需要翻查 `.claude/skills/` 来找到技能名称时，这个技能要么命名不好，要么使用不足。

**测试在智能体首次尝试时通过的比例高于失败的比例。** 这与完美无关。而是关于比率。当 TDD 执行机制正常工作时，Amelia 在实现之前编写测试，而她编写的实现能够通过这些测试。当比率颠倒过来——大多数故事在测试通过之前需要纠正循环——CLAUDE.md 中的 TDD 约定就没有被可靠地遵循。

**每次变更后下意识地检查仪表盘。** 当用量追踪正常工作时，它成为一种习惯。开发者运行一条命令，然后扫一眼仪表盘看它更新。这个习惯不是单独的行动——它是工作流的一部分。当这个习惯需要被记住时，反馈闭环就不够紧密。

### "不可见基础设施"基准

如果你在工作会话中发现自己思考 Claude Code 的机制，说明系统的某些方面需要改进。不是因为这些机制不重要——显然它们重要，否则这本书就不会存在——而是因为在会话进行时，这些机制应该是不可见的。

开发者的注意力是一种有限的资源。每一刻花在思考"如何让 Claude Code 做 X"上的时间，就是一刻没有花在思考"OpenTalon 应该做什么"上的时间。系统应该自动回答前一个问题。

### 在 OpenTalon 中

在 OpenTalon 中，这些绿灯是可测试的。运行黄金集并观察会话：CLAUDE.md 的约定是否需要重新陈述？技能是否需要解释？测试是否需要纠正循环？答案会告诉你系统在哪里仍然可见。第 21.2 节探讨了可见摩擦的表现形式及其通常的含义。
