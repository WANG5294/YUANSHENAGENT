## 21.5 节：OpenTalon V2：接下来是什么

V1 已上线。它工作。一个陌生人可以用一条命令安装它，并在实时仪表盘中观看其 Token 用量的累积。这是 V1 的正确基准：它能工作，而不是它很完美。

先诚实评估。

### V1 的局限性

**CLI 仅限 macOS。** Homebrew 分发在 macOS 上工作。Linux 和 Windows 用户需要克隆仓库并手动配置环境。对于初始用户群来说，这不是一个阻塞性限制——macOS 上的独立开发者是主要用户画像——但它限制了覆盖范围。

**仪表盘功能可用但不够精致。** 柱状图正常渲染。模型细分表正常渲染。加载状态正常。这不是一个美观的产品。关心呈现的独立开发者会注意到。关心数据的独立开发者则不会。

**Web 平台本身没有免费层。** OpenRouter 提供免费模型。OpenTalon API 必须自托管，或者用户必须接受 Railway 的计费。对于想要完全托管、零成本设置的开发者来说，当前架构需要技术自托管能力或接受少量的托管成本。

**无协作功能。** 每个账户一个用户。如果同一项目上的两个开发者想要共享用量可见性，他们创建单独的账户。这是 V1 的范围，但对于小团队来说这是一个真实的限制。

### V2 的方向

这些是基本原则性的方向，而非产品路线图。每个都针对 V1 中的一个真实限制。

**通过 Ollama 支持本地模型。** 代理设计已经支持任何与 OpenAI 兼容的端点。将 CLI 指向本地运行的 Ollama 实例只需要配置变更。V2 将其正式化：`opentaion --local` 使用 Ollama 替代代理，为本地运行模型的开发者完全消除网络延迟和 Token 成本。

**团队账户。** 一个共享账户，多个用户的 API 密钥链接到同一个用量仪表盘。数据库模式已经支持这一点（usage_logs 上的 user_id 可以扩展为 org_id）。产品设计是更难的部分：谁能看到谁的用量，谁能撤销谁的密钥。

**带有告警的用量预算。** 每个用户的月度 Token 预算。当预算即将用完 10% 时，发送告警。PRD 中"不在范围内"部分将其排除在 V1 之外，因为仪表盘是核心功能。V2 使仪表盘变得可操作。

**更好的上下文管理。** 当前上下文管理器使用 tiktoken 和固定的 100K Token 预算。V2 使用向量存储（ChromaDB 或类似方案）实现跨会话的语义记忆——智能体可以从之前的会话中回忆相关上下文，而无需将所有内容保留在上下文窗口中。

### 如何规划 V2

到了本书的这个阶段，读者已经知道如何在没有指导的情况下规划 V2。Mary 引导产品简介。John 编写 PRD。Winston 设计架构。Bob 创建故事文件。方法论已经确立。

与第 13 章的区别在于经验。读者现在知道哪些规划问题重要（API 合约、数据库模式、认证模型），哪些为时过早（UI 美化、未来集成）。他们知道规模-领域-自适应原则适用——V2 可能需要完整的 BMAD 规划序列，也可能不需要，取决于选择推进哪些方向。

### 元反思

在序言中，本书提出了一个主张：理解智能体工程的最佳方式是去做。不是阅读它，而是用工具构建真实的东西，并观察会发生什么。

在下一个项目上运行 `/write-section 6.21.5` 的感觉，将与在第 2 章中第一次运行 `/write-section` 时截然不同。模型没有变。读者变了。他们拥有关于智能体工程需要什么的上下文——不仅仅是知识：先规划后工作流的纪律、TDD 的可靠性、超详细故事文件带来的清晰度、以及让过程保持诚实的黄金集的价值。

那个上下文就是这本书的意义所在。
