## 18.1 节：理解债务问题

当人类编写每一行代码时，他们理解每一行。当智能体编写了大部分代码时，就出现了一个缺口：代码库中存在开发者没有编写，可能也不完全理解的代码。这个缺口就是理解债务（Comprehension Debt）。
理解债务是智能体开发中相当于技术债务的概念。技术债务是指比应有状态更难变更的代码。理解债务是指比应有状态更难调试，修改或解释的代码。两者相互叠加：你不理解的代码会更快地积累技术债务，因为你无法判断哪些变更是安全的。
### 为什么这很重要

这个问题是具体的。生产中出了故障。用户报告了一个错误。开发者查看堆栈跟踪，找到相关文件，阅读代码——这是 Amelia 在第 7 次会话中编写的代码，实现了 story-014，遵循了 Bob 在技术笔记中描述的模式。开发者之前作为故事审查的一部分读过这段代码，但阅读不等于理解。他们能看到代码做了什么。他们不知道为什么要做出特定的选择。
调试不熟悉的代码比自己编写的代码更慢，风险更高，也更容易引入新 bug。此即理解债务的真实代价——它在出问题的时候浮出水面，而这恰恰是最不适合缓慢和不确定的时机。
### 缓解策略

有三种实践可以保持理解债务在可控范围内。它们不能消除理解债务——那需你自己编写所有内容，这就失了意义——但它们可以将其保持在成为生产隐患的阈值以下。
**在合并前审查，以理解为目标。** Amelia 实现的每个故事在故事状态设置为"done"之前都会由人类审查。质量门控的审查清单是针对捕获错误的。而这里的审查是为了建立理解。要问的问题是："如果这段代码在凌晨 2 点出故障，我能调试它吗？"如果不能，在合并之前让 Claude Code 解释这段代码。
```
"Explain how the date series join in the /usage/daily endpoint works.
Walk through the SQL logic and tell me what would happen if
created_at was stored in UTC but the user's timezone is UTC+9."
```

这个问题产生的解释会留在工作记忆中，建立理解，并经常暴露出值得测试的边缘情况。
**在发布前询问，而不是在出故障后。** 每段开发者不理解的代码都值得在部署之前一次快速的解释会话。现在花五分钟解释，比以后花三十分钟调试要快。
**测试即文档。** 测试描述了代码应该做什么。Quinn 为用量计量代码编写的测试套件记录了预期行为：什么输入产生什么输出，哪些边缘情况被处理，哪些错误被抛出。阅读测试通常比阅读实现代码能更快地建立对系统行为的直觉。
### 诚实的立场

在高 AI 辅助水平下，理解债务是真实存在且不可避免的。大量用 Claude Code 实现的开发者，其仓库中会存在他们部分理解而非完全理解的代码。这不是一种失败模式——这是 AI 辅助项目的自然状态。
上述实践可以管理它，但不能消除它。把每一行 AI 生成的代码都当作和自己手写代码一样完全理解的开发者，比承认差距并加以补偿的开发者处于更危险的位置。
### 在 OpenTalon 中

在 OpenTalon 中，审查要求被编码在 BMAD 故事生命周期中：在人类审查 Amelia 的实现之前，故事不算完成。这项审查不是可选的。它是开发者保持足够理解，能调试自己系统的机制。Quinn 生成的测试套件是第二个机制——它以可执行的形式记录行为，开发者在出故障时可以运行它。强制性审查加上全面的测试套件，共同将理解债务保持在危险阈值以下。
