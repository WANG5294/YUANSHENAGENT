## 18.2 节：用于端到端测试的 Playwright MCP

单元测试验证单个函数是否按预期工作。集成测试验证组件之间是否正确协作。端到端测试验证真实用户是否能完成真实的工作流。三层都是必需的；而 E2E 层是最容易被跳过的。
E2E 测试被跳过的原因是摩擦：设置浏览器自动化环境，编写可靠的选择器，管理测试数据，针对实时服务运行测试。Playwright MCP 通过让 Claude Code 成为编写和调试 E2E 测试的一等参与者，消除了大部分摩擦。
### 三个关键流程

OpenTalon 有三个端到端流程必须正确工作，产品才能有用：
**流程 1：注册。** 访客访问网站 → 输入电子邮件 → 收到魔法链接 → 点击链接 → 到达仪表盘。
**流程 2：API 密钥创建。** 已认证用户 → 创建新密钥 → 复制明文密钥 → 关闭弹窗 → 在表格中看到密钥前缀。
**流程 3：用量显示。** 用户通过 CLI 发起 API 调用 → 刷新仪表盘 → 看到 Token 计数已更新。
这些流程跨越了组件边界。流程 1 涉及 React 前端和 Supabase Auth 服务。流程 2 涉及前端和 FastAPI API。流程 3 涉及 CLI，API 代理，数据库和前端。只有 E2E 测试能一起练习所有这些组件。
### 使用 Claude Code 编写 Playwright 测试

```
/bmad-agent-bmm-qa

"Write Playwright tests for the OpenTalon registration flow.
The test should:
1. Navigate to http://localhost:5173
2. Enter a test email address
3. Click the submit button
4. Verify the success message appears

Use the accessibility snapshot approach — read the page's
accessible elements before writing selectors, not the DOM."
```

Quinn 用 Playwright MCP 的可访问性快照工具来读取页面结构：
```python
# The Playwright MCP accessibility snapshot for the registration page:
# heading "Sign in to OpenTalon"
# textbox (placeholder "you@example.com")
# button "Send magic link"

# Quinn generates:
async def test_registration_shows_success_message(page: Page):
    await page.goto("http://localhost:5173")
    await page.get_by_placeholder("you@example.com").fill(
        "test-e2e@example.com"
    )
    await page.get_by_role("button", name="Send magic link").click()
    await expect(
        page.get_by_text("Check your email for a sign-in link")
    ).to_be_visible()
```

可访问性快照方法用 `get_by_role`，`get_by_placeholder` 和 `get_by_text` 选择器，而不是 CSS 类或 XPath。这些选择器能经受住样式变更——改变 CSS 类的重新设计不会破坏测试，因为测试是通过可见文本和 ARIA 角色来选择，而不是通过类名。
### 在 CI 中运行 E2E 测试

```yaml
# In .github/workflows/tests.yml
- name: Start services for E2E
  run: |
    docker-compose up -d api web
    # Wait for services to be ready
    sleep 10

- name: Run Playwright tests
  run: |
    cd web && npx playwright test --reporter=github

- name: Stop services
  run: docker-compose down
```

E2E 测试针对本地 Docker 启动的堆栈运行。CI 期间不调用任何外部服务——Supabase 被存根用于认证流程测试，OpenRouter 被模拟用于代理测试。这保持了 CI 运行速度，并消除了外部服务可用性引起的测试不稳定。
### 测试数据策略

每个 E2E 测试创建自己的测试用户：
```python
@pytest.fixture
async def test_user(supabase_client):
    # Create a test user
    user = await supabase_client.auth.admin.create_user(
        email=f"test-{uuid4()}@example.com",
        email_confirm=True
    )
    yield user
    # Clean up after the test
    await supabase_client.auth.admin.delete_user(user.id)
```

没有共享的测试状态。每个测试从干净状态开始。这防止了测试顺序依赖——一个依赖前一个测试遗留状态才能通过的测试，会在顺序改变时随机失败。
### 在 OpenTalon 中

在 OpenTalon 中，三个关键流程各自有一个 E2E 测试。注册测试用 Supabase 存根，避免在 CI 期间发送真实的魔法链接邮件。API 密钥测试针对测试数据库创建真实密钥，并验证前缀出现在表格中。用量显示测试发起一个真实的代理请求（指向模拟的 OpenRouter 端点），并验证仪表盘更新。这些测试是测试金字塔的顶层——它们在 CI 中最后运行，是最能增强信心的检查，用以验证系统能否端到端地工作。
