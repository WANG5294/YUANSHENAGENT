## 17.5 节：钩子系统作为 CI/CD 治理层

第 6 章的钩子——代码质量执行，API 密钥检测，操作系统通知——在开发期间本地运行。它们是两层治理模型的第一层。CI/CD 流水线是第二层。两层都执行相同的规则。
这种冗余并非浪费。本地层在代码提交之前捕获问题。CI 层捕获那些漏网或被绕过本地层引入的问题。两者共同构成一个系统，使违反规则需主动规避两个独立的机制。
### 双层模型

```
第一层：开发钩子（第 6 章）
  PostToolUse → 每次文件写入后格式化 Python
  PreToolUse  → 阻止包含 API 密钥模式的写入
  Notification → 任务完成时发出警报

第二层：CI 流水线（第 17 章）
  PR 审查    → Claude Code 对照 CLAUDE.md 检查
  测试门控   → pytest + Playwright 必须通过
  密钥扫描   → grep 常见密钥模式
  覆盖率     → 不得低于阈值
```

这两层并不相同。开发钩子在每次文件写入时运行，并可以修改文件（重新格式化）。CI 层在每次 PR 上运行，产生通过/失败判定。不同的机制，相同的规则。
### allowManagedHooksOnly 设置

在团队环境中，开发者可以修改他们本地的钩子配置。一个觉得代码格式化钩子烦人的开发者可以禁用它。Claude Code 组织配置中的 `allowManagedHooksOnly` 设置可以防止这种情况：
```json
// Organization-level settings
{
  "allowManagedHooksOnly": true
}
```

启用此设置后，开发者无法覆盖组织级别定义的钩子策略。治理是强制的，而非建议的。
独立开发者，不需这个设置——你既是执行者也是被执行者。但基于 OpenTalon 架构构建的团队，知道这个设置的存在是有价值的。
### 审计追踪

CI 中的每个 Claude Code 会话都会生成一个日志：读取了哪些文件，执行了哪些命令，写入了什么。此日志在 GitHub Actions 工作流运行输出中可用。
对于合规敏感的项目，此审计追踪记录了代码库的每次变更都是通过一组已知的，具有特定权限的工具处理的，并且输出是由特定版本的 Claude Code 配合特定提示词生成的。JSON 输出中的 `session_id` 将审查与特定的 Claude Code 会话关联起来。
### 里程碑 M16：触发完整流水线

```bash
# Create a test PR
git checkout -b test-ci-pipeline
echo "# Test" >> docs/ci-test.md
git add docs/ci-test.md
git commit -m "test: verify CI pipeline is working"
git push origin test-ci-pipeline
# Open PR via GitHub UI or gh pr create
```

预期的流水线行为：
1. 测试运行——通过（没有代码变更）
2. Claude Code 审查运行——输出 "OK: No issues found"（仅文档变更）
3. 密钥扫描运行——通过（文档中没有密钥）
4. 覆盖率检查——通过（没有代码变更）
5. 所有门控通过 → 允许合并

通过合并测试 PR 来确认。确认变更日志工作流在合并时运行，并向 CHANGELOG.md 提交新条目。
### 在 OpenTalon 中

在 OpenTalon 中，CI/CD 流水线在系统部署之前就已运行。这个顺序很重要：流水线验证将要部署的软件。在流水线设置好之前部署意味着第一次生产部署是未经验证的。在流水线设置好之后部署意味着每次后续变更在到达用户之前都经过验证。第 18 章增加了最终的测试层。第 19 章部署经过验证的系统。

### 刚刚发生了什么

OpenTalon 代码库现在有两层质量执行机制：开发期间的钩子和 CI 期间的门控。两层执行相同的规则。流水线不是事后才考虑的事项——它在生产部署之前构建，确保第一个部署的版本已过验证。每次后续变更都必须通过四个自动化检查才能到达用户。
