## 17.3 节：合并前的质量门控

质量门控（Quality Gate）是一项必须通过的检查，代码才能到达主分支。正确实现门控意味着 main 分支始终处于可部署状态——main 上的每次提交都已通过测试，审查和安全检查。没有例外。
OpenTalon 的质量门控栈有四层。它们共同捕获不同类别的问题。
### 第一层：测试必须通过

```yaml
# .github/workflows/tests.yml (partial)
- name: Run API tests
  run: |
    cd api && uv run pytest tests/ -v --tb=short

- name: Run web build
  run: |
    cd web && npm run build

- name: Run Playwright tests
  run: |
    cd web && npx playwright test
```

测试失败时，不允许合并新代码。这不是可选的。失败的测试意味着合约被破坏——Quinn 对照 Amelia 实现编写的验收标准没有通过。门控强制执行故事文件中做出的承诺。
### 第二层：Claude Code 审查不得发现关键问题

第 17.2 节的自动 PR 审查是一项必须通过的检查。审查输出中的 CRITICAL 问题会阻止合并。
什么算 CRITICAL：安全漏洞，API 密钥泄露，公开端点缺少输入验证，违反架构安全决策（如，存储明文 API 密钥）。什么不算 CRITICAL：命名风格，代码组织偏好，重构现有代码的建议。
该检查实现为一个 GitHub Actions 步骤，解析 Claude Code 审查输出：
```bash
# In the CI step after the review runs
if echo "$REVIEW_OUTPUT" | grep -q "^CRITICAL:"; then
  echo "Critical issues found. Merge blocked."
  exit 1
fi
```

### 第三层：代码中不得包含密钥

第 6 章的 API 密钥检查钩子在开发期间本地运行。CI 等效检查针对已提交的代码运行：
```yaml
- name: Check for exposed secrets
  run: |
    # Check for common secret patterns
    if git diff --name-only HEAD~1 | \
       xargs grep -l "sk-\|OPENROUTER_API_KEY\s*=" 2>/dev/null; then
      echo "Potential secret exposure detected"
      exit 1
    fi
```

这可以捕获本地钩子被绕过或配置错误的情况。
### 第四层：测试覆盖率不得下降

```yaml
- name: Coverage check
  run: |
    cd api && uv run pytest tests/ --cov=opentaion_api \
      --cov-fail-under=${{ env.COVERAGE_THRESHOLD }}
```

阈值从测试套件在 M15 时达到的覆盖率水平开始，此后不得下降。新代码必须经过测试，或明确标记为不需覆盖（这种情况很少见）。门控防止覆盖率无声侵蚀。
### 分支保护配置

在 GitHub 的仓库设置 → 分支 → `main` 的分支保护规则中：
```
✓ Require status checks to pass before merging
  Required checks:
    - tests / api-tests
    - tests / web-build
    - tests / playwright
    - review / claude-code-review
    - secrets / no-exposed-secrets
    - coverage / coverage-threshold

✓ Require branches to be up to date before merging
✓ Include administrators (no exceptions)
```

包括管理员可以防止人类在时间压力下绕过门控。如果门控有问题，就修复门控——不要绕过它们。
### 元观察

Claude Code 既是此仓库中大部分代码的作者，也参与了审查过程。编写了 story-001 实现的同一个智能体，将审查下一个 PR 的变更是否符合 CLAUDE.md 中的约定。这并非循环论证——而是一致性检查。Claude Code 可以验证新代码是否遵循了它在之前会话中确立的模式，而新的人类审查员可能并不知道这些模式。
### 在 OpenTalon 中

在 OpenTalon 中，最有价值的门控是覆盖率检查。Amelia 的实现方法倾向于对核心路径良好的测试，但错误处理偶尔会有缺口。覆盖率门控在缺口变成生产事故之前捕获它们。每次门控阻止合并时，都是一个在快乐路径上满足但在错误路径上未测试的故事验收标准——这是真实的缺口，而不是官僚主义的抱怨。
