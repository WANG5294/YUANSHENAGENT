## 19.2 节：在 Railway 和 Vercel 上部署 Web 平台

两个部署，两个平台，一次推送。Railway 部署 FastAPI API。Vercel 部署 Vite React 前端。两者连接到同一个 GitHub 仓库，监视不同的目录，并在每次通过 CI 的推送到 `main` 时自动部署。
### Railway 上的 API

Railway 是一个平台即服务（PaaS），可以从 Dockerfile 或 buildpack 部署。对于 OpenTalon 的 FastAPI API，它用 Dockerfile：
```dockerfile
# api/Dockerfile
FROM python:3.12-slim

WORKDIR /app

# Install uv
RUN pip install uv

# Copy dependency files
COPY pyproject.toml uv.lock ./
RUN uv sync --no-dev

# Copy application code
COPY src/ ./src/

# Run with uvicorn
CMD ["uv", "run", "uvicorn", "opentaion_api.main:app",
     "--host", "0.0.0.0", "--port", "8080"]
```

位于 `api/` 的 Railway 配置文件：
```json
// api/railway.json
{
  "build": {
    "builder": "DOCKERFILE",
    "dockerfilePath": "Dockerfile"
  },
  "deploy": {
    "startCommand": "uv run uvicorn opentaion_api.main:app --host 0.0.0.0 --port $PORT",
    "healthcheckPath": "/health",
    "healthcheckTimeout": 10
  }
}
```

在 Railway 仪表盘中设置环境变量（不在代码中）：
```
SUPABASE_URL=https://[project-id].supabase.co
SUPABASE_ANON_KEY=[your-anon-key]
SUPABASE_SERVICE_ROLE_KEY=[your-service-role-key]
OPENROUTER_API_KEY=[your-openrouter-key]
```

连接 GitHub 仓库并选择 `api/` 目录后，Railway 会在每次推送到 `main` 时部署。API URL 在 Railway 仪表盘中可用：`https://opentaion-api.up.railway.app`。
### Vercel 上的 Web

Vercel 会自动检测 Vite 项目。设置最少：
1. 将 GitHub 仓库连接到 Vercel
2. 将根目录设置为 `web/`
3. Vercel 检测到 Vite 并自动配置构建命令
4. 设置一个环境变量：

```
VITE_API_URL=https://opentaion-api.up.railway.app
```

Vercel 在每次推送到 `main` 时部署。Web URL 在 Vercel 仪表盘中可用：`https://opentaion.vercel.app`。
### 环境变量策略

代码中不包含密钥。Git 仓库中不包含密钥。Railway 和 Vercel 在构建时和运行时注入环境变量。应用程序用 `os.environ.get()`（Python）或 `import.meta.env.VITE_API_URL`（Vite）来读取它们。
`api/` 和 `web/` 中的 `.env.example` 文件记录了需哪些变量，但不包含任何值：
```bash
# api/.env.example
SUPABASE_URL=
SUPABASE_ANON_KEY=
SUPABASE_SERVICE_ROLE_KEY=
OPENROUTER_API_KEY=
```

克隆仓库的开发者将 `.env.example` 复制为 `.env`，并填入自己的值用于本地开发。Railway 和 Vercel 用自己的环境变量管理方式用于生产环境。
### 部署流水线

推送到 `main` 时的完整部署流程：
1. GitHub Actions 运行：测试、PR 审查（如果是 PR）、密钥检查
2. 所有检查通过 → Railway 检测到新提交并重建 API 容器
3. Railway 健康检查通过 → 新容器上线
4. Vercel 检测到新提交并重建前端
5. Vercel 部署成功 → 新的前端被提供
6. 变更日志工作流运行并更新 CHANGELOG.md

整个流水线从推送到实时部署大约需 5-7 分钟。推送后无需手动步骤。
### 在 OpenTalon 中

在 OpenTalon 中，两个部署在系统提升到生产环境之前就已连接好了。首次部署在完成后立即通过运行生产黄金集测试。如果有任何黄金集任务失败，在 Railway 中回滚部署（一键操作），调查问题后再重新部署。回滚能力是为什么 V1 Railway 优于更复杂的部署目标的原因——失败的部署可以在 30 秒内恢复。
