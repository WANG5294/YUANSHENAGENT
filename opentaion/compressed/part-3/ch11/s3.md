## 第 11.3 节：Git Worktree 隔离：三个智能体，三个分支

编排器-Worker 模式解决了上下文隔离——每个 Worker 从头开始。但它没有解决文件系统隔离：如果两个 Worker 同时在同一个 git 仓库上运行，它们都会修改同一个分支上的同一个文件，产生难以理清的冲突。

Git worktree 解决了此问题。worktree 是工作目录，链接到同一个底层 git 仓库，但签出到不同的分支。两个拥有不同 worktree 的 Worker 可以独立地写文件、运行测试和提交——执行期间无需任何协调，在不同的目录和不同的分支上工作。冲突只解决一次，集成时，而非构建中持续发生。

### 为 OpenTalon 设置 Worktrees

```bash
# 创建三个用于并行开发的分支
git checkout -b cli-development
git checkout main
git checkout -b web-development
git checkout main
git checkout -b api-development
git checkout main

# 创建三个 worktree，每个在自己的分支上
git worktree add ../opentaion-cli-work cli-development
git worktree add ../opentaion-web-work web-development
git worktree add ../opentaion-api-work api-development
```

此设置之后：
-`opentaion/`（原始目录）→`main`分支
-`opentaion-cli-work/`→`cli-development`分支
-`opentaion-web-work/`→`web-development`分支
-`opentaion-api-work/`→`api-development`分支

每个 worktree 都是工作目录。在`opentaion-cli-work/`中运行的 git 命令操作`cli-development`分支。在`opentaion-web-work/`中运行的 git 命令操作`web-development`分支，共享对象数据库（故存储效率高），但有独立的工作树和分支。

### Claude Code 如何使用 Worktrees

Claude Code 通过`--worktree`标志支持 worktree：

```bash
claude --worktree ../opentaion-cli-work "根据 cli/SPEC.md 实现 CLI 智能体循环"
```

这会指定的 worktree 目录中打开一个 Claude Code 会话。所有工具调用（Read、Write、Edit、Bash）都相对于该目录操作。除非显式更改目录——权限规则应阻止这种情况——否则该会话无法访问其他 worktree。

当编排器使用 Task 工具派生 Worker 时，它将 worktree 路径作为任务上下文的一部分传递：

```
Worker A 任务（在 ../opentaion-cli-work 中运行）：
"你的工作目录是 ../opentaion-cli-work，位于 cli-development 分支。
实现智能体循环... [规格说明]"
```

### 合并策略

当三个 Worker 都完成时，每个都已将工作提交到各自的分支。合并发生在主仓库中：

```bash
cd opentaion/

# 在合并前审查每个分支
git diff main..cli-development
git diff main..web-development
git diff main..api-development

# 满意后合并
git merge cli-development --no-ff -m "合并 CLI 智能体循环实现"
git merge web-development --no-ff -m "合并 Web 仪表盘脚手架"
git merge api-development --no-ff -m "合并 FastAPI 实现"
```

`--no-ff`标志创建合并提交（merge commit）而非快进（fast-forward），将并行开发历史保留 git 日志中。

因三个组件占据完全独立的目录（`cli/``web/``api/`），应该不会出现文件级的合并冲突。每个组件的文件是独立的。集成冲突——API 合约不对齐、环境变量命名不一致——不是 git 意义上的合并冲突，是逻辑上的不对齐，需要审查和调整。第 11.5 节涵盖了集成检查清单。

### 清理 Worktrees

合并后，移除 worktree：

```bash
git worktree remove ../opentaion-cli-work
git worktree remove ../opentaion-web-work
git worktree remove ../opentaion-api-work

# 可选：删除分支
git branch -d cli-development
git branch -d web-development
git branch -d api-development
```

Worktrees 是并行构建的临时工具，产生的提交现在已在`main`中。Worktrees 本身不再需要。

### OpenTalon 的关联

OpenTalon 中，三个 worktree 是并行构建的执行环境。CLI 智能体在`opentaion-cli-work`中工作，构建智能体循环并使用`uv run pytest`测试它。Web 智能体在`opentaion-web-work`中工作，搭建 Vite 项目并运行`npm run build`。API 智能体在`opentaion-api-work`中工作，实现 FastAPI 路由。每个都有自己的分支、自己的提交历史和自己测试结果——合并前完全隔离。上述设置应第 11 章的实现部分开始并行构建前执行。
