## 第 11.5 节：合并并行工作

三个智能体已独立构建了三个组件。CLI 智能体循环运行并通过了测试。Vite Web 项目构建成功。FastAPI 应用程序启动并响应健康检查。现在，这三个组件必须被集成到一个系统中：CLI 调用 API，API 调用 OpenRouter，Web 仪表盘显示 API 记录的内容。

这是集成时刻。三个组件基于同一规格说明构建，故主要架构冲突应该无。微小的不对齐是预期且正常的。

### 集成检查清单

宣布并行构建完成前，逐一运行每个检查项：

**1. 分别编译每个组件。**

```bash
# CLI
cd cli && uv run pytest tests/ -v
# Web
cd web && npm run build
# API
cd api && uv run pytest tests/ -v && uv run uvicorn opentaion_api.main:app --port 8000
```

集成开始前，每个组件必须通过自己的测试和构建。在测试失败的组件上进行集成工作是考古，而非工程。

**2. 验证 API 合约（contract）。**

CLI 调用 API 上的`POST /v1/chat/completions`。API 返回形状如下的响应：

```json
{
  "choices": [
    {
      "message": {
        "content": "..."
      }
    }
  ]
}
```

确认：
- CLI 的`OpenRouterClient.complete()`调用指向正确的端点路径
- API 的代理路由以该形状返回数据
- 认证头格式匹配（`Authorization: Bearer ***`）

端点路径、认证头格式或响应形状的任何不匹配都会导致 CLI 调用本地 API 而非直接调用 OpenRouter 时失败。这是最常见的集成不对齐。

**3. 验证环境变量名称。**

每个组件假定了特定的环境变量名称。常见冲突：

| 组件 | 变量 | 可能的名称 | |------|------|-----------| | CLI | API 基础 URL |`OPENTAION_API_URL`| | API | Supabase URL |`SUPABASE_URL`| | API | OpenRouter 密钥 |`OPENROUTER_API_KEY`| | Web | API 基础 URL |`VITE_API_URL`（Vite 要求使用`VITE_`前缀以便在浏览器中暴露） |

检查每个组件的配置文件（如果创建了`.env.example`）并确认名称匹配。

**4. 运行集成测试。**

在 API 运行于`localhost:8000`且 CLI 配置指向其条件下：

```bash
cd cli && OPENTAION_API_URL=http://localhost:8000 uv run python -m opentaion "列出 cli/src/ 中的 Python 文件"
```

预期结果：CLI 将提示词发送到本地 API，API 代理到 OpenRouter，OpenRouter 返回响应，API 转发它，CLI 显示结果。这是完整的集成路径。

### 需要调整的内容

**API 合约不对齐**是最常见的。CLI 构建时假设它直接调用 OpenRouter。API 构建时接收看起来像 OpenRouter 调用的请求，在格式化请求或解析响应方面的小差异可能需要修正。API 的代理路由中调整，而非 CLI 的客户端中——CLI 与 OpenRouter 的接口是正确的且经过测试的。

**Web 项目的 Tailwind 配置**：Vite + Tailwind 配置有时需要额外的步骤（`npx tailwindcss init -p`），Web 智能体可能运行了也可能没有。若 Web 构建成功但样式未在浏览器中应用，这很可能是原因。

**API 中的 CORS 配置**：运行在`localhost:5173`上的 Web 仪表盘须调用运行在`localhost:8000`上的 API。FastAPI 的 CORS 中间件必须配置为允许此源。若未配置，浏览器请求将因 CORS 错误而失败。

### 合并提交

一旦集成测试通过：

```bash
cd opentaion/
git merge cli-development --no-ff -m "合并 CLI：智能体循环 + 上下文管理器 + OpenRouter 客户端"
git merge web-development --no-ff -m "合并 Web：Vite + React 脚手架 + 仪表盘"
git merge api-development --no-ff -m "合并 API：FastAPI 路由 + SQLAlchemy 模型"
```

### 刚刚发生了什么

三个组件已集成。每个都独立构建、独立测试，现在作为一个互联系统运行，并行构建实现了顺序构建能达到的效果

### 里程碑 M10：CLI + Web + API 并行构建完成

OpenTalon 中，此里程碑标志着核心组件构建的完成。三个独立开发的软件片段现在形成了一个功能系统。集成需要上述检查清单——合约验证、环境变量对齐，及根据典型的并行构建结果，一个 CORS 配置修复。合并提交在仓库中。集成测试通过。
