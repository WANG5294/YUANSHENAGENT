## 第 10.2 节：测试作为规范

上一节确立了测试智能体开发中为何重要。本节确立如何编写它们。原则狭窄且意义深远：测试应该描述行为，而非实现。一个与实现细节耦合的测试，是一个实现被重构时（即使行为未变）也会失败的测试。一个描述行为的测试能重构中存活下来，并继续捕获回归。

### "行为，而非实现"的含义

考虑上下文管理器的截断函数。一个与实现耦合的测试可能看起来像：

```python
def test_context_manager_uses_deque():
    ctx = ContextManager(max_tokens=1000)
    # ... checks that the internal data structure is a deque
    assert isinstance(ctx._messages, deque)
```

若实现从 deque 改为 list，即使截断仍然正常工作，这个测试也会失败，测试的是实现，而非行为。

同一组件的一个行为导向的测试：

```python
def test_context_manager_truncates_when_over_limit():
    ctx = ContextManager(max_tokens=100)
    # Add messages until over limit
    for i in range(20):
        ctx.add({"role": "user", "content": f"Message {i}" * 10})
    # Verify that the context fits within the limit
    assert ctx.total_tokens() <= 100
    # Verify that the most recent messages are preserved
    messages = ctx.get_messages()
    assert messages[-1]["content"].startswith("Message 19")
```

这个测试不关心截断是如何实现的。它关心截断发生了、结果符合限制、且最近的消息被保留。这个测试能任何满足这三个行为的实现中存活。

### 命名作为规范

测试名称传达了意图。命名约定`test_{subject}_{condition}_{expected_outcome}`使测试名称可读为规范：

```python
def test_context_manager_truncates_when_over_limit():
    ...

def test_context_manager_preserves_system_prompt_on_truncation():
    ...

def test_openrouter_client_retries_on_429():
    ...

def test_openrouter_client_raises_after_max_retries():
    ...

def test_agent_loop_terminates_after_max_turns():
    ...

def test_agent_loop_calls_tool_when_requested():
    ...
```

不读测试体，只读这些测试名称，开发者就能理解模块保证了什么。此即规范的价值：测试套件是代码做什么的文档，以一种自我验证的形式呈现。

### 测试先于实现

TDD 工作流颠倒了正常顺序。不是先实现函数然后编写测试验证它，而是先编写测试然后实现函数来使它们通过。

给 Claude Code 的指令：

```
这是上下文管理器的测试文件：

[粘贴测试文件]

在 cli/src/opentaion/context.py 中实现最小代码以使这些测试通过。
不要修改测试文件。不要添加未在此处测试的功能。
```

这些约束是关键。"最小代码"防止 Claude Code 实现一个添加规范中没有的功能的通用解决方案。"不要修改测试文件"防止 Claude Code 用更改测试内容使测试通过。"不要添加未在此处测试的功能"保持范围紧凑。

### 约束对 Claude Code 的影响

没有最小代码约束，Claude Code 倾向于追求完整性。被要求实现一个上下文管理器，它将实现令牌计数、截断、压缩、汇总，可能还有持久化——因为所有这些都是上下文管理器可能合理做的事情。测试文件指定了四个行为；Claude Code 实现了十二个。

有了约束，Claude Code 确切地实现测试要求的内容。若测试没有指定汇总，实现就不会包含汇总。范围由测试定义，而非由 Claude Code 认为可能有用的内容。

这是一个特性，而非限制。即 TDD 循环结束时实现的是满足规范的最小实现，中的一切都因测试要求它才存在的。没有什么是推测性的。

### 与 OpenTalon 的关联

OpenTalon 中，CLI 组件的测试文件将实现开始前编写。上下文管理器的测试文件指定了截断行为和系统提示保留。OpenRouter 客户端的测试指定了重试逻辑，测试文件是指导致实现的规范——它们比 SPEC.md 文本更精确，比自然语言更可验证。当第三部分的实现章节开始时，测试存在，规范存在；还无的是同时满足两者的代码。
