## 第 10.1 节：为什么 TDD 是最有效的智能体模式

规划优先的工作流解决了会话间一致性问题：每个会话以一个书面规范开始，防止架构漂移。但规范本身不能防止另一种失败模式：看起来合理但错误的代码。

Claude Code 产生的代码看起来是正确的。驱动文本生成的令牌预测机制产生局部一致输出——看起来像工作代码、遵循既定模式并能编译的代码。"看起来正确"和"实际上正确"是两回事。没有正确性的客观度量，Claude Code 就无法发现差异，只有编译成功且与给定的模式匹配的反馈。

测试提供了这个客观度量。

### 核心洞察

没有测试，Claude Code 可用的唯一反馈是语法层面的：代码能解析吗？导入正确吗？与类似代码的模式匹配吗？所有这些代码功能上有错误时都能成功。一个应该为无效 API 密钥返回错误、但却返回带有空体的成功响应的函数，能通过所有语法检查。

有了测试，反馈就是行为层面的：代码是否按照指定要求工作？一个发送无效 API 密钥并断言特定错误响应的测试，若实现返回一个成功体，则会失败。Claude Code 读取失败信息，理解它所产生的内容与测试要求之间的差距，并纠正实现。

这不是一个微小的改进。这是一个生成正确代码的代理与一个验证自身输出的代理之间的区别。

### 对齐观察

这里有一个更深的属性值得命名。Claude Code，像任何语言模型一样，被优化为生成对人类读者看起来正确的输出。训练数据中，看起来正确的代码通常确实是正确的。模型已学会了生成看起来正确的代码，且这种倾向是强烈的。

测试引入了评估者。测试不在乎代码看起来什么样。它只在乎代码是否按指定方式行为。这是一个比"看起来正确"更严格的标准，且这是生产软件所要求的标准。TDD 并不覆盖 Claude Code 产生合理代码的倾向——它增加了一个验证步骤，将合理与正确区分开来。

### 规范益处

一个编写良好的测试是比自然语言更精确的规范。比较：

- 自然语言："函数应该优雅地处理速率限制错误"
- 测试：`assert response.retry_count == 3 and response.final_status == 429`

自然语言留下了开放问题："优雅地"是什么意思？它重试吗？多少次？重试用尽后返回什么？测试以零歧义回答了所有这些问题。当 Claude Code 收到测试作为规范时，它确切地知道要实现什么——不是"优雅地"对人类读者意味着什么，而是什么行为能通过测试。

### 回归益处

测试捕获跨会话的破坏。当 Claude Code 第六个会话中修改一个函数以添加新功能，而该修改破坏了第三个会话中工作的一个边缘情况时，第三个会话的测试套件立即捕获了回归。没有测试，回归会更晚时被发现——当特性生产中失败时，或当不同会话中的相关测试碰到被破坏的行为时。

这是测试覆盖率的复合收益。每个编写的测试都是一个断言，每次后续修改中、在每个会话中、代码库的整个生命周期中被检查。

### 诚实的告诫

TDD 提前增加时间。实现前编写测试比直接编写实现更慢，因为任何东西工作前你要写两倍的代码。一个为书构建项目的独立开发者来说，这个前期成本是可见且真实的。

回报是调试更便宜。一个能立即捕获回归的测试套件，比一个每次更改都需要跨多个特性进行手动验证的代码库维护成本更低。时间是提前一次性支付的（测试编写中），而非反复支付的（调试会话中）。

即使有了 TDD，仍然存在两种失败模式：测试可能出错（测试了错误的行为），且 Claude Code 可能以意外使测试通过（实现了测试而非行为）。两者都须关注。第 10 章直接解决了它们。

### 与 OpenTalon 的关联

OpenTalon 中，TDD 是 CLI 实现的主要质量机制。上下文管理器被编写前，指定其行为的测试将存在。OpenRouter 客户端被编写前，验证重试逻辑的测试将存在。实现不是代码看起来正确时才"完成"——而是测试通过时才完成，区分第三部分中贯穿始终，且第 10.5 节在上下文管理器上完整演示了它。
