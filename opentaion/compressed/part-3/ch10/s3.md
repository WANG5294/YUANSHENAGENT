## 第 10.3 节：与 Claude Code 的红-绿-重构循环

TDD 循环有三个阶段：RED（一个指定所需行为的失败测试）、GREEN（使测试通过的最小实现）和 REFACTOR（保持测试通过的改进代码）。对人工开发者来说，这些阶段单个上下文中发生，具有连续的记忆。当 Claude Code 作为实现者时，每个阶段都需要明确的指令来过渡。

本节追踪 OpenTalon 一个组件——OpenRouter 客户端中的重试逻辑——的完整循环，从失败测试到干净的实现。

### RED：失败测试

编写任何实现代码前，先编写测试。该测试指定了确切所需的行为。

```python
# cli/tests/test_llm.py
import pytest
from unittest.mock import AsyncMock, patch
from opentaion.llm import OpenRouterClient

@pytest.mark.asyncio
async def test_openrouter_client_retries_on_429():
    """Client retries up to 3 times when receiving 429 responses."""
    client = OpenRouterClient(api_key="sk-test-key")

    call_count = 0
    async def mock_post(*args, **kwargs):
        nonlocal call_count
        call_count += 1
        if call_count < 3:
            # Return 429 for first two calls
            mock_response = AsyncMock()
            mock_response.status_code = 429
            mock_response.headers = {"Retry-After": "1"}
            return mock_response
        # Return success on third call
        mock_response = AsyncMock()
        mock_response.status_code = 200
        mock_response.json = AsyncMock(return_value={
            "choices": [{"message": {"content": "response"}}]
        })
        return mock_response

    with patch("httpx.AsyncClient.post", side_effect=mock_post):
        response = await client.complete("test prompt")

    assert call_count == 3
    assert response.content == "response"

@pytest.mark.asyncio
async def test_openrouter_client_raises_after_max_retries():
    """Client raises after 3 failed attempts."""
    client = OpenRouterClient(api_key="sk-test-key")

    async def always_429(*args, **kwargs):
        mock_response = AsyncMock()
        mock_response.status_code = 429
        mock_response.headers = {"Retry-After": "1"}
        return mock_response

    with patch("httpx.AsyncClient.post", side_effect=always_429):
        with pytest.raises(RuntimeError, match="Rate limit exceeded after"):
            await client.complete("test prompt")
```

运行测试：
```bash
cd cli && uv run pytest tests/test_llm.py -v
```

期望输出：
```
FAILED tests/test_llm.py::test_openrouter_client_retries_on_429 - ModuleNotFoundError: No module named 'opentaion.llm'
FAILED tests/test_llm.py::test_openrouter_client_raises_after_max_retries - ModuleNotFoundError: No module named 'opentaion.llm'
```

这是 RED 状态。测试失败因`opentaion.llm`无。失败原因确认了测试本身是正确的——它不因错误的断言而失败，而因实现缺失。

### GREEN：最小实现

给 Claude Code 的指令：
```
在 cli/src/opentaion/llm.py 中实现最小代码，以使 cli/tests/test_llm.py
中的测试通过。不要修改测试文件。不要添加测试未要求的任何功能。
```

Claude Code 读取测试，识别所需的接口（带有异步`complete()`方法的`OpenRouterClient`429 重试逻辑，超过最大重试次数后的异常），并实现它：

```python
# cli/src/opentaion/llm.py
import asyncio
import httpx

class OpenRouterClient:
    MAX_RETRIES = 3

    def __init__(self, api_key: str, model: str = "deepseek/deepseek-r1"):
        self.api_key = api_key
        self.model = model
        self.base_url = "https://openrouter.ai/api/v1"

    async def complete(self, prompt: str) -> "CompletionResponse":
        for attempt in range(self.MAX_RETRIES):
            async with httpx.AsyncClient() as client:
                response = await client.post(
                    f"{self.base_url}/chat/completions",
                    headers={"Authorization": f"Bearer {self.api_key}"},
                    json={"model": self.model, "messages": [{"role": "user", "content": prompt}]},
                    timeout=60.0,
                )
            if response.status_code == 429:
                if attempt < self.MAX_RETRIES - 1:
                    retry_after = int(response.headers.get("Retry-After", "2"))
                    await asyncio.sleep(retry_after)
                    continue
                raise RuntimeError(f"Rate limit exceeded after {self.MAX_RETRIES} attempts")
            response.raise_for_status()
            data = response.json()
            return CompletionResponse(content=data["choices"][0]["message"]["content"])

class CompletionResponse:
    def __init__(self, content: str):
        self.content = content
```

运行测试：
```
PASSED tests/test_llm.py::test_openrouter_client_retries_on_429
PASSED tests/test_llm.py::test_openrouter_client_raises_after_max_retries
```

GREEN。在此提交：`git commit -m "Add OpenRouter client with retry logic (GREEN)"`。

### REFACTOR：清理

GREEN 实现通过了测试，但存在问题：`CompletionResponse`类很单薄，可以是一个数据类；重试休眠依赖于`Retry-After`头但没有合理的默认值；httpx 客户端每次调用时重新创建。

指令：
```
重构 cli/src/opentaion/llm.py 以提高清晰度和正确性，同时保持
cli/tests/test_llm.py 中的所有测试通过。不要添加新功能。
```

Claude Code 进行重构，保持测试套件全程为绿色。重构后再次提交。

### 提交节奏

该循环产生两次提交：一次在 GREEN（工作实现，可能粗糙），一次在 REFACTOR（干净实现）。两者都是检查点。若重构破坏了什么，`git reset`回到 GREEN 提交。测试套件是仲裁者——如果测试通过，状态就是安全的；若失败，说明出了问题。

### 与 OpenTalon 的关联

OpenTalon 中，这个确切的循环——编写测试，确认 RED，实现到 GREEN，重构，每个阶段提交——为 CLI 中的每个组件运行：上下文管理器、OpenRouter 客户端、工具集和代理循环本身。测试组件存在前就存在，故从第一次迭代开始，每个实现都是可验证的。到第 10.5 节时，测试套件已足够全面，可以作为第三部分其余部分的回归基线。
