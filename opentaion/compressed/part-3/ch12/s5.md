## 第 12.5 节：黄金集：你的智能体系统的回归任务

软件工程师有回归测试（regression tests）。他们在每次更改后运行，以验证之前工作的内容仍然有效。智能体开发系统需要同样的纪律，但应用在更高的一层上。不是针对代码——测试覆盖了代码——而是针对流程。黄金集（Golden Set）是开发系统本身的回归套件。

概念很简单：一个固定的任务列表，智能体系统应始终能正确完成。在对 CLAUDE.md 进行任何重要更改后运行它。在升级 Claude Code 后运行它。本书的每个新部分开始时运行它。若列表中的任何任务失败或产出质量下降，说明系统中的某些东西发生了漂移。

### 什么属于黄金集

黄金集任务具有三个属性。第一，它必须锻炼一个真实的系统能力——真正测试开发环境是否正常工作的内容。第二，它必须产生可观察的输出——通过/失败或可质量评估的结果。第三，它必须是稳定的——预期行为不应随代码库增长而改变，只有系统是否仍能交付它才会改变。

不好的黄金集任务："向 CLI 添加一个新功能。"这太开放了，无法有稳定的预期行为。

好的黄金集任务："向 CLI 添加一个`--version`标志，打印`cli/src/opentaion/__init__.py`中的版本。"这有明确的成功标准，锻炼了特定的系统路径（技能调用、Click 命令注册），且无论其他哪些内容发生了变化，都应该以相同工作。

### OpenTalon 在里程碑 M11 的黄金集

```
黄金集：OpenTalon（第 12 章状态）

任务 1 —— 技能测试：
"向 CLI 添加一个名为 `explain` 的新 Click 命令。它应
接受一个文件路径参数并打印该文件功能的摘要。
使用 /opentaion-component 技能。"

成功标准：
- /opentaion-component 技能被调用
- 命令在 cli/src/opentaion/main.py 中注册
- 更改后 uv run pytest tests/ 通过
- 未修改 web/ 或 api/

任务 2 —— 上下文导航测试：
"找出 CLI 代码库中 token 计数的计算位置
并解释其逻辑。不实现任何东西。"

成功标准：
- 智能体在读取之前使用 Grep（在工具调用中可观察）
- 识别出正确的文件和行号
- 解释准确对应实际实现
- 在到达答案之前上下文预算未被耗尽

任务 3 —— TDD 强制执行测试：
"为 OpenRouter 客户端收到 503 响应的情况
编写一个失败的测试，然后实现使其通过的最少代码。"

成功标准：
- 测试在实现代码之前编写
- 测试在实现前运行时失败（RED 状态）
- 编写最少代码以通过（GREEN 状态，无过度工程）
- 测试和实现分别提交

任务 4 —— 集成测试：
"运行 cli/ 和 api/ 的完整测试套件并报告任何
失败，包括失败的测试名称和错误消息。"

成功标准：
- 使用正确的命令执行测试（uv run pytest）
- 所有测试通过，或正确定位失败
- 当存在失败时，不会虚假报告"所有测试通过"
```

### 何时运行黄金集

四个触发条件：

**在 CLAUDE.md 发生任何重大更改后。** 新规则、删除的规则及现有规则的重新措辞都可能以并非立即明显改变 Claude Code 的行为。一个看似澄清的规则可能引入了一个冲突。

**在 Claude Code 升级后。** 版本变化可能改变默认行为。一个依赖特定工具调用格式的钩子可能中断。一个使用已弃用命令语法的技能可能停止工作。

**每个新部分开始时。** 本书的第一部分到第六部分各自引入了新的系统能力。开始第四部分前，确认第一部分到第三部分的所有功能仍然有效。黄金集记录了基础是完好的。

**当系统感觉迟钝或不可靠时。** 退化并不总是二元的。有时系统仍然产生输出，但输出质量较低——更多幻觉、更多偏离规格的行为、更多偏离 CLAUDE.md 指令的漂移。黄金集使这一点可观察。

### 失败即诊断

黄金集的价值不在于通过的情况，而在于失败。任务 1 失败（技能未被调用、修改了错误的文件）指向 CLAUDE.md 漂移或技能文件损坏。任务 2 失败（未使用 Grep、上下文耗尽）指向导航指令的侵蚀。任务 3 失败（测试实现后编写）指向 TDD 强制执行的崩溃。任务 4 失败（虚假的成功）指向测试命令配置错误。

每个失败都有特定原因。黄金集定位了漂移。

### 在 OpenTalon 中

OpenTalon 中，当所有四个黄金集任务集成的 50+ 文件代码库上通过时，达到里程碑 M11。该里程碑确认了本章中的上下文管理策略正在工作：域路由指令正确限定范围、导航规则生效、技能在并行构建和合并后仍然功能正常、TDD 强制执行在从 15 个文件增长到 50+ 个文件后得以保持。黄金集与测试套件一起提交到`docs/golden-set.md`——它是系统必须始终能做到什么的文档，而非系统曾经做到过的记录。第三部分在此结束。第四部分以 BMAD 和构建 CLI 现在连接的 Web 平台的工作开始。

### 刚刚发生了什么

黄金集为"系统工作"提供了、可重复的定义。不是一种感觉。不是一段回忆。四个具有可观察成功标准的任务，在集成的代码库上运行。全部四个通过意味着本章中的上下文管理策略不仅在理论上是合理的——它们真实条件下也成立。

此模式与单元测试类似。单元测试定义代码必须始终做什么。黄金集定义开发过程必须始终做什么。两者都是回归预防。两者都值得你需要它们前编写。
