## 第 12.2 节：子智能体作为上下文容器

上下文从两个方向填满。第 12.1 节描述了累积过程——对话历史和工具调用输出的逐步增长。但还有一种尖峰：当一个任务须写任何代码前读取二十个文件时。压缩和连续性标志解决了第一个问题。第二个问题，有一个完全不同的工具。

子智能体（Subagents）不仅仅用于并行构建，是上下文容器——有界的作用域，吸收大量信息负载，只返回父智能体继续工作所需的内容。

### 关键见解

当 Claude Code 会话使用 Task 工具派生一个子智能体时，子智能体自己的上下文窗口中运行，可读取文件、运行命令、积累工具调用输出并进行长时间的推理。当它完成时，父会话只收到一条消息：子智能体的最终输出。而非文件内容。不是中间推理。不是工具调用历史。只有最终消息。

此即使子智能体对上下文管理有用的特性。考虑直接读取一个源文件目录的成本：

```
父会话读取 cli/src/ —— 15 个文件，每个约 3,000 token
消耗的上下文：约 45,000 token
实际需要什么：哪些文件包含 token 计数逻辑
```

现在考虑将同样的任务委托给子智能体：

```
父会话派生子智能体："读取 cli/src/ 中的所有文件。
识别哪些文件包含 token 计数逻辑并
总结它们的接口。只返回摘要。"

子智能体消耗：约 45,000 token（在自己的上下文窗口中）
父会话接收：约 500 token 摘要
父会话中消耗的上下文净额：约 500 token
```

子智能体消耗自己的上下文，而非父智能体的。父会话接收到有针对性的摘要，而非原始文件内容。

### 探索模式

该模式有一个标准形式。在父会话中，实现任何需要架构理解的功能前：

```
Task 工具调用：
"你的任务只是探索——不要编写任何代码。
读取 cli/src/opentaion/ 中的所有文件。识别：(1) 上下文管理器
在哪里，(2) token 计数函数签名是什么，
(3) 哪些模块导入了上下文管理器。
返回不超过 300 字的摘要。不要包含文件内容，
不要包含代码块——只包含架构地图。"
"

对子智能体的指令明确指定了输出格式。不加约束的子智能体会精确返回它所读取的内容——这违背了目的。将其约束为摘要。说明字数限制。指出需要回答的三个问题。

父会话接收一个 250 字的相关代码地图。后续的实现会话以架构理解和几乎完整的上下文预算开始。

### 权衡

子智能体委派（delegation）并非免费。每次子智能体启动会带来：

- **延迟（Latency）**——子智能体必须被派生、运行并完成，父会话才能收到其输出。一个读取 20 个文件的子智能体需要两到四分钟。
- **成本（Cost）**——子智能体的 token 消耗需要计费。在子智能体中读取 15 个文件的成本与直接读取它们相同。节省的是父会话的上下文，而非总的 token 预算。

这种权衡决定了何时委派是合适的：

| 任务 | 使用子智能体？ | 原因 |
|------|---------------|------|
| 实现前读取 2-3 个文件 | 否 | 直接读取花费 6,000 token；委派开销超过节省 |
| 实现前探索整个模块（15+ 个文件） | 是 | 直接读取花费 45,000+ token；子智能体摘要花费 500 |
| 运行测试套件并返回摘要 | 是 | 测试输出可能有数千行；父会话需要通过/失败计数 |
| 编写单个函数 | 否 | 无需探索；委派增加不必要的延迟 |
| 审计某 API 在整个代码库中的所有使用 | 是 | Grep 能找到引用，但理解每次使用需要读取 |

启发式：当信息收集在父会话中会消耗超过 10,000 token，且父会话需要的是综合而非原始数据时，使用子智能体。

### 何时不使用

需避免两种故障模式：

**过度委派（Over-delegation）。** 为每一次小的读取都派生子智能体是反射性的开销。一个直接读取两个文件模块的智能体使用 5,000 token。一个派生子智能体来读取这两个文件并返回摘要的智能体，使用子智能体的 5,000 token 加上延迟和子智能体派生开销。对于小范围的工作，直接读取更快且更便宜。

**输出规格不明确（Under-specifying the output）。** 一个被指示"探索 CLI 代码库"却没有输出约束的子智能体会返回它的全部发现——可能比直接读取成本更高的 token 数量。子智能体模式需要显式的输出规格：返回什么、省略什么、使用什么格式。

### 在 OpenTalon 中

在 OpenTalon 中，子智能体作为上下文容器的模式出现在两个重复的时间点。在实现任何跨组件功能（涉及 cli/、api/ 和 web/ 的更改）之前，一个探索性子智能体会绘制三个组件的当前状态地图。父会话接收地图并在完整的上下文预算下进行实现。在并行构建之后，当代码库超过 50 个文件时，该模式成为跨多个模块的功能工作的默认方式。第 12.4 节涵盖了后续大代码库阶段的完整导航策略。
