## 第 12.3 节：文档中的渐进式披露

子智能体模式活跃会话期间管理上下文。渐进式披露（Progressive disclosure）将上下文作为代码库本身的结构属性管理——这是一种适用于 Claude Code 可能读取的每个文件的设计原则。

原则是：只提供智能体当前需要的上下文，并附上清晰指向更多细节所在位置的指引。应用于文档、应用于 CLAUDE.md 文件、应用于规格说明，即构建一个分层的信息架构，Claude Code 按需导航，而非到达时详尽加载。

### 一切可用的问题

对上下文限制的自然反应是让最重要的信息最大程度可访问——将其合并、前置、放根目录的 CLAUDE.md 中，以便它总是被读取。这是错误的反应。

一个 400 行的 CLAUDE.md 每次会话开始时被完整读取，无论该会话是否涉及第 200–400 行中记录的任何边缘情况。一个 400 行的规格说明实现单个函数前被完整读取，即使实现只需要第 40–70 行。在一开始就加载所有内容，工作开始前就消耗了上下文预算。

替代方案不是减少文档。而是有组织的文档——一个引用层次结构，需要时将 Claude Code 引导到所需内容。

### 应用于 CLAUDE.md 文件

根 CLAUDE.md 包含适用于整个项目的规则，无论触及哪个组件。组件级的 CLAUDE.md 文件包含组件特定的细节，只处理该组件时才重要。

当 Claude Code 启动一个会话用于处理 CLI 时，它读取根 CLAUDE.md。若工作涉及`cli/`它读取`cli/CLAUDE.md`。除非任务明确跨越组件边界，否则它不会读取`web/CLAUDE.md`或`api/CLAUDE.md`。

```
根 CLAUDE.md（约 80 行）——始终加载
├── 技术栈（权威列表）
├── 语气和风格规则
├── 禁止规则
└── "有关组件特定规则，请阅读相应组件的 CLAUDE.md"

cli/CLAUDE.md（约 40 行）——仅在进行 CLI 工作时加载
├── uv 命令（不是 pip，不是 poetry）
├── Click 命令注册模式
├── 测试约定（pytest，不是 unittest）
└── "有关 API 合约细节，请参阅 api/CLAUDE.md"

api/CLAUDE.md（约 40 行）——仅在进行 API 工作时加载
├── FastAPI 路由注册模式
├── SQLAlchemy 异步会话约定
├── 迁移命名约定
└── "有关 CLI ↔ API 集成测试，请参阅 integration/SPEC.md"
```

分层结构意味着 CLI 会话从 120 行相关上下文开始，而非 160 行加上不相关的 API 细节。随着代码库增长，新的组件特定规则放入相应的组件文件中。根 CLAUDE.md 保持在 100 行以内。

### 应用于文档注释

函数文档字符串（docstring）应描述函数的功能及其合约。若实现说明是必要的——为什么选择特定算法、需要特殊处理的边缘情况——这些放在由文档字符串引用的单独位置。

```python
async def count_tokens(text: str, model: str) -> int:
    """
    计算给定模型的文本 token 数量。

    对 OpenAI 兼容模型使用 tiktoken，对其他模型使用字符估计
    （字符数 / 4）。有关完整模型覆盖地图和准确性基准，
    请参阅 docs/token-counting.md。
    """
```

实现工作中读取此函数的 Claude Code 会得到合约——函数做什么、传入什么、期望什么返回。若需要准确性基准，它读取`docs/token-counting.md`。若不需要，它不必付出上下文成本。

反模式是文档字符串中内联包含实现说明：

```python
# 反模式：所有内容都在文档字符串中
async def count_tokens(text: str, model: str) -> int:
    """
    计算 token 数量。对 gpt-*、claude-*、deepseek-* 使用 tiktoken，
    使用 cl100k_base 编码。对于 llama-*、qwen-* 和 phi-* 模型，
    使用字符估计回退，因为 tiktoken 没有它们的词表。
    估计假设每 token 4 个字符，对英文文本准确度在 15% 以内，
    但对代码（每 token 6-8 字符）和 CJK 文本（每 token 1-2 字符）
    会退化。有关准确度分布，请参阅测试套件
    tests/test_token_counting.py...
    [还有 80 多行]
    """
```

无论 token 计数准确性是否与当前任务相关，这个文档字符串都会 Claude Code 读取文件时被完整读取。

### 应用于规格说明

第 9 章编写的 SPEC.md 在功能层面描述 CLI——它做什么、使用什么工具、接受标准是什么。每个故事（story）的实现细节存放故事文件中（在第 16 章引入），Claude Code 实现该故事时一次读取一个。

SPEC.md 始终相关。一个关于"实现 --model 标志"的故事文件只实现该标志时相关。开始一个会话前只加载 SPEC.md 花费 300 token。加载 SPEC.md 加上所有 24 个故事文件花费 8,000 token，且大部分内容永远不会被用到。

实现这一点的结构：SPEC.md 引用故事存放的位置。故事文件存依赖关系时相互交叉引用。Claude Code 沿着引用链深入到任务所需的深度，不再深入。

### 在 OpenTalon 中

OpenTalon 中，渐进式披露从一开始就内置于仓库结构中。根 CLAUDE.md 明确说明："有关组件特定的规则，请阅读相关组件的 CLAUDE.md。除非正在该组件中积极工作，否则不要加载组件文件。"这条指令是承重性的——它防止了读取一切可用内容的自然倾向。cli/CLAUDE.md、api/CLAUDE.md 和 web/CLAUDE.md 各自保持在 50 行以内。当一个规则超出此范围时，说明该规则应属于一个单独文档，从 CLAUDE.md 中引用，而非放 CLAUDE.md 本身中，纪律使信息架构项目通过第四部分和第五部分增长时保持可导航性。
