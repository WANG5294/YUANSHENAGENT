## 第 9.5 节：何时打破模式

本章描述的规划优先的纪律有其实际成本。探索→规划→编码→提交循环需要四十五分钟到一个小时，然后才能编写一行生产代码。计划模式为一个会话增加十五分钟。范围不明确的大型特性，这些成本被它们产生的连贯性抵消。小的、完全指定的更改，同样的成本变成了纯粹的额外开销。

这种纪律因有价值，正因它不是普遍适用的。将其应用于每个更改——包括修复注释中的拼写错误、增加版本号、添加缺失的导入——是官僚主义的浪费，使工作流感觉像是惩罚而非赋能。当工作流感觉像是惩罚时，开发者就会停止遵循它。当开发者选择性地停止遵循它时，他们开始在中型事情上打破它。然后是大型事情。纪律彻底崩溃。

解决方案是一个诚实的启发式方法。

### 启发式方法

**这个更改会影响你无意触及的任何代码吗？**

若答案是肯定：遵循完整的工作流。任何有依赖关系的更改——可能波及代码库、触及从多个地方调用的函数、修改跨多个表使用的 schema——都需要一个计划。

若答案是否定：直接编码，跳过探索或规划阶段。一个局限于一个位置、任务描述中完全指定且没有可能副作用的更改，可直接跳到编码。

需要完整工作流的更改示例：
- 添加新的 API 端点（影响路由、认证、数据模型、测试）
- 更改函数签名（影响所有调用者）
- 修改上下文压缩算法（影响所有长会话）
- 重构 OpenRouter 客户端（影响所有 LLM 调用）

不需要的更改示例：
- 修复注释中的拼写错误
- 更改常量值（如果该常量只有一个使用处）
- 为独立函数添加缺失的文档字符串
- 更新`pyproject.toml`中的版本号

### 侵蚀的危险

当你为第一类情况打破模式时，你也在为第二类情况打破它。启发式方法的区分对于防止这种侵蚀最重要。"这可能影响我无意触及的东西吗？"此问题的答案开始变成"可能不会"而非"肯定不会"的那一刻，启发式方法就已被破坏了。

妥协的情况：某件事可能是一个小更改，但有 20% 的不确定性。正确的反应不是跳过计划。而是使用计划模式（只读探索），而不创建完整的 PLAN.md 文档。花两分钟计划模式下阅读相关文件，确认范围与预期一致——这是最小可行的检查点，能捕获那 20% 的情况，而没有完整计划的额外开销。

诚实的回答：有经验的实践者能凭直觉感受到这种区别。适合直接编码的小更改有一种特定的品质——你开始前就确切地知道你要做什么，若 diff 包含任何你没有预料到的东西，你会感到惊讶。任何不具备这种品质的事情，至少值得做一个简短的计划模式检查。

OpenTalon 中，规划优先的模式完全适用于架构性工作（CLI 代理循环、上下文管理器、OpenRouter 客户端），而完全不适用于机械性任务（添加常量、修复测试断言）。第三部分中的会话任务提示中明确命名其模式——"此会话使用计划模式"或"直接编码：此更改局限于一个文件"——故模式始终是刻意的选择，而非默认值。

### 里程碑 M8：OpenTalon CLI SPEC.md 完成

SPEC.md 已编写并提交。Claude Code 已阅读它。CLI 实现的架构已决定。用在一个新的 Claude Code 会话中运行以下验证确认这个里程碑：

```
不要写任何代码。阅读 cli/SPEC.md 并总结：
1. CLI 用一句话能做什么
2. 它暴露的六个工具
3. "完成"的验收标准
4. 非目标部分中最重要的两个约束
如果规范中有任何不明确之处，请询问我。
```

Claude Code 应该产生一个基于实际 SPEC.md 内容的总结，而非基于记忆或推断。若总结准确，说明 SPEC.md 清晰，架构基础已就位。若 Claude Code 提出澄清性问题，这些问题识别了规范中应构建开始前解决的差距。

一个 Claude Code 不能准确总结的 SPEC.md，是一个不能有效约束 Claude Code 实现决策的 SPEC.md。继续前修复不明确之处。
