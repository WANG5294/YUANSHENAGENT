## 第 9.2 节：探索 → 规划 → 编码 → 提交工作流

前一节确定了问题：跨会话的架构不一致性。解决方案是一个会话之间创建共享上下文检查点的工作流。本节以具体的术语描述该工作流——不是作为抽象原则，而是作为一组具有定义顺序的特定提示和产物。

探索 → 规划 → 编码 → 提交工作流是"打开一个会话并开始编写"的结构化替代方案。每个特性花费的时间更长。产生的特性更好。任何大于两个文件更改的任务，额外的时间是值得的。

### 四个阶段

**探索（Explore）。** Claude Code 提出任何实现前先阅读代码库。关键的约束是明确的：用告诉 Claude Code 不要编写代码开始探索阶段。

```
现在还不要写任何代码。阅读现有代码库，确定该特性如何融入当前架构，
并就任何不明确的内容向我提出三个问题，然后我们再规划实现。
```

"问我三个问题"这个指令迫使探索阶段浮出不确定性，而非默默地解决它。Claude Code 倾向于上下文不明确时做出假设——要求提出问题的指令覆盖了这种倾向。这三个问题往往是实现中最重要的三个决策。

探索阶段在你回答了 Claude Code 的问题且双方都理解了范围和约束之后结束。

**规划（Plan）。** Claude Code 生成一个技术计划，作为 Markdown 文档，通常命名为`PLAN.md`或保存到 docs/ 目录。该计划描述：

- 哪些文件将被创建或修改
- 哪些函数或类将被实现，及其签名
- 哪些测试将验证正确性
- 任务之间存哪些依赖关系（X 必须在 Y 前完成）
- 任何未解决的问题或风险

计划不是代码。它是代码将是什么样子的结构化描述。仔细审查它。更改架构决策的时间是规划期间——而非在实现存在后。

计划模式（通过 Shift+Tab 或说"切换到计划模式"访问）是 Claude Code 在此阶段使用的工具：它将 Claude Code 限制为只读操作，防止计划被审查时过早实现。计划模式是规划阶段的工具，并非规划阶段本身。没有计划模式的规划讨论仍然是规划阶段；没有规划讨论的计划模式就是计划模式。

**编码（Code）。** Claude Code 一次一个任务地实现计划。实现阶段使用`acceptEdits`模式进行文件创建和编辑。PLAN.md 是参考文档：如果一个实现决策不清楚，Claude Code 读取计划而非做出假设。

实用的节奏：实现第一个单元（测试、数据模型或接口定义），验证它有效，然后继续下一个。不要要求 Claude Code 单个会话中实现整个计划。一个影响所有后续工作的错误的复合成本太高了。一次一个单元，每个单元之后验证。

**提交（Commit）。** 当实现完成且测试通过时，Claude Code 创建一个 git 提交。提交消息引用计划："实现上下文管理器——参见 docs/PLAN-context-manager.md。"提交是一个检查点：如果下一个会话做出的决策被证明不正确，上一个提交就是恢复点。

PLAN.md 与实现一起提交。它是意图的记录。未来的会话可以阅读它，以理解实现为什么看起来是这样的——代码本身没有表达的架构推理。

### 为什么提交是工作流的一部分

提交阶段不是行政开销。它是工作从"进行中"转变为"永久"的时刻。一个带有未提交更改的代码库，是一个当前会话中的任何错误都可能抹去之前会话工作的地方。一个已提交的代码库有一个安全保障网。

每个完成的工作单元后提交的纪律意味着，一个糟糕的会话——Claude Code 走向了错误的方向，或产生了以意外方式失败的代码——最多损失一个会话的工作，而非整个特性的累积工作。

### 实际的节奏

每个有意义的单元工作完成探索→规划→编码→提交循环。"有意义的单元工作"不是"一个函数"——那太小了，额外开销占主导，也不是"整个特性"——那太大了一个有意义的单元是特性的一个部分，具有清晰的接口、可识别的测试和一个会话可以完整实现的范围。

OpenTalon CLI：一个循环用于上下文管理器，一个循环用于 OpenRouter 客户端，一个循环用于代理循环，一个循环用于工具集。构建 CLI 核心需要四个循环——每个都有一个计划、一个测试套件、一个实现和一个提交。

### 与 OpenTalon 的关联

OpenTalon 中，这个工作流不是理论上的——第三部分中的每个实现章节都使用它。第 10 章使用探索阶段来理解实现上下文管理器前需要编写哪些测试。第 11 章使用规划阶段来协调三个并行代理同时构建 CLI、Web 和 API。第 12 章使用提交纪律创建回归任务的黄金集合，作流是使多会话、多组件开发变得可处理的结构。
