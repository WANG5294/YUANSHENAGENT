## 21.3 节：演进触发：何时变更

并非所有摩擦都表明系统出了问题。有些摩擦表明系统已超出了其最初的设计。区别很重要，因为修复方式不同：出问题的系统需修理；超出设计的系统需演进。
### 三个合理的触发因素

**新的项目类型。** 该系统是为 OpenTalon 的技术栈设计的：Python CLI，FastAPI API，Vite React 前端。添加一个 Go 服务，一个移动应用或一个机器学习组件，就引入了 CLAUDE.md 约定，技能和 BMAD 模板并非为此设计的项目类型。这里的摩擦是合理的——系统未经修改时无法处理新的类型。修复方式是演进：新的 CLAUDE.md 部分，新的技能，新的模板。
**工具可用性变化。** 新的 MCP 服务器使旧的变通方案变得多余。新的 Claude Code 能力使之前需自定义技能的模式成为可能。当一个更好的工具可用时，继续用旧的方法是选择保持低效。演进触发因素是认识到旧的变通方案可以被替换，而不是它正在失效。
**规模变化。** 第 12 章的策略对 50 个文件有效。到 500 个文件时，其中一些需重新校准——领域路由规则可能需更细化，子智能体委托模式可能需新的阈值，黄金集可能需覆盖新添加的子系统的额外任务。摩擦不是因为任何东西坏了；而故系统是为一个已不存在的代码库设计的。
### 需要避免的触发因素：无聊

无聊不是合理的演进触发因素。因为系统感觉陈旧而改变它——因为 CLAUDE.md 已六个月没变了，新的东西一定更好——是最危险的系统演进形式。它引入了稳定性却没有相应的好处。系统本来是工作的。新系统是未经测试和未知的。
区分无聊驱动的变更和合理演进的方法是黄金集。在任何提议的变更前后运行它。如果提议的变更改善了黄金集的结果，这是真正的改进。如果黄金集的结果相同，就不需变更——无论新方法感觉有多好。
### 演进过程

当出现合理的触发因素时：
1. **识别摩擦。** 要具体："来自新服务的 TypeScript 编译器错误被标记为 Python 约定违反"是具体的。"系统感觉不对劲"则不是。

2. **诊断根本原因。** 摩擦通常追溯到三个来源之一：一个不适用于新上下文的 CLAUDE.md 规则、一个做出新项目类型违反的假设的技能、或一个对新组件类型没有对应物的模板。

3. **设计变更。** 具体需要改变什么？一个新的 CLAUDE.md 部分？一个新技能？一个修改后的模板？变更应该是在解决根本原因所需的最小限度。

4. **实施并运行黄金集。** 进行变更，运行黄金集。如果黄金集结果改善：提交、记录、继续。如果结果相同或更差：重新考虑。

演进过程是迭代的，而非戏剧性的。丢弃整个组件并替换它们有时是正确的答案——第 21.2 节描述了 CLAUDE.md 中的约定错误需替换的情况。但替换应由旧方法失败的证据驱动，而非对新方法的偏好驱动。
### 在 OpenTalon 中

在 OpenTalon 中，第一个合理的演进触发因素可能出现在以下情况：当第二个开发者被添加到项目中（如果这会发生的话），或者当 Ollama 本地模型支持在 V2 中添加时。这两种情况都会改变运行假设：第二个开发者改变了 CLAUDE.md 规则适用于谁；Ollama 改变了技术栈和模型选择矩阵。两者都没有坏。两者都需演进。第 21.5 节描述了 V2 的方向；第 21.4 节以支配所有这一切的原则作为结束。
