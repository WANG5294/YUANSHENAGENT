## 第 15.2 节：乏味技术原则

Winston 在架构中做出了刻意的选择：选择 Vite 而非 Next.js，FastAPI 而非 Django，Supabase 而非自托管数据库，Railway 而非 Kubernetes。这些选择都倾向于较不令人兴奋的选项。这不是限制——这是原则。
乏味的技术（Boring technology）为独立开发者胜出，因为它易于理解。更好的文档，Stack Overflow 上更多已回答的问题，更少的意外边缘情况以及更低的维护负担。令人兴奋的选项通常更新，在特定方面更强大。但它也文档更少，更可能需变通方案，并且更可能引入你尚未遇到过的边缘情况下的故障模式。
### OpenTalon 的技术选择

**Vite 而非 Next.js。** Next.js 增加了服务端渲染，API 路由和基于文件的路由。这些都是真正有用的能力。对于 OpenTalon 的 Web 平台——一个没有 SEO 需求的三视图 SPA——它们是不必要的。Vite 生成一个静态包，在 30 秒内部署到 Vercel。Vite 的心智模型是"构建文件"；Next.js 的心智模型是"构建文件 + 运行服务器 + 处理路由 + 管理数据获取策略。"对于这个项目，额外的复杂性没有回报。
**FastAPI 而非 Django。** Django 是一个完整的 Web 框架——ORM，管理面板，模板引擎，会话管理，迁移。FastAPI 是一个 API 框架——路由，请求验证，异步支持。OpenTalon 的 API 是一组没有管理面板，没有模板，没有 Django 风格会话管理的端点。FastAPI 产生一个更精简的服务，更容易推理。"做什么"和"实际运行什么"之间的代码行越少越好。
**Supabase 而非自托管 PostgreSQL。** 自托管 PostgreSQL 需服务器，备份，监控，升级和连接池。Supabase 以托管服务的形式提供所有这些，外加认证和文件存储，并有免费套餐。权衡：依赖外部服务。对于 V1 的独立项目，这种权衡显然是值得的。Supabase 的免费套餐支持 OpenTalon 在头六个月将有的用户量。如果将来有必要，迁移到自托管 PostgreSQL 的路径是直接的。
**Railway 而非 Kubernetes。** Kubernetes 是面向拥有数千用户和复杂扩展需求的，多服务生产系统的正确部署平台。OpenTalon 的 API 是一个单一的 FastAPI 进程。Railway 通过 git push 从 Dockerfile 部署它。Kubernetes 需编写清单，管理集群，理解 Pod 网络，以及维护与实际产品无关的基础设施。Railway 移除了所有这些，且成本更低。
### 两个刻意的省略

两个常见选择明显不在 OpenTalon 的技术栈中，它们值得解释：
**不用 shadcn/ui。** shadcn/ui 是一个流行的 React 组件库。它不是传统的 npm 包——它用交互式 CLI 将各个组件作为可编辑的源文件添加到项目中。安装过程需配置 TypeScript 路径别名，运行交互式设置 CLI 以及逐个添加组件。一个有侧边栏，柱状图，表格和弹窗的仪表盘，这个开销超过了收益。Tailwind 直接处理布局和样式。这些组件足够简单，用 Tailwind 工具类实现它们比配置 shadcn/ui 更快。
**不用 React Router。** OpenTalon Web 平台正好有两种认证状态：未认证（显示注册页面）和已认证（显示仪表盘）。基于 Supabase 的 `onAuthStateChange` 回调的条件渲染在十行代码内完成处理。React Router 是一个为具有多个路由，复杂导航层次结构和基于 URL 状态的应用程序设计的路由库。OpenTalon 的 Web 平台不是那种应用程序。添加 React Router 意味着学习它的心智模型，管理路由配置，并将应用程序包裹在 Router 提供者中——所有这些都只是为了用更复杂的解决方案替换十行条件渲染代码。
### 乏味技术的测试

实用测试："我能在前三个搜索结果中找到问题的答案吗？"如果能，技术就足够乏味。Vite 的文档很优秀。FastAPI 的文档很优秀。Supabase 有每个集成模式的广泛指南。Railway 的文档很少，但该工具足够简单，文档已足够了。
当答案需翻过第一页搜索结果，阅读 GitHub 问题或在社区论坛中寻找变通方案时，技术已从乏味转为复杂。这是信号，而不是禁令——但对于 V1，这是值得注意的信号。
### 在 OpenTalon 中

在 OpenTalon 中，Winston 的乏味技术原则不仅适用于 OpenTalon Web 平台。它也适用于本书本身。本书中推荐的每个工具都有可用且文档完备的答案，回答读者将遇到的最常见问题。当读者遇到困难时，第一个搜索结果应该包含答案。这是对本书技术选择的设计约束，而不仅是对正在构建的软件。服务于 OpenTalon 用户的原则通过同样的逻辑服务于本书的读者。
