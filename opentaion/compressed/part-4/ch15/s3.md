## 第 15.3 节：代理/网关设计

架构文档中最具决定性的决策是代理设计。其他一切——数据库模式，API 端点，前端布局——都源于理解为什么所有 LLM 调用都流经 OpenTalon API，而不是直接从 CLI 到 OpenRouter。
代理不是技术上的便利。它是使用量跟踪成为可能的架构原则，而无需修改 CLI 的核心行为。
### 请求流程

当用户要求 OpenTalon 做某事时，请求路径如下：
```
用户输入：opentaion "修复 auth.py 中的类型错误"
     ↓
CLI（cli/）构造一个 OpenAI 兼容的聊天请求
     ↓
CLI 发送：POST http://localhost:8000/v1/chat/completions
           X-Api-Key: opentaion_abc123...
     ↓
API（api/）验证 API 密钥
API 代理：POST https://openrouter.ai/api/v1/chat/completions
           Authorization: Bearer ***
     ↓
OpenRouter 转发到所选模型
     ↓
模型响应通过 OpenRouter 流式返回
     ↓
API 从响应中提取 token 计数
API 写入 UsageLog 条目：user_id, model, input_tokens, output_tokens
API 将响应体流式返回给 CLI
     ↓
CLI 在终端显示流式响应
```

用户看到 token 在终端中出现。在后台，API 已计量了用量，而 CLI 无需知道关于计量的任何事情。
### OpenAI 兼容性

OpenTalon API 在 `/v1/chat/completions` 暴露 OpenAI 聊天补全接口。这不是巧合。第 10 章中构建的 CLI 用 OpenAI 兼容的端点直接调用 OpenRouter。将其指向 OpenTalon API 只需更改一个环境变量——基础 URL。
```bash
# 之前：CLI 直接调用 OpenRouter
OPENROUTER_BASE_URL=https://openrouter.ai/api

# 之后：CLI 调用本地代理
OPENROUTER_BASE_URL=http://localhost:8000
```

CLI 代码没有变化。第 10 章中测试过的 `OpenRouterClient` 类向任一 URL 发送相同的请求体。API 处理转换——接收 OpenAI 格式的请求，转发到 OpenRouter，返回 OpenAI 格式的响应。
这种兼容性正是为什么即使本书从未直接用过 OpenAI，OpenAI 接口也是正确的设计选择。该接口稳定，文档完善，并且受代码库中已有的客户端库支持。
### Token 计量

OpenRouter 响应在响应体中包含用量元数据：
```json
{
  "choices": [{ "message": { "content": "..." } }],
  "usage": {
    "prompt_tokens": 1240,
    "completion_tokens": 318,
    "total_tokens": 1558
  }
}
```

API 的代理处理器在将响应流式返回给 CLI 之前读取此元数据。Token 计数与用户 ID，模型名称和时间戳一起写入 `usage_logs` 表。CLI 永远不会看到这个过程——它接收 `choices` 数组并渲染内容。计量 CLI 和用户是不可见的。
这种分离是代理设计的关键属性。添加新的计量能力——成本计算，按模型费率，预算阈值——只需在 API 中更改。CLI 继续接收 OpenAI 格式的响应而不需修改。
### 代理启用的功能

代理设计解锁了没有它就不可能实现的能力：
- **使用量计量**——每次请求、每个模型、每个用户的 token 计数
- **密钥验证**——无效或已吊销的 API 密钥在 LLM 调用之前被拒绝
- **速率限制**——未来：在 API 层强制执行每用户请求限制
- **模型路由**——未来：为简单请求透明地路由到更便宜的模型
- **响应缓存**——未来：缓存相同的提示词以降低成本

这些功能都不需 CLI 知道它们的存在。代理是用户提示词和模型响应之间发生的一切的扩展点。
### 在 OpenTalon 中

在 OpenTalon 中，代理设计是 Web 平台完全值得构建的原因。没有代理，就没有 token 计量。没有计量，仪表盘就没有数据。架构文档显式地捕获了这种依赖：Web 平台的核心功能（用量可见性）完全依赖于 API 的计量功能，而计量功能又完全依赖于代理设计。第 16 章逐个故事地实现这个设计。第 15.4 节指定了代理和 API 暴露的所有端点。
