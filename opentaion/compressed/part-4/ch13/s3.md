## 第 13.3 节：两大支柱：规划与上下文工程

BMAD 建立在两大支柱之上。分别理解每一个以及它们如何配合，能使完整的方法论变得可理解，而非程序化。
第一个支柱是智能体规划（agentic planning）。第二个是上下文工程化开发（context-engineered development）。掉任何一个，方法论都会崩溃。没有上下文工程的规划会产生一个规格良好的系统，但实现智能体会误解它，因为每个故事只包含它们所需的一部分。没有规划的上下文工程会产生规格良好的故事，但系统设计很差。
### 支柱 1：智能体规划

在 BMAD 的规划阶段，专门的智能体与人类协作，产生三个文档：产品简介，PRD 和架构文档。每个智能体都有独特的功能。Mary（业务分析师）获取需求。John（产品经理）将它们结构化为一个包含史诗和故事的 PRD。Sally（UX 设计师）指定用户体验。Winston（架构师）设计系统。
每个智能体通过斜杠命令作为 Claude Code 角色被调用。每个产生一个或多个 markdown 文件。每个后续智能体读取前一个智能体的输出作为输入。基于文件的手递意味着规划阶段做出的每个决策都被捕获在一个文档中——不是在一个会被遗忘的聊天消息中，而是在一个持久存在，可编辑且将在数周后被实现智能体读取的文件中。
规划阶段的关键输出不仅仅是计划本身。它是在实现开始之前消除架构模糊性。当实现智能体（Bob，Amelia，Quinn）开始工作时，它们针对的是一个已在每个主要决策点由人类验证过的规格说明。
### 支柱 2：上下文工程化开发

第二个支柱解决了实现智能体的一个根本问题：它们在全新的上下文窗口中运行。一个实现 story-013 的智能体不记得在 stories 001 到 012 中做出的决定。它不记得来自第 13 章的架构讨论。它不知道为认证选择了哪种方法，或为什么。
Bob（Scrum Master 智能体）通过将规划文档转换为超详细的故事文件来解决这个问题。每个故事文件是自包含的。它描述了要实现的功能，验收标准，适用的架构约束，必须遵守的 API 合约以及测试要求。它包含足够的上下文，使一个重新开始的实现智能体可以正确完成故事，而无需读取 PRD，架构文档或任何其他故事文件。
"超详细"是关键操作词。一个传统的用户故事是三行：作为用户，我想要 X，以 Y。一个 BMAD 故事文件是 50–150 行。它包括：故事陈述，上下文（此故事开始时存在的系统状态），验收标准（具体的，可测试的），技术说明（要修改哪些文件，遵循哪些模式，应用哪些架构约束）以及测试要求（编写哪些测试，覆盖哪些边缘情况）。
这种详细程度因存在，故如果一个决定不在故事文件中，实现智能体就会编造一个——而这个编造可能与计划冲突。故事文件是防止这种情况的规格说明。
### 基于文件的手递作为连接组织

每个 BMAD 智能体产生 markdown 文件。每个 BMAD 智能体消费 markdown 文件。文件是系统——不是聊天消息，不是会话历史，不是人对已做决定的内存。
这使 BMAD 会话相互独立。业务分析师会话结束。产品经理会话在第二天开始，读取 `product-brief.md`，并不丢失上下文时继续。架构师会话读取 `prd.md` 和 `epics-and-stories.md`。Bob 读取 `architecture.md`。Amelia 一次读取一个故事文件。
没有智能体需记住先前会话中发生了什么。没有人需向下一个智能体介绍前一个智能体的决定。文件将上下文向前传递。
此即应用于流程而非单个会话的上下文工程。使 CLAUDE.md 有效的相同原则——持久，结构化，前向引用的上下文——使 BMAD 文件链有效。
### 在 OpenTalon 中

在 OpenTalon 中，两大支柱在 Web 平台的构建方式中可见。第四部分涵盖支柱 1：Mary，John，Sally 和 Winston 产生规划文档。第 13–15 章产生 `product-brief.md`，`prd.md`，`epics-and-stories.md`，`ux-spec.md` 和 `architecture.md`——全部在 `_bmad/artifacts/` 中。第四部分的最后一章（第 16 章）启动支柱 2：Bob 将这些规划文档转换为 24 个故事文件，Amelia 逐个故事实现，Quinn 用自动化测试验证。Web 平台是从规划文档构建的，而不是从聊天消息构建的，且每个实现会话从一个自包含的故事文件开始，而不是重建的会话历史。
