## 第 14.5 节：实现就绪检查

现在 `_bmad/artifacts/` 中存在四个工件：`product-brief.md`，`prd.md`，`epics-and-stories.md` 和 `ux-spec.md`。在写下一行实现代码之前，必须回答一个问题：这四个文档是否相互一致？
这不是官僚主义的关卡。这是对一种特定且昂贵的故障模式的保险：在实现了三周后发现 PRD 要求的功能架构无法支持，或者某个故事引用了一个从未被指定的技术能力。实现就绪检查（Implementation Readiness check）在问题还很便宜就能修复时抓住这些不对齐——在任何代码存在之前。
### 运行就绪检查

调用 John 故事审查：
```
/bmad-agent-bmm-pm

"审查 epics-and-stories.md 与 prd.md 的对应关系。验证：
1. PRD 中的每个功能需求都至少在
   一个故事中得到了体现
2. 每个故事的验收标准是具体且可测试的
3. 没有故事引用 PRD 范围外节中的功能

列出任何差距或不对齐之处。"
```

John 审查两个文档并返回一个结构化的差距报告。OpenTalon 的一个真实感输出：
> **差距 1：** FR-2（用量计量）要求分别记录 `input_tokens` 和 `output_tokens`。故事 3.2（按模型细分）在表格中显示了这些作为独立列，但故事 2.1（创建 API 密钥）和故事 1.1（注册）没有与计量相关的验收标准。这是正确的——计量在 API 级别发生，而非前端注册流程。无需操作，但应在架构中注明。
>
> **差距 2：** 没有故事涵盖用户的 API 密钥从两个设备同时使用的情况。PRD 没有涉及此问题；UX 规格说明也没有。建议：向故事 2.1 添加一个验收标准——"当同一 API 密钥在多个并发请求中使用时，所有请求都被无冲突地代理（无会话锁定）。"
>
> **差距 3：** 范围外节说"没有预算警报"，但故事 3.1（每日图表）有一条验收标准是"接近 100,000 token 的天数显示警告色。"这与范围外节矛盾。从故事 3.1 中移除警告色标准，或将预算警报明确包含在范围内。

第二和第三个差距需人类决策。John 将其提出来；人类解决它们。差距 2 是一个缺失的验收标准——将其添加到 `epics-and-stories.md` 并继续。差距 3 是一个范围冲突——做出决定并更新。
这些差距现在被发现，在规划文档中，而不是在 Amelia 在开发的第三周实现了矛盾的行为之后。
### 就绪检查确认了什么

当检查通过时——或者更准确地说，当它提出的差距被解决时——规划阶段就完成了。四个对齐的文档描述了一个系统，它：
- 服务于正确的用户（product-brief.md）
- 有明确的需求和清晰的范围（prd.md）
- 将工作划分为具有可测试标准的故事（epics-and-stories.md）
- 为最复杂的组件提供了清晰的 UX（ux-spec.md）

这不是完美。在架构设计和实现过程中仍会发现差距。但常见类别的规划矛盾——没有故事的 PRD 需求，没有验收标准的故事，节与节之间的范围冲突——已被消除。

### 刚刚发生了什么

三个智能体跨越三个会话为 OpenTalon Web 平台生成了一个完整的规划包。每个智能体以文件形式读取前一个智能体的输出。每个输出在成为下一个的输入之前都经过审查。人类在每个审查点参与——确认，纠正，决定差距——但不负责生成任何文档。
这种劳动分工是 BMAD 在实践中的第一支柱：智能体规划与每个阶段转换时的人工验证。第二支柱从下一章开始。
