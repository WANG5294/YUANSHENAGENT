## 第 14.2 节：John 通过引导式对话创建 PRD

Mary 交出了产品简介。John 在说一个字之前先阅读它。此即基于文件的手递在起作用：John 不问关于目标用户或核心问题的问题，因为 `product-brief.md` 已回答了这些问题。John 从 Mary 离开的地方精确接手。
调用 John：`/bmad-agent-bmm-pm`。Claude Code 采用 John 角色并立即读取 `_bmad/artifacts/product-brief.md`。过渡是干净的——下一个智能体将前一个智能体的输出作为其起始上下文。
### 引导式对话模式

John 按节处理 PRD 结构。他不会倾倒一个完整的草稿并要求一轮反馈。他呈现每一节，要求人类确认或纠正，并且只在每节获得批准后才继续。此即使 PRD 可靠的检查点结构。
OpenTalon PRD 对话的示例：
> **John：** 基于 Mary 的简介，我有核心问题。让我从问题陈述开始验证它，然后再进入需求。
>
> *问题陈述草稿：* "使用 AI 编码助手的独立开发者没有实时可见性来了解 token 消耗。使用量数据的缺失产生了账单焦虑，限制了本可提高生产力的 AI 工具的采用。"
>
> 这准确代表了问题吗，还是需要调整？
>
> **人类：** 这很准确。"账单焦虑限制采用"的表述完全正确。
>
> **John：** 好的。进入用户画像。我从 Mary 的简介中有一个主要画像——预算有限的独立开发者。我想添加一个：管理小团队（2-4 名开发者）AI 工具支出的技术团队负责人。这第二个画像反映了一个你希望在 V1 中服务的真实用户，还是超出了范围？
>
> **人类：** V1 超出范围。从一个用户类型开始。
>
> **John：** 确认——单一画像，保持范围紧凑。现在功能需求...

John 的问题很精确。他不是在探索——他是在确认。当他问第二个画像是否在范围内时，他想要一个是/否答案，而不是一场讨论。引导式对话因高效，故 John 在提问之前已完成了起草工作。
### PRD 结构

John 生成的 OpenTalon PRD 包括：
```markdown
# OpenTalon PRD

## 问题陈述
使用 AI 编码助手的独立开发者没有实时
可见性来了解 token 消耗。这造成了账单焦虑，
限制了采用。

## 用户画像
**主要：预算有限的独立开发者**
[来自 product-brief.md——引用，而非重复]

## 功能需求
**FR-1：CLI 智能体**
CLI 必须将所有 LLM 调用路由到 OpenTalon API 代理，
而非直接调用 OpenRouter。

**FR-2：用量计量**
API 必须记录每次请求的 token 使用量，包括：
用户 ID、模型名称、输入 token、输出 token、时间戳。

**FR-3：仪表盘**
Web 平台必须显示：过去 30 天的每日 token
使用量（柱状图）、按模型细分（表格）、
当前 API 密钥状态。

**FR-4：认证**
通过魔法链接邮件注册。不存储密码。

**FR-5：API 密钥管理**
用户可以创建、查看（仅前缀）和吊销 API 密钥。

## 非功能需求
**NFR-1：** CLI 对 LLM 调用响应时间增加 < 200ms 延迟
**NFR-2：** 仪表盘在标准连接上加载时间 < 2 秒
**NFR-3：** API 密钥哈希使用 bcrypt（不存储明文）

## V1 范围外
- 多用户团队（每账户单个用户）
- 预算警报和通知
- 模型级速率限制
- 自托管部署（V1 使用 Railway + Vercel）

## 成功指标
[来自 product-brief.md]

## 风险
- 代理延迟 NFR 可能具有挑战性——提前验证
```

John 在任何故事被编写之前呈现完整结构供人类审查。
### 本书的处理方式

本书不打印完整的 OpenTalon PRD。John 通过对话生成的 PRD 将反映读者在其 Mary 会话中做出的具体选择——已确认的画像，已声明范围外的项目，已接受或缓解的风险。重要的是结构和对话模式。
读者将拥有自己的 `prd.md`，在细节上与这里描述的不同。这是正确的。引导式对话是重点，而不是特定的输出。
### 在 OpenTalon 中

在 OpenTalon 中，PRD 是规划阶段和实现阶段之间的合约。每个功能需求至少成为一个故事。每个非功能需求成为一个故事的验收标准。范围外的项目防止范围蔓延——当未来的会话试图添加团队管理或预算警报时，PRD 明确记录了这些是 V2 功能。John 的引导式对话产生了一个准确且完整的文档，因为人类在每节开始之前已验证了每一节。PRD 被提交到 `_bmad/artifacts/prd.md`。
