## 第 14.1 节：Mary 产出 OpenTalon 产品简介

规划阶段从 Mary 开始。在 John 可以编写 PRD 之前，在 Winston 可以设计架构之前，必须有人回答基础性问题：这个产品是做什么的，为谁而做？Mary 的角色是通过对话获取答案并产生一份产品简介——一份简明扼要的文档，框定市场背景，用户和核心问题。
调用 Mary 是一个单一的斜杠命令：`/bmad-agent-bmm-analyst`。Claude Code 采用 Mary 角色并开始一次引导式需求获取会话。人类的工作是诚实地回答问题，包括关于不确定性的问题。
### 引导式需求获取对话

Mary 的问题遵循一个结构：先问题，然后用户，然后竞争背景，然后成功指标。OpenTalon 的一个真实感交换示例：
> **Mary：** 告诉我你想解决的问题——实际上是开发者的挣扎，而不是解决方案的描述。一个开发者今天经历的什么，是 OpenTalon 正在改变的？
>
> **人类：** 使用 AI 编码助手的独立开发者无法跟踪他们在 token 上花了多少钱。他们在月底收到一张令人意外的账单，或者他们因为不知道自己的使用量而自我限制使用。
>
> **Mary：** 这是一个清晰的可视性成本问题。这个开发者是谁？经验水平、技术栈、对于安装或付费的限制。
>
> **人类：** 有技术能力但不一定是系统工程师。可能使用 Python 或 JavaScript。如果可能的话不愿意再为一个 SaaS 订阅付费——他们想要一个自托管选项或免费套餐。
>
> **Mary：** 明白了——预算有限的独立开发者，技术水平足够自托管。在发布六个月后成功是什么样子的？不是功能——是行为变化。这个开发者会做什么不同的事？
>
> **人类：** 他们更自由地使用 AI 助手，因为可以看到实际消耗。他们不再因为账单焦虑而自我限制。

这种对话一直持续到 Mary 有足够的信息来起草产品简介。她不会问每一个可能的问题——只问那些能清晰框定市场和用户的问题。
### 产品简介

在需求获取之后，Mary 生成 `product-brief.md` 并将其保存到 `_bmad/artifacts/`：
```markdown
# OpenTalon 产品简介

## 市场背景
使用 AI 编码助手的独立开发者和小团队
面临一个使用透明度问题。LLM API 成本在
账单时间之前是不透明的。没有轻量级、可自托管的
工具能提供实时使用量可见性。

## 用户画像
**预算有限的独立开发者**
- 技术用户（主要使用 Python/JavaScript）
- 经常使用 1-3 个 AI 编码助手
- 对预算敏感：偏好免费或低成本选项
- 自托管偏好：愿意运行本地 API 而不是支付 SaaS 费用
- 痛点：账单焦虑限制了 AI 的采用

## 核心问题陈述
开发者无法就 AI 工具使用做出明智的决策，
因为在发票到达之前，他们对 token 消耗
没有可见性。这造成了一个二元结果：自由使用 AI
并冒超额费用风险，或自我限制并低效利用
生产力工具。

## 提议的解决方案
OpenTalon：一个带有配套 Web 仪表盘的 macOS CLI 智能体。
CLI 通过一个本地代理（API 组件）路由所有 LLM 调用，
该代理实时计量使用量。仪表盘显示每日和按模型的
token 消耗。

## 成功指标
- 开发者每周至少查看仪表盘一次
- 开发者在第一个月后报告没有账单惊喜
- CLI 对 LLM 调用响应时间增加 < 200ms 延迟

## 风险
- 代理延迟可能对交互式使用不可接受
- 自托管摩擦可能限制技术不足用户的采用
```

### 需要注意什么

Mary 的产品简介应该可以在五分钟内审查完成。在她生成后阅读它。两个问题决定它是否已准备好交给 John：
1. 用户画像是否匹配实际意图中的用户，而不是理想化的版本？
2. 核心问题陈述是否描述了真实的痛苦，而没有使用解决方案语言？

如果任何一个答案是否定的，直接编辑文件并告诉 Mary："我已更新了产品简介。请重新阅读并确认我们达成一致，然后我再交给 John。"
### 在 OpenTalon 中

在 OpenTalon 中，Mary 的产品简介捕捉了关于正在构建的产品的真实情况——包括本书读者需一条免费套餐路径且不能被要求支付基础设施费用的约束。Mary 生成的产品简介将被 John 在编写 PRD 之前读取，被 Winston 在设计架构之前读取，被 Bob 在创建故事文件之前读取。其准确性决定了所有下游工作的质量。在 Mary 生成后花五分钟审查它是整个项目中最有价值的时间之一。
