## 第 16.4 节：Quinn 生成自动化测试

Amelia 实现。Quinn 验证。这两个智能体被设计为相互独立——Quinn 的测试是针对故事文件中的验收标准编写的，而不是针对 Amelia 的实现选择。这种分离确保 Quinn 的测试能捕获 Amelia 实现错误的情况，而不仅仅是 Quinn 会以不同方式实现的情况。
调用 Quinn：`/bmad-agent-bmm-qa`。Claude Code 采用 Quinn 角色。Quinn 读取已完成的故事文件和 Amelia 的实现。
### Quinn 对 OpenTalon 的方法

Quinn 用两个测试工具：
- **pytest** 用于 FastAPI API（单元测试和集成测试）
- **Playwright** 用于 Web 平台（端到端测试）

TEA 模块（测试卓越加速器，Testing Excellence Accelerator）提供 34 个测试模式。对于 OpenTalon，其中四个最为相关：
**模式 1：认证状态测试**
覆盖每个受保护端点的已认证和未认证状态的测试。Quinn 测试：
- 对 `/api-keys` 的未认证请求返回 401
- 使用有效 Supabase JWT 的请求返回预期响应
- 使用过期 JWT 的请求返回 401（不是 500）

```python
# api/tests/test_auth_states.py
async def test_api_keys_requires_auth(client: AsyncClient):
    response = await client.get("/api-keys")
    assert response.status_code == 401

async def test_api_keys_with_valid_jwt(
    client: AsyncClient, auth_headers: dict
):
    response = await client.get("/api-keys", headers=auth_headers)
    assert response.status_code == 200
```

**模式 2：API 密钥生命周期测试**
验证完整生命周期的测试：创建，用，吊销，以及拒绝已吊销的密钥。
```python
async def test_api_key_lifecycle(
    client: AsyncClient, auth_headers: dict
):
    # 创建
    create_resp = await client.post("/api-keys",
                                     headers=auth_headers)
    assert create_resp.status_code == 201
    key = create_resp.json()["key"]
    prefix = create_resp.json()["prefix"]

    # 使用
    proxy_resp = await client.post(
        "/v1/chat/completions",
        headers={"X-Api-Key": key},
        json={"model": "test", "messages": []}
    )
    # （代理在测试环境中可能无法到达 OpenRouter——没关系）
    assert proxy_resp.status_code != 401

    # 吊销
    revoke_resp = await client.delete(
        f"/api-keys/{prefix}", headers=auth_headers
    )
    assert revoke_resp.status_code == 204

    # 尝试在吊销后使用
    revoked_resp = await client.post(
        "/v1/chat/completions",
        headers={"X-Api-Key": key},
        json={"model": "test", "messages": []}
    )
    assert revoked_resp.status_code == 401
```

**模式 3：用量计量验证**
测试代理端点在请求后是否正确记录了 token 用量。
```python
async def test_proxy_records_usage(
    client: AsyncClient, valid_api_key: str, db_session
):
    initial_count = await db_session.scalar(
        select(func.count()).select_from(UsageLog)
    )

    await client.post(
        "/v1/chat/completions",
        headers={"X-Api-Key": valid_api_key},
        json={"model": "mock-model", "messages": [
            {"role": "user", "content": "hello"}
        ]}
    )

    final_count = await db_session.scalar(
        select(func.count()).select_from(UsageLog)
    )
    assert final_count == initial_count + 1
```

**模式 4：每日端点的零填充验证**
架构注记将此识别为特殊情况：用量为零的天数必须返回零值行，而不是省略的行。
```python
async def test_daily_usage_includes_zero_days(
    client: AsyncClient, auth_headers: dict
):
    response = await client.get(
        "/usage/daily?days=7", headers=auth_headers
    )
    data = response.json()
    assert len(data) == 7  # 始终 7 条记录，即使使用量为 0
    for entry in data:
        assert "date" in entry
        assert "total_tokens" in entry
        assert entry["total_tokens"] >= 0  # 0 是有效的
```

### Quinn 和 Amelia 协同工作

Quinn 的测试在 Amelia 将故事标记为 done 之后，但在故事被认为完全完成之前编写。如果 Quinn 的测试失败，故事状态恢复为"in-progress"，Amelia 必须修复实现。
这不是对抗性的。Quinn 测试行为，而非实现。当 Quinn 的测试失败时，这意味着一个故事验收标准未被完全满足——这正是 Amelia 针对性修复所需的信息。

### 在 OpenTalon 中

在 OpenTalon 中，Quinn 为所有三个史诗生成测试。FastAPI 测试套件从两个方向覆盖 API 端点：直接（单元测试和集成测试）和通过 Web 平台的行为（通过 Playwright 针对本地运行栈）。第 18 章涵盖了 Quinn 的工作所贡献的完整测试金字塔。现在，每个故事在 Quinn 的测试通过之前不算完成。
