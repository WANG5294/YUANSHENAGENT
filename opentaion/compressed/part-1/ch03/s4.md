## 章节 3.4：工具调用的经济学

每次工具调用都有代价。理解成本结构会改变你设计任务的方式。

成本有三项：Token、延迟，有时还有金钱。Token 成本最不直观。Claude Code 运行打印 500 行日志输出的 Bash 命令时，这 500 行被追加到对话历史。后续每次迭代须处理现已长 500 行的历史。一次冗长工具调用就可能给后续每次迭代增加数千 Token。成本是复利。

此复利正是 Grep 和 Read 之间选择比看起来更重要的原因。一次找到含某函数调用的三行的 Grep 调用可能返回 150 Token。读这三行所在整个文件可能返回 4,000 Token。两者都给 Claude Code 相关信息，但一个比另一个在上下文中留了 3,850 Token 空间供后续使用。二十次迭代时，差异是 77,000 Token——相当于每提示后为会话剩余时间加了一篇短篇小说长度。

延迟是第二成本。工具调用需时间：文件读取快，测试套件慢，网络抓取依赖网络。每次文件编辑后都运行完整测试套件的智能体，付出的延迟成本使会话缓慢，在数十次迭代中累积。正确设计通常是先批量编辑再运行测试，而非单独验证每次编辑。反馈循环有价值；每次按键都触发反馈循环则痛苦。

金钱成本真实，依赖模型。API 定价下，Token 成本直接转费用。订阅定价下，转速率限制消耗。两者都是有限资源。重复读同一大文件的智能体工作流——因开发者没设计上下文持久化相关信息——是在为重复付费的工作流。

多智能体工作流复利成本是上述所有成本的乘数。三个并行智能体处理三独立任务，各有独立上下文窗口。每个支付自己上下文累积成本。生成它们的编排器支付自己上下文加从每个工作者接收的结果。三智能体并行工作流通常比同组合任务的单智能体工作流贵 2.5-3 倍，因乘数效应。任务真正独立且时间节省重要时成本值得；开发者因感觉更强而选并行则不。

诚实数字：智能体工作流约消耗同任务下单轮聊天交互的 10-20 倍 Token。一直免费使用 Claude Chat 后转用 Claude Code 的开发者会很快遇到此数字，且可能是冲击。10-20 倍乘数来自多轮对话二次增长（每轮都处理所有前轮的完整历史）、工具输出冗长、以及智能体迭代时自我修正循环成倍增加 Token 消耗。

对 OpenTalon 的开发者（我们的 CLI 用户），这些经济学正是本书含整章成本优化内容的原因。第 20 章中，提示缓存（prompt caching）、模型路由（model routing）和 `/effort` 命令将把会话成本降约 70%。工具调用的经济学非抽象问题，而是我们正在构建的产品的定价模型。
