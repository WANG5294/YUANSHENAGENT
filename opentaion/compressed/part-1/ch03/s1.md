## 章节 3.1：为什么工具使用不同于文本生成

有了 CLAUDE.md 和已初始化项目，问题从 Claude Code 知道什么转向它能做什么。

文本生成是预测。语言模型产生函数时，是在预测当前上下文中最可能满足提示的 Token 序列。预测可能正确，也可能看似合理实则错误——可编译但失败的函数、处理快乐路径却边界崩溃的实现、因断言不正确而非行为正确而通过的测试。模型不知区别。它预测了 Token，这些 Token 是否对应可工作软件是预测之外的问题。

工具使用是世界中的行动。Claude Code 读文件时，接收文件中的实际字节——非对文件内容的预测。它运行测试时，接收测试实际输出——非对测试可能说什么的合理描述。它运行类型检查器时，接收实际类型错误——非对哪些类型可能冲突的猜测。区别在于预测现实与观察现实。

此差异——预测与观察——在 AI 研究中有一名字：接地（grounding）。接地的智能体，其推理锚定于世界实际状态，而非世界可能样子的统计模式。每次工具结果为下一步推理提供接地。Claude Code 读文件后，上下文中含文件实际内容。它运行测试后，上下文中含实际测试结果。"预测了什么"与"实际是什么"的差距随每次工具调用缩小。

接地效应累积。已读十二个文件并运行三条命令的智能体，上下文比只收代码库描述的智能体在现实中接地得多。此即长时间智能体任务——需读许多文件、运行许多命令、迭代代码直到测试通过的任务——可产生一次性生成无法产生的结果。

非所有工具接地效果相同。不可逆性频谱从安全到有后果：读文件无副作用；写文件可撤销（版本控制）；运行测试不改变状态；删文件无备份不可撤销；部署到生产改变运行系统。为智能体设计任务即设计它该用哪些工具及以何顺序——从观察开始（读、搜索、分析），然后才进入行动（写、执行、创建），使不可逆行动前推理已有良好接地。

此设计原则产生一模式：给智能体提供能验证自身工作的工具。测试、linter 和构建命令服务于质量，还服务于更深层——它们是智能体观察自身行动结果并了解行动是否正确的方式。无验证工具的智能体是生成代码后就停止的智能体；有验证工具的智能体则生成代码、运行它、读发生了什么、然后生成更好的代码。

在 OpenTalon 中，接地效应是 CLI 对编码任务比聊天机器人更可靠的原因。第 9 章实现智能体循环时，循环会在编辑文件前先读文件，报告成功前先运行测试，继续前检查自己工具输出。每次检查都是一次接地操作——智能体的预测与现实相遇并相应更新的时刻。循环架构即接地架构。
