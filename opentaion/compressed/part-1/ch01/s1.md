## 章节 1.1：置信陷阱（The Confidence Trap）

此失败模式一致到足以拥有名字。

开发者让 AI 编码助手实现功能。代码很快出现，通过编译，基本测试通过。函数做了名字所说的事。三十分钟后，代码评审或生产事故中问题浮出：新函数假设了六个月前就被代码库废弃的会话模型，或引入仅在两模块同时加载时才出现的循环导入，或重复 AI 未读的工具文件中已有逻辑。每部分独立正确，系统却不正确。

此即置信陷阱（confidence trap）。陷阱不在 AI 生成糟糕代码——而在于它无论有无足够上下文来生成好代码，都同样自信。未读过代码库的人类工程师会有所保留："我觉得是这样，但你去检查 auth 模块。"AI 不会。它产出完整、连贯、格式良好的函数，而输出的自信程度不能告诉你产生此输出的上下文是否完整。

其机制是 Token 预测。语言模型通过预测给定前文后最可能的下一 Token 来生成代码。此优化是局部的：下一 Token 应与当前响应中之前的 Token 匹配。它非全局的：生成代码不需适应模型尚未完整读取的代码库架构。上下文窗口只含一个提示和一个函数签名时，模型生成适合该提示和签名的代码。此代码是否适配系统其余部分，生成过程不会追问。

类比建房。总承包商雇佣了熟练专业人员：水管工、电工、木工。每人在各自房间都做得非常出色。水管工按图纸精确布置管道。电工按图纸精确铺设导线管。木工按图纸精确搭建墙壁框架。但无人检查管道路线与导线管路线是否穿过同一墙的同一空腔。每个房间独立看都正确。但房子有结构性问题，只有封墙后才能发现。

当开发者把每个提示当独立任务处理时，AI 辅助开发就会以此精确模式失败。"给此函数加缓存。""重构此模块。""实现此端点。"每个响应局部连贯——在给定上下文中解决了陈述的问题。代码库中积累着这些响应，架构不连贯性也随之积累。

在 OpenTalon 中，此问题在我们试图连接 CLI 和 API 时便出现。CLI 用一种认证流程模型编写，API 用另一种。两者孤立看都没错，都很自信。集成测试失败，而 CLI 和 API 代码中无任何迹象表明存在不匹配。第 9 章中，我们在编写任何实现文件前先编写 CLI 规范——正是构建防止此情况的机制：一个两组件都读取的文档，使它们各自的自信建立在同一共享模型上。

问题不在工具，而在使用工具时未建立让工具安全的思维模型。此思维模型有三部分，即接下来三章的主题。第一部分是理解智能体到底是什么——非"更好的自动补全"，而是有特定架构、特定失败模式和特定优势的系统。此架构从一个循环开始。
