## 章节 1.3：Claude Code 主循环的工作原理

终止条件比你想象的简单。

Claude Code 主循环运行到模型响应不含任何工具调用为止。此时控制权返回用户。模型根据当前上下文判断任务是否完成。若完成，返回纯文本。若未完成，调工具，观察结果，然后运行下一次迭代。此循环是单线程的：一次一迭代，每次建立在前一次之上。

模型在每个迭代中看到完整对话历史加每次工具调用的结果。非摘要，非压缩版，而是自会话开始累积的消息、工具输入和工具输出的实际序列。此即上下文管理如此重要的原因——每次工具调用都会增加历史记录，而它在体积和成本上都在增长。

精确机制：你输入提示后，Claude Code 将其连同当前对话历史发送给模型。模型响应要么 (a) 纯文本消息，结束本次迭代；要么 (b) 一个或多个工具调用规范。若响应含工具调用，Claude Code 执行每个，收集结果，追加到历史，然后将整批送回模型进行下次迭代。循环持续到模型产生无工具调用的迭代。

此架构的一个特性直接改变你与长时间任务交互的方式：异步输入队列（asynchronous input queue）。Claude Code 运行时——执行工具调用、等待响应——你可输入新指令。Claude Code 将其排队，在下一次迭代边界传递给模型。你无需停止任务、等待完成、再用新指令重新开始。可在飞行中做中途修正。此即"打断"运行中智能体的感觉背后的机制。

此架构的简洁性是刻意的。有清晰终止条件的单线程循环是可预测的。你可推理它在做什么及为何。更复杂的架构——多并行智能体、嵌套编排、持久化后台进程——确实存在且在某些上下文中有用，但更难调试，也更难信任。Claude Code 主循环用复杂性换可控性。对构建生产软件的独立开发者，此交换正确。

实际经济影响：每次工具调用都消耗 Token 和时间。模型须处理每次迭代中不断增长的历史。需二十次工具调用的任务并非二十个独立操作——而是二十次迭代，每次处理比上一次略大的上下文。工具调用与成本间关系非线性，且比线性更糟：每次调用都增加后续调用必须处理的上下文。此算术使上下文管理成为真正的工程问题，而非学术问题。

在 OpenTalon 中，Claude Code 主循环是每章工作的环境。Claude Code 在第 9 章编写 CLI 智能体循环时，正用自己主循环构建结构相似之物。第 12 章观察到实现会话中工具调用的累积时，我们将目睹同一循环运行得比它被设计高效运行的时长更长。理解此循环有成本、有上下文、有终止条件，是智能管理它的先决条件。此管理即第 2 章主题。
