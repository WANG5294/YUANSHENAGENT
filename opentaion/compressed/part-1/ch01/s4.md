## 章节 1.4：为何 OpenTalon 不同

理解 Claude Code 主循环揭示了什么才是工具真正具备智能体能力——以及什么达不到标准。

多数 AI 编码工具并非智能体。它们是智能包装器（smart wrappers）：语言模型连到输入框，返回文本。输出可能是代码。用户粘贴到编辑器，运行，见错误，粘贴回输入框，接收修复，粘贴修复，重复。模型是聪明部分，用户是循环。工具从不观察自己输出结果，从不据代码运行后情况修改方法。它生成、停止、等待。

三个属性区分真正智能体工具与智能包装器。第一：持久化工具使用（persistent tool use）——在开发者实际环境而非沙盒预览中读、写、执行的能力。第二：环境反馈（environmental feedback）——观察行动结果（测试失败、编译错误、文件变更）并将其纳入下一步推理的能力。第三：多轮规划（multi-turn planning）——分解复杂任务、跟踪已完成步骤、并在数十次工具调用中维持计划的能力。

OpenTalon 将拥有全部三者。

在 OpenTalon 中，CLI 读取并修改用户项目中的文件。编写代码并运行测试时，接收测试输出作为工具结果。此结果改变它接下来做什么。测试通过则报告成功；测试失败则读错误、诊断问题、编辑相关文件、再次运行测试。非我们将配置的特性——而是感知-推理-行动循环在对真实环境有持久化访问时所做的。

OpenTalon V1 的多轮规划比 Claude Code 简单：它在单会话内跨工具调用维护简短任务列表。不会在会话间持久化计划，不会在途中分支或重新考虑。这些限制刻意的——使系统可理解可调试。更复杂的规划机制是理解了更简单机制的成本后再添置之物。

诚实评估：OpenTalon V1 达不到 Claude Code 的复杂程度，而这正是构建首个智能体系统的独立开发者的正确目标。Claude Code 背后有多年工程积累——广泛工具定义、精细权限系统、会话连续性、多智能体编排。OpenTalon 将有可工作的智能体循环、少量工具和清晰架构。价值不在功能集，而在理解构建每块的成本。

这让我们回到置信陷阱。OpenTalon 将面临与第 1.1 节相同失败模式：不同上下文中构建 CLI 和 API 时出现集成不匹配；Claude Code 跨多会话实现功能时，若先前决策未持久化到文件，它会忘记；添加多组件功能时，一组件自信假设契约，另一组件自信违反它。这些非假设——它们是第三部分要解决的真实问题。

OpenTalon 与智能包装器不同，非因避免这些问题，而因它由理解这些问题的人构建。
