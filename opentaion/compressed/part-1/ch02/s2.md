## 章节 2.2：什么在会话之间存活

Claude Code 有遗忘症（amnesia）。每次会话从零开始。

非 Bug，而是语言模型工作方式的后果：对话历史存在于上下文窗口中，会话结束时窗口被清空，下个会话无法访问上个会话发生了什么。昨天花三小时实现 CLI 认证模块的模型，今天毫无记忆。它不知哪些文件被创建，哪些决策被做出，哪些方法被尝试又放弃。它像项目不存在一样开始。

这让单次会话使用过 AI 助手并发现其连贯的开发者惊讶。一次会话内，Claude Code 是连贯的——它记得一切，因一切仍在窗口里。此错觉在你首次关闭终端、第二天重新打开、让其继续上次停下的地方时破碎。它不知你停在哪里，只知你在新会话开始时告诉它的内容。

会话间存活的，精确且仅限写入磁盘的内容：

**CLAUDE.md 文件**——每个会话开始时被重读。最可靠的持久化形式，因 Claude Code 被设计为读取它们。无论你在 CLAUDE.md 中写什么——项目约定、技术栈决策、架构规则、禁止模式——都可在该目录下运行的每个会话开始时访问到。

**自动记忆笔记（Auto-memory notes）**——简短的 Markdown 文件，Claude Code 写入 `~/.claude/projects/[project-hash]/memory/`，在会话间持久化观察结果。与 CLAUDE.md 不同：自动记忆由 Claude Code 在会话期间更新；CLAUDE.md 由开发者编写维护。

**会话记忆摘要（Session Memory summaries）**——Claude Code 在会话被压缩或恢复时自动写入的背景笔记。在同一项目上下文内持久化，可在 `/compact` 后存活。比 CLAUDE.md 更脆弱——不能存活所有会话边界——但弥补积极开发中项目与暂停项目间的差距。

**设置和权限持久化**：`.claude/settings.json` 在会话间存活，因此权限配置、钩子定义和工具限制无需重设。

什么不存活：工具调用历史、中间推理、模型上次会话中通过读文件建立的"对代码库的理解"。昨天读了项目中每个文件的智能体，今天一个也没读。

实际后果：无刻意持久化机制的每个会话，都是盲开始的会话。智能体会读所需文件，但除非有东西告诉它，它不知要读哪些。除非有东西记录，它不知上周做了哪些决定。除非有东西跟踪，它不知项目正处于 20 个里程碑中的第 3 个。遗忘症非模型能解决的问题——是开发者必须围绕设计的约束。

在 OpenTalon 中，此遗忘问题是 `book/progress.md` 和 `book/opentaion-state.md` 存在的原因。此二文件是项目的持久化记忆。Progress.md 精确告诉 Claude Code 写作停在哪、接下来做什么。Opentaion-state.md 精确告诉 Claude Code 哪些代码存在、哪些不存在。此二文件不依赖 Claude Code 之前读过——它们被编写为可被无会话历史的模型冷读理解。此设计要求非偶然，而是上下文工程（context engineering）的核心洞见，第 2.4 节将其命名并审视。
