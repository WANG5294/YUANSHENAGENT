## 章节 2.3：记忆层级

并非所有持久化平等。记忆层级（memory hierarchy）将四层级从最可靠到最不可靠排列——你选的层级决定了信息实际存活多久。

四层级，从最持久到最不持久：

**第一层——CLAUDE.md 文件。** 最可靠层级。每个会话开始时被读取，`/compact` 后被重读，会话期间生成的任何子智能体都可访问。无限期存活，因是版本控制下的普通文件。信息需在下周、下月或别人打开项目时可用，应放在 CLAUDE.md。这是不会退化的层级。

**第二层——自动记忆笔记（Auto-memory notes）。** Claude Code 会话期间写入 `~/.claude/projects/[project-hash]/memory/` 的笔记。比单会话持久，但比 CLAUDE.md 差——可被手动清除，默认不在版本控制下，同一项目其他开发者无法访问。对独立开发者，自动记忆对中期连续性可靠；对团队则不然。

**第三层——会话记忆摘要（Session Memory summaries）。** 会话中存活 `/compact` 操作的背景笔记。上下文窗口填满且 `/compact` 运行时，Claude Code 写入关键决策、文件路径和任务状态摘要。会话继续用压缩后上下文，但摘要保留基本结构。会话记忆关终端重开后不存活，是工作日内的连续性，非跨天连续性。

**第四层——对话历史（Conversation history）。** 最不稳定层级。只在会话运行且窗口未被清除时存在。它含一切：每提示、每响应、每次工具调用、每次文件读取。信息最丰富的层级，也最短暂。会话结束即消失。

此层级有压倒性实践含义：若你通过聊天向 Claude Code 传达信息而未被记录在上层，则可认为已丢失。对话中做出但未写入文件的决策，是下个会话不知道的决策，会被重新做出，可能不同方式。

这听起来显而易见。实践中，人们易把聊天当持久。"我告诉过 Claude Code 全程用异步 SQLAlchemy。"你写进 CLAUDE.md 了吗？若无，下个会话不知。依赖聊天记忆保存架构决策的开发者会发现决策被从未听说过的智能体推翻、忽略或矛盾。

在 OpenTalon 中，每个持久化决策都存在于文件中——永不在聊天消息中。架构决策在 `opentaion-state.md`，项目约定在根 `CLAUDE.md`，写作状态在 `progress.md`。新 Claude Code 会话启动时，读这些文件并立即定位。它不需被告知已构建了什么、要遵守什么约定、从哪开始——因所有这些信息都在记忆层级第一层，在我们刻意更改前一直存在。

此即使长期运行的智能体项目保持连贯的设计原则。下节解释为何需独特术语——上下文工程（context engineering）——来命名应用此原则的学科。
