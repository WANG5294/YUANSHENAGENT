## 章节 2.1：上下文窗口作为有限资源

上下文窗口（context window）非特性，而是预算——你花掉的每个 Token，都意味着后续工作可用 Token 少了一个。

此认识很重要，因开发者习惯性视上下文为"可用空间"。模型有 200K Token 上下文？空间大了去了。把整个代码库喂给它，附上每个相关文件，再加错误日志、文档、项目历史。模型会找出什么是重要的。

它不会。它所做的，是在每次迭代中处理完整 200K Token，计算成本随输入长度平方增长。20K Token 时，模型快速且准确。100K Token 时，它变慢并开始丢失前面内容。180K Token 时，它昂贵、明显变慢、且易遗漏数千 Token 前的指令。上下文窗口不会断崖式下降——逐渐退化。开发者不会在退化开始时收到错误消息。模型悄无声息变差，且易被误归因。

n² 问题：注意力机制（attention）——允许语言模型将上下文中任何 Token 与其他任何 Token 关联的机制——计算成本随序列长度平方增长。上下文翻倍，计算量翻四倍。此即性能平滑退化而非突然崩溃的原因，也是实际可用上下文远小于技术最大值的原因。200K Token 上下文窗口不意味着可可靠使用 200K Token。它意味着模型会尝试处理 200K Token，并在高端产生退化输出。

退化在实践中：智能体重复已完成工作，重读已读过的文件，产生忽略会话初期指定架构决策的代码，或询问两千 Token 前已回答的问题。这些都不像编译错误那样明显，而是细微错误，累积导致会话无法推进。

在 OpenTalon 中，上下文管理是构建连贯软件的开发会话与产生局部正确但全局不连贯代码的分界线——正是第 1 章的置信陷阱，但如今我们理解了机制。Claude Code 在数小时会话中构建 CLI 时，上下文窗口会被工具结果、中间推理和文件内容填满。无刻意管理，会话会在实现完成前退化。防止退化的策略是第 2 章和第 12 章的主题。第一步是理解会话结束时什么能存活。

上下文管理是区分智能体工程（agentic engineering）与氛围编码（vibe coding）的技能。氛围编码即向模型提问、接受输出、继续前进。智能体工程是设计信息环境——模型读什么、何时读、什么在会话间持久化——以便在整个项目生命周期中产生连贯输出。第 2 章其余部分构建此设计工具。
