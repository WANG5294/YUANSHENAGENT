## 章节 4.5：反模式：不可维护的 CLAUDE.md

前三节构建了好 CLAUDE.md 的图景：层次结构化、简短、具体、可操作。本节从失败模式探讨同一问题。四种反模式占据绝大多数 CLAUDE.md 功能失调。了解它们保护你不重蹈覆辙。

### 反模式 1：垃圾场

从 80 行开始、三个月内不受控增长到 600 行的 CLAUDE.md 是垃圾场。每次出问题——Claude Code 建议错测试命令、推荐不在技术栈的库、忘命名约定——就有人加规则。加的感觉像改进。整体效果则非。

问题不在个别添加，而在缺乏删减。CLAUDE.md 应通过有意决策成长，非积累每会话中的每个教训。加规则时问：已有规则是否已覆盖此情况只是不够具体，或根本问题在规则未被读到。超 200 行无强理由的文件已变垃圾场。修复法非添规则——是清理，删所有代码本身已传达的内容。

### 反模式 2：矛盾工厂

不同时间由不同冲动添加的规则互相矛盾。六月前说"用基于类的视图"，上月加的规则说"优先用函数组件"。Claude Code 遵循哪个？不确定——智能体不确定行为意味不可预测输出。

矛盾隐蔽，因它们看似全面。从两面覆盖某话题的文件（"所有 I/O 用 async"和"同步 httpx 对简单脚本可接受"）看似详尽。实际上给了 Claude Code 做任何一个的许可，它会在当下做上下文最方便的。

修复需定期审查——非仅加规则，而是将文件作连贯文本来读。任何对另一规则创建例外的规则都是删或合并的候选。引导问题：若 Claude Code 只能遵循两条规则之一，应遵循哪条？删另一条。

### 反模式 3：过时引用

CLAUDE.md 说"对 API 设计参见 @docs/api-contract.md。"但 `docs/api-contract.md` 在六周前重构中移到 `api/specs/contract.md`。Claude Code 加载 CLAUDE.md，找 @-引用，尝试加载文件，然后静默失败或啥也不加载。

过时引用比矛盾难捕捉，因需检查文件系统非仅读 CLAUDE.md。预防措施：将 CLAUDE.md 当代码对待——需定期通过"此引用存在吗？"检查。补救：定期运行 `grep "@" CLAUDE.md` 验证每被引用文件仍在指定路径。此亦第 6 节钩子包括会话前检查的原因——在会话开始前而非任务中捕捉过时引用。

### 反模式 4：一厢情愿的指令

"编写安全、生产质量的代码。""遵循最佳实践。""更改前仔细思考。"这些指令感觉重要，无行为后果。

若 Claude Code 可基于指令做具体决策，则它可操作化。"不对用户输入用 `eval()` 或 `exec()`"可操作化。"编写安全代码"则非——它要求 Claude Code 推断"安全"意味什么，推断因上下文而异。"提交前运行 `uv run pytest tests/`"可操作化。"测试你的更改"则非。

测试简单：对 CLAUDE.md 中任何规则，问"什么具体行动会违反此规则？"不能具体回答则是一厢情愿。要么让它具体，要么删。

### 维护协议

CLAUDE.md 是活文档，非一次设置永忘。每会话揭示智能体做你不会做的决策时，都应审查——非添责怪智能体的规则，而是问现有规则是否够清晰、或规则已过时。视 CLAUDE.md 为回归目标：若 Claude Code 在新会话中行为与 CLAUDE.md 指定不匹配，要么 CLAUDE.md 错，要么它不够具体。

OpenTalon 中，第 2 章根 CLAUDE.md 被设计一开始干净。添组件文件前，用反模式审视它值吗？有规则看起来像垃圾场添加吗？有规则矛盾吗？有引用指向不存在文件吗？你对 CLAUDE.md 第一次编辑，就是把它当作受维护文档的开始。

### 里程碑 M3：生产级 CLAUDE.md 和单仓库结构

带反模式在脑，里程碑：创建三组件 CLAUDE.md，各保持简短，验证它们一起构成连贯无矛盾系统。

在 OpenTalon 仓库中创建以下三文件：

```
# cli/CLAUDE.md
```

```markdown
# CLI — 组件规则

## 环境
- 包管理器：uv（绝不用 pip，绝不用 poetry）
- 运行智能体：`uv run python -m opentaion "你的提示"`
- 运行测试：`uv run pytest tests/ -v`
- Lint：`uv run ruff check . && uv run black --check .`

## Click 约定
- 命令使用 @click.command() 和 @click.option() 装饰器定义
- 分组在 `cli = click.group()` 下
- 通过 Click 上下文（ctx.obj）传递配置——而不是全局变量

## Rich 显示
- 所有控制台输出通过 `rich.console.Console()`
- 进度：`with console.status("[bold]Working..."):`
- 错误：`console.print("[red]Error:[/red] message", err=True)`

## 反模式（IMPORTANT）
- 绝不从 web/ 或 api/ 导入——CLI 是独立工具
- 绝不使用同步 httpx——所有 HTTP 调用使用 async/await
- 绝不要硬编码 API 基础 URL——从配置读取
```

```
# web/CLAUDE.md
```

```markdown
# Web — 组件规则

## 环境
- Node.js 20+。框架：Vite 5 + React 18 + TypeScript（严格模式）
- 开发服务器：`npm run dev`
- 构建：`npm run build`
- 测试：`npm run test`（vitest）

## 组件约定
- 函数组件配合显式 TypeScript prop 类型
- 局部状态：仅使用 useState——无外部状态管理
- 样式：仅使用 Tailwind 工具类——无独立 CSS 文件
- 图表：所有数据可视化使用 Recharts 库
- 无路由库——基于 Supabase 认证状态的条件渲染

## 反模式（IMPORTANT）
- 绝不使用内联样式——仅使用 Tailwind 类
- 绝不在 web 代码中访问 api/ 或 cli/
- 如果 Tailwind 或 Recharts 已满足需求，绝不添加 npm 包
```

```
# api/CLAUDE.md
```

```markdown
# API — 组件规则

## 环境
- 包管理器：uv
- 运行：`uv run uvicorn opentaion_api.main:app --reload`
- 运行测试：`uv run pytest tests/ -v`

## FastAPI 约定
- 所有路由返回带类型的 Pydantic 响应模型——绝不返回原生的字典
- 端点路径：连字符连接、名词优先（`/usage-records`、`/api-keys`）
- 认证：通过 `Depends(get_current_user)` 提取当前用户——不手动读取头部

## SQLAlchemy 约定
- 所有数据库操作使用异步会话
- 每个资源一个模型文件，位于 models/ 中
- 通过 Alembic 进行迁移——绝不手动修改数据库模式

## 反模式（IMPORTANT）
- 绝不提交连接字符串——使用环境变量
- 绝不编写原生 SQL——使用 SQLAlchemy 查询 API
- 绝不返回带有错误体的 HTTP 200——使用正确的状态码
```

三文件与根 CLAUDE.md 并存，单仓库结构完整。各组件加领域特定规则，不重复根文件覆盖内容。根文件指定技术栈；组件文件指定如何使用。Claude Code 打开 `cli/` 时知 Python/uv/Click 约定；打开 `web/` 时知 React/TypeScript/Tailwind 约定。两组件文件互不知晓对方。

现在通读全部四个 CLAUDE.md——根加三组件——据开发环境做调整。测试命令可能不同，Python 版本可能不同。这是你对 CLAUDE.md 作为受维护文档的第一次有意编辑。本节反模式是你添前删什么的指南。

### 刚刚发生了什么

本章以加载机制开始，以完整四文件 CLAUDE.md 系统结束。机制解释为何结构有效，反模式解释为何维护不良结构会失败。组件文件将原则付诸实践。

里程碑 M3 是生产级 CLAUDE.md：第 2 章优化根文件加覆盖完整单仓库的三组件文件。从此刻起，每打开 OpenTalon 的会话都拥有当前相关组件中工作所需上下文——不加载不相关规则、无矛盾、无过时引用。

下章从被动上下文转向主动工具：技能系统，让你直接将可复用程序嵌入 Claude Code 命令集。
