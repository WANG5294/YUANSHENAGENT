## 章节 4.2：@-引用系统与条件加载

前一节描述了 Claude Code 如何基于目录层次自动加载 CLAUDE.md。自动加载处理最常见情况：处理某组件时始终适用的规则。但还有第二种加载机制——你从 CLAUDE.md 内部刻意触发的——处理另一情况：大型、专业、仅有时相关的参考文档。

`@`-引用是你将文件包含到另一文件的方式。带特定分隔符的文件包含。仅此。

### 基本 @-引用

任何 CLAUDE.md 内部，如下行使文件在被处理时被包含：

```
@docs/architecture.md
```

路径相对仓库根。Claude Code 加载含此行的 CLAUDE.md 时，读 `docs/architecture.md` 并将其内容插入到 `@` 引用位置。结果与将 `architecture.md` 内容直接粘贴到 CLAUDE.md 完全相同——从上下文窗口看，它们是一个文件。

这有直接用途：将规则与参考材料分离。规则是简短命令式语句（"用 uv 别用 pip"）。参考材料是冗长描述性内容（二十张表的数据模型、四十个端点的 API 规范）。混在一块使文件难维护。@-引用让你将规则留在 CLAUDE.md，参考材料放专用文件，在相关处引用。

@-引用非条件式。行若存在，文件加载。总是、每会话、从 CLAUDE.md 被处理起。这对第 3.4 节成本分析重要：@-引用文件每会话计入上下文预算，如内容直接写入 CLAUDE.md。导入 4,000 Token 架构文档意味着每会话开始短 4,000 Token。

### 条件加载：散文模式

替代方案是实践者称条件加载（conditional loading）的方式，尽管 Claude Code 无内置条件语法。机制是散文指令，非代码。

CLAUDE.md 中可写：

```
## 当处理 API 时

当你在 api/ 中编辑文件时，在对任何路由处理器进行更改前，
请先阅读 @api/specs/endpoint-reference.md。该文件包含当前
API 契约。不要以会破坏它的方式修改端点。
```

`@api/specs/endpoint-reference.md` 引用在散文中，非顶层独立行。Claude Code 看到指令，处理 API 代码时，当描述条件触发时加载引用文件。处理 CLI 代码时，无加载此文件的指令，所以不加载。

关键区分：始终加载 vs. 按需加载。CLAUDE.md 顶层 @-引用始终加载。嵌入条件散文指令中的 @-引用按需加载——仅当描述条件被满足。

成本差异迅速累积。假设 OpenTalon 架构文档 3,000 Token。若通过顶层 @-引用始终加载，每个处理 CLI 的会话（与文档无关）从开始支付 3,000 Token。十会话即 30,000 Token 花在从未需要的文档。条件加载——"即将做架构更改时请读 @docs/architecture.md"——这些 Token 只在相关时才花费。

### 何时用 @-引用

实用规则：对大型、稳定且始终相关的文档用 @-引用。对大型、稳定但有时相关的文档用条件散文。

"始终相关"包括项目技术栈规范、数据模型模式、构建客户端时的 API 契约。这些被频繁引用——始终加载成本低于重复按需加载。

"有时相关"包括完整测试策略文档、部署运行手册、第三方服务 API 参考。这些需时至关重要，其他时候无关。条件散文让它们保持在上下文外，直到工作特别需要。

要避免的模式：将一切作风险对冲导入。@-引用了十二个文档"以便 Claude Code 总有上下文"的 CLAUDE.md，牺牲了上下文预算而不确定获得了什么。导入要有目的。说不出哪个特定工具调用需要此文档就别导入。

### OpenTalon 连接

OpenTalon 中，`opentaion-state.md` 是不应被 @-引用的最清晰例子。它描述代码库当前状态——存在哪些文件、实现了什么、上里程碑产出了什么。编写涉及代码的章节时必要，编写无里程碑的概念性章节时不相关。`write-section` 技能在其七步骤中明确读它，因明确读取应在技能运行时发生而非会话开始时。若 `opentaion-state.md` 被 @-引用到根 CLAUDE.md，其 400+ Token 内容会在每会话加载——包括无代码的会话。条件模式（技能需时读）更便宜且同样可靠。
