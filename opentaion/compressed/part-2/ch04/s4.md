## 章节 4.4：包含什么和排除什么

每个与 CLAUDE.md 打过交道的经验开发者，都至少犯过同一错一次：加东西因感觉重要而非可操作。文件变大，会话变慢，智能体开始提技术上符合规则但实践中错的建议——因它试图同时满足二十条指令，最新的被其他十九条噪音稀释。

本节是纠正。CLAUDE.md 有大小纪律（size discipline）——200 行是软上限，保持低于上限的原因非美学而是性能。超 200 行的 CLAUDE.md 试图做太多。

### 200 行纪律

上下文有限。CLAUDE.md 中每个 Token 都是不能用于代码、测试输出或工具调用结果的 Token。400 行 CLAUDE.md 在开始工作前就消耗约 2,000 Token。会话尾声，上下文已复杂时，此 2,000 Token 可能成为智能体保持连贯与丢失对早期决策跟踪的分水岭。

200 行限制强制纪律：每行须证明地位。若规则被删而智能体仍在 90% 情况正确行为，就删它。10% 不值 20 行上下文成本。

### 什么属于 CLAUDE.md

**确切的命令。** 最可操作的东西是构建、测试、运行项目所需的精确 Shell 命令。非描述——是实际命令。

```
Build: uv run python -m build
Test: uv run pytest tests/ -v
Lint: uv run ruff check . && uv run black --check .
Run: uv run python -m opentaion "在此输入提示"
```

这些不会很快过时。变化时文件易更新。还防止了最常见失败模式：Claude Code 猜错测试命令并运行错误东西。

**工程原则。** 有明确行为后果的规则。"函数不超 40 行——超出则提取辅助函数。""所有 HTTP 调用异步。""CLI 中无全局状态。"这些非样式偏好——是影响每个代码决策的架构约束。

**文件夹结构。** 简要描述什么在哪。三到五行。足以让 Claude Code 将新文件放正确位置而无需猜测。非每文件完整清单。

**命名约定。** 当名称遵循非自明模式时。"API 路由处理器命名为 `handle_{resource}_{action}`。""测试文件镜像源文件结构：`tests/core/test_agent.py` 对应 `core/agent.py`。"那些违反会造成人类需修复的不一致性的约定。

**关键规则标记为 IMPORTANT。** 当规则关键——违反导致安全问题、生产事故或根本性架构错误——明确标记：

```
IMPORTANT: 绝不要将环境变量提交到仓库中。
所有秘密放在 .env（已 gitignore）中。API 密钥验证钩子
将阻止包含裸 API 密钥字符串的提交。
```

`IMPORTANT:` 标签非装饰。Claude Code 会权衡。标记为 IMPORTANT 的规则比埋在散文中获得更强关注。

### 什么不属于 CLAUDE.md

**linter 已强制执行的代码风格规则。** 若 `ruff` 配为强制执行 88 字符行限制，不需再说"保持行数 88 字符以下"。工具自动强制执行。写在 CLAUDE.md 造成重复，最终分歧——linter 更新了，CLAUDE.md 没，互相矛盾。

**关于技术如何工作的解释。** CLAUDE.md 非给读者的文档，而是给智能体的指令。"FastAPI 用依赖注入"是 Claude Code 已有背景知识。"用 `Depends(get_current_user)` 认证——绝不手动读 Authorization 头部"是指令。

**个性指令。** "回答中保持简洁专业。"Claude Code 已有自己语调。关于听起来怎样的指令只是噪音，稀释编码指令空间。

**每会话加载的全包式 @-引用。** "开始工作时加载 @docs/all-design-notes.md。"若文档始终相关，其关键点应属 CLAUDE.md 本身。若非始终相关，用条件散文。无条件加载大型文档而仅有时需，是无保障回报的上下文税。

**一厢情愿的指令。** "始终编写生产质量、安全、经过良好测试的代码。"非指令。是愿望。Claude Code 无法操作化。"所有用户输入函数须在边界验证——无原始输入到数据库查询"可操作化。模糊质量愿望则非。

### 基准测试

添任何东西到 CLAUDE.md 前问：从未见过此项目的有能力工程师是否需要知道此点才能正确工作？若是，它属。若工程师会通过读代码弄明白，它不属于。CLAUDE.md 填补代码本身无法传达的空白——架构约束、工具选择、禁止模式、操作项目所需命令。

### OpenTalon 连接

OpenTalon 中，第 2 章的根 CLAUDE.md 按这些原则设计。会话前检查清单可操作。技术栈规范具体。"Claude Code 绝不能做什么"清单是行为约束非愿望。第 4.5 节写三组件 CLAUDE.md 时，同样纪律适用。组件文件会短——可能各 40-60 行——因只覆盖组件在根文件基础上真正需要的东西。简短非关注不足的迹象，简短就是目标。
