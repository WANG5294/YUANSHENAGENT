## 第 7.1 节：MCP 的工作原理：协议与传输

第 3 章介绍的工具——Read、Write、Edit、Bash、Glob、Grep——在本地文件系统和 shell 上操作，让 Claude Code 能访问项目的代码、配置和测试输出。但它们没有让 Claude Code 访问外部系统：存放 PR 的 GitHub 仓库、用户数据所在的生成数据库，或 Web 仪表板运行的浏览器。

MCP——模型上下文协议（Model Context Protocol）——就是这座桥梁，定义了一个标准，规定了 Claude Code 如何连接到外部工具服务器、发现其能力及调用其工具。第 7 章构建 OpenTalon 开发所需的三个外部连接：GitHub 仓库管理、PostgreSQL 数据库检查、Playwright 浏览器测试。

### 协议基础

MCP 将 Claude Code（客户端）与其使用的工具（服务器）分离。MCP 服务器是一个实现该协议的进程：它接收来自 Claude Code 的请求，执行请求的操作，并返回结构化结果。Claude Code 用启动时请求获取其工具列表发现服务器能做什么。列表中的每个工具都有一个名称、一个描述和一个描述其参数的 JSON schema。Claude Code 使用描述和 schema 来决定何时调用该工具及传递什么参数。

此分离因重要，有两个原因。首先，即工具集可以不修改 Claude Code 本身时扩展——新的 MCP 服务器增加新的能力。其次，即工具可以以与 Claude Code 不同的权限运行。一个只读的 PostgreSQL MCP 服务器无论 Claude Code 要求它做什么，都无法写入数据库；权限边界存在于服务器层面，而非 Claude Code 的配置中。

### 两种传输类型

Claude Code 支持两种传输机制连接到 MCP 服务器：

**stdio** —— MCP 服务器是一个由 Claude Code 生成的本地进程。Claude Code 通过其标准输入和输出流与之通信。GitHub MCP 服务器、PostgreSQL MCP 服务器和 Playwright MCP 服务器都支持 stdio 传输。本地 stdio 服务器会话启动时启动，会话结束时停止，与 Claude Code 运行在同一台机器上，可以访问本地文件系统和网络。配置指定了要运行的命令及它所需的环境变量。

**HTTP** —— MCP 服务器是一个可通过 URL 访问的远程进程。Claude Code 向该服务器发送 HTTP POST 请求。此传输适用于共享的团队服务器、云托管的工具提供商，或必须特定环境中运行的 MCP 服务器（例如，一个包含不应存在于开发者笔记本电脑上的数据库凭据的服务器）。OpenTalon 开发中只使用 stdio 服务器。

### 工具发现与上下文预算

会话启动时，Claude Code 连接到每个已配置的 MCP 服务器并请求其工具列表。工具列表——名称、描述和 JSON schema——被加载到上下文窗口中。每个工具定义都会消耗令牌。一个有 40 个工具的服务器可能仅在工具定义上就消耗 20,000 个令牌，而这时对话还没有开始。

此即工具过载问题。三个 MCP 服务器合计 80 个工具，其工具定义可能消耗 70,000 多个令牌——占据上下文窗口的大部分。第 7.5 节介绍了解决此问题的工具搜索（Tool Search）功能。Claude Code 可以从加载所有工具定义切换到只加载与当前任务相关的定义，从而将上下文消耗减少约 85%。

### .mcp.json 文件

项目级的 MCP 服务器在项目根目录的`.mcp.json`中定义。该文件被提交到 git，并在所有处理该项目的会话（及团队成员）之间共享：

```json
{
  "mcpServers": {
    "playwright": {
      "command": "npx",
      "args": ["@playwright/mcp"],
      "type": "stdio"
    }
  }
}
```

用户级的 MCP 服务器（那些配置了个人 API 令牌的，如 GitHub）在`~/.claude/mcp.json`中定义，不提交到项目中。此分离将令牌排除在版本控制之外。

### 与 OpenTalon 的关联

OpenTalon 中，三个 MCP 服务器构成了外部感知层：GitHub 看到代码审查对话，PostgreSQL 看到数据，Playwright 看到 UI，共同赋予 Claude Code 在外部系统上闭合反馈循环的能力，而不仅仅是在本地文件上。当 Claude Code 第 16 章调试使用追踪 bug 时，它可直接查询数据库以查看数据的实际样子——而非推理数据可能是什么样子，直接访问就是诊断与猜测之间的区别。接下来的三个部分将依次连接每个服务器。
