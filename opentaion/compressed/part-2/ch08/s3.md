## 第 8.3 节：macOS 上的操作系统级沙箱

权限模式和规则在 Claude Code 层面运行，控制 Claude Code 会做什么——哪些工具调用需要批准，哪些模式允许或拒绝，不控制 Claude Code 在操作系统层面能做什么。一个拒绝`Bash(rm -rf:*)`的规则阻止 Claude Code 发出该命令，但它依赖于 Claude Code 遵守规则。操作系统级沙箱在下面一层运行：它限制进程本身被允许做什么，无论它请求什么。

macOS 提供了一个内置的沙箱机制，称为 Seatbelt（也称为 sandbox-exec）。Claude Code macOS 上的项目目录中运行时自动使用它，无需任何额外配置。

### macOS 沙箱强制执行的内容

默认情况下，Claude Code 激活的 macOS 沙箱将 Claude Code 进程限制为：

**允许读取：**
- 当前工作目录及所有子目录
- 标准系统库和框架
- 用户的 Claude Code 配置目录（`~/.claude/`）

**阻止读取：**
-`~/.ssh/`—— SSH 密钥、known_hosts、authorized_keys
-`~/.aws/`—— AWS 凭据
- 其他通常保存秘密的凭据目录

**网络：**
- 允许连接到已配置的 API 端点
- 沙箱不限制所有网络访问——Claude Code 必须能访问 Anthropic API、OpenRouter 和 MCP 服务器

**进程生成：**
- 允许在沙箱内执行 Bash 工具调用的子进程
- 生成的子进程继承相同的路径限制

关键保护：项目目录中运行的 Claude Code 无法读取`~/.ssh/`或`~/.aws/`中的文件，即使它尝试这样做。操作系统会拒绝读取。这种限制是结构性的——它不依赖于规则、钩子或 Claude Code 自身的合规性。

### 沙箱不能阻止什么

沙箱不是完整的防御。几个重要的限制：

**沙箱内的损害是可能的。** 如果一条对抗性指令导致 Claude Code 删除项目目录中的所有文件，沙箱不会阻止它。沙箱将爆炸半径限制在项目目录内，而非零。

**网络调用在很大程度上不受限制。** 除非特定端点被其他方式阻止，否则导致 Claude Code 将数据发送到外部 URL 的注入攻击将成功。对任意 URL 的 WebFetch 调用默认情况下不会被沙箱限制。

**提示注入绕过沙箱。** 嵌入在 Claude Code 读取的文件中的恶意指令将以 Claude Code 的所有允许权限执行。沙箱控制 Claude Code 可以访问什么；它不过滤 Claude Code 遵循哪些指令。

沙箱只是一层防御，而非全部防御。它与权限规则、钩子和第 8.5 节中的注入感知配置相结合，形成纵深防御法。

### 何时进一步加强

大多数 OpenTalon 开发，默认沙箱已足够。项目目录包含源代码、配置文件和测试夹具——没有什么是灾难性丢失或暴露的，因为仓库在 GitHub 上。更敏感的资产（`.env`中的生产 API 密钥、`~/.ssh/`中的 SSH 密钥）受到默认沙箱的保护。

若项目目录本身包含敏感数据——用于调试的生产数据副本、用于集成测试的凭据文件——在沙箱中配置显式的路径限制以排除这些特定文件：

```json
{
  "sandbox": {
    "additionalDenyPaths": [
      ".env.production",
      "tests/fixtures/sensitive-data/"
    ]
  }
}
```

OpenTalon，`.env`包含真实的 API 密钥，但被 git 忽略。项目目录的默认沙箱保护与 gitignore 相结合是合适的。

### 与 OpenTalon 的关联

OpenTalon 中，macOS 沙箱是一个不可见的保护层，无需设置和持续维护，将任何失误的爆炸半径——措辞不当的提示、注入的指令、意外的工具调用——限制在项目目录内。最重要的文件（SSH 密钥、凭据文件、其他项目目录）默认情况下在该边界之外。下一节在此基础上构建一个显式的威胁模型：编录沙箱保护什么，及什么需要额外措施。
