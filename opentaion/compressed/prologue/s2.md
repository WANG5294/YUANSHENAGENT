## 章节 P.2：OpenTalon 系统概览

开始构建前，你需要一张地图。

OpenTalon 由三组件构成：运行于 macOS 终端的 CLI、位于 CLI 与语言模型间的 API、供用户注册与管理访问权限的 Web 平台。各组件足够独立，可单独构建——我们将在第 11 章并行构建它们。但三者通过单一流程连接，你应在编写第一个文件前理解此流程。

CLI 是用户实际工作的界面。它接收自然语言提示，调用 OpenTalon API，将模型响应流式返回终端，并应用结果——编辑文件、运行命令、报告变更。此乃通过 Homebrew 分发的 Python 应用，使用 Click 构建命令接口，使用 Rich 构建终端显示。

API 是平台骨干。它对每个请求做 API 密钥认证，将 Token 用量记录到数据库，并将调用代理至 OpenRouter。从 CLI 角度看，API 与 OpenAI API 完全一样——使用相同协议。从平台角度看，API 是计费、速率限制和用量追踪的所在地。CLI 和用户浏览器均不直接调用 OpenRouter。

Web 平台是用户注册和管理访问权限之处。它处理三项任务：魔法链接邮箱认证、API 密钥创建和撤销、展示 Token 随时间消耗的用量仪表盘。这是部署于 Vercel 的 Vite + React 应用。仪表盘通过查询 API 用量端点，用 Recharts 按模型分组的柱状图展示每日 Token 用量。

在 OpenTalon 中，每个架构决策源于一个约束：API 须位于 CLI 与模型之间。仅此约束就让用量追踪成为可能，让 API 密钥认证实用，让 CLI 保持轻量 Python 脚本而无需知晓计费。当你在后续章看到看似不必要的间接层时，请回到此图。

OpenRouter 是 OpenTalon 用户的 LLM 提供商。它将多个开源模型聚合在一个 API 密钥后，通过 OpenAI 兼容接口暴露。图中四模型都可在 OpenRouter 免费层使用——无需信用卡。无 OpenRouter 预算的读者也可跟进每章，部署可工作系统，无需在模型推理上花钱。

先决条件：Python 组件（CLI 和 API）需 Python 3.12+，用 `uv` 管理。Web 平台和 BMAD 方法论工具（第 13 章安装）需 Node.js。若机器已有 Python 和 Node.js，便拥有一切所需。
