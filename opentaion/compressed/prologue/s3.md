## 章节 P.3：你将构建完成的内容

开始前，用具体术语描述最终状态。

陌生人通过链接找到 OpenTalon。她访问 Web 平台，输入邮箱，收到魔法链接。点击后进入仪表盘，创建 API 密钥。密钥完整显示一次——她复制下来。终端运行 `brew install yourgithub/opentaion/opentaion`。不到一分钟，她就有 CLI。设置两个环境变量：`OPENTAION_API_KEY` 和 `OPENTAION_API_URL`。然后运行第一个任务：

```bash
$ opentaion "auth.py 里有类型错误，找出来并修复"
```

CLI 读取 `auth.py`。它思考类型注解，识别函数签名与调用点间的冲突，产生编辑。它应用编辑，运行 `mypy auth.py` 验证修复，然后报告：

```
✓ OpenTalon v1.0.0  模型: deepseek/deepseek-r1  上下文: 6,840 tokens
→ 正在读取 auth.py...
→ 发现 authenticate() 和 verify_token() 中的 2 个类型错误
→ 正在应用修复...
→ 正在运行 mypy...
✓ 完成  (4 次工具调用 · 2.1 秒 · $0.00)
```

她检查仪表盘。6,840 个 Token 以 `deepseek/deepseek-r1` 归属出现在今日柱状图中。她可看到过去 30 天总用量，按天和模型细分。

此即最终产品。书中其余一切，皆从当下到彼处的路径。

完成状态下，OpenTalon 仓库含约 35 个源文件，分布在三组件中：CLI 有 8 个 Python 文件，API 有 14 个 Python 文件（含迁移文件），Web 平台有 8 个 TypeScript 文件。总应用代码约 3,500 行。测试套件覆盖 CLI 的智能体循环、上下文管理器和工具实现；API 的认证、密钥管理、代理和用量端点；Web 平台通过 Playwright 端到端测试的关键流程。约 20 个测试文件。非全面覆盖——而是独立开发者能实际维护的覆盖范围。

这是 V1 版本。此描述精确，非谦虚。仪表盘无分页。CLI 不支持多并发任务。错误信息功能齐全但未必友好。部署手动，未完全自动化。这些非失败——而是独立开发者交付可用之物而非完美设计所做的决策。每个权衡都经深思熟虑，在书中决策处均有解释。

完成系统确实证明了一个论点：独立开发者用本书技术，可设计、构建、测试并部署可工作的多组件软件平台——CLI、API 和 Web——以 Claude Code 为主要实现工具。开发者理解并维护的系统，即自己构建的系统。理解债务真实存在，本书有专门论述。但代码能工作、测试能通过、用户可注册并使用。

此即"发布"对独立开发者的意义。

让这一切成为可能的思维模型，出现在第一行代码之前。第 1 章从一个问题开始，每个用过 AI 编码工具的开发者最终都会问：为何输出看起来正确，做的事却是错的？
