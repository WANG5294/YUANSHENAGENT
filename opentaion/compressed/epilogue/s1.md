大约 3,500 行应用程序代码分布在三个组件中，78 个测试，20 个里程碑达成。Claude Code 编写了大约 85% 的应用程序代码。人类编写了规划文档，审查了每个故事的输出，做出了架构决策，并修复了就绪检查发现的三处缺口。
### 现在可行的用户旅程

一个陌生人发现了网上的 OpenTalon。他们访问 `https://opentaion.vercel.app`，输入他们的电子邮件地址，点击"发送魔法链接"。一封邮件到达。他们点击链接。他们看到一个空仪表盘。
他们导航到 API Keys，点击"创建密钥"，复制密钥。他们打开一个终端：
```bash
brew tap yourgithub/opentaion
brew install opentaion
export OPENTAION_API_URL=https://opentaion-api.up.railway.app
export OPENTAION_API_KEY=opentaion_abc12345...
opentaion "what files are in the current directory?"
```

Token 出现在终端中。他们刷新仪表盘。柱状图显示了今天的用量。模型细分显示了哪个模型处理了请求。
从注册到首次有用结果：五分钟以内。此即 V1 达到的基准。
### Claude Code 的贡献

Claude Code 编写了 CLI 的智能体循环，FastAPI 路由，React 组件，测试套件，BMAD 故事实现，CI/CD 工作流和 Homebrew 公式。它生成的代码干净，经过测试，遵循 CLAUDE.md 中的约定，在首次尝试时通过的概率高于不通过的概率。
Claude Code 没有贡献的部分：架构决策。用哪个数据库，代理应该如何工作，API 合约应该是什么，认证应该如何运作——这些都是 Winston 的决策和人类的验证。架构才是实现成功的原因。实现是更容易的部分。
### 比预期更难的事情

规划阶段。不是因为 BMAD 工具很难，而故好的规划需诚实地面对你不知道的事情。Mary 询问目标用户。诚实的答案是"注重预算的独立开发者"——而不是"所有地方的每个开发者"。承诺这个答案限制了设计，有时会让人感到不适（没有团队账户，没有企业功能），但最终是正确的。
### 比预期更容易的事情

集成。并行构建的三个组件（第 11 章）在单次会话中完成了集成。API 合约在第一次尝试时就近乎完美。环境变量名称匹配了。CORS 配置工作了。这不是运气——这是 SPEC.md 和架构文档在发挥作用。
### V1 的诚实状态

它是一个功能性的工具，不是精致的产品。仪表盘是实用主义的。错误信息本可以更有帮助。引导文档是最简的。关心这些事情的开发者在完成本书后有工作要做。关心拥有一个带有用量追踪的工作 AI 编程助手的开发者，现在已有了它。
此即 V1 的正确范围。
