# 麦克风闭环验证（agent"听"实机声音）

主机预览 WAV 只验证算法；实机声音叠加了 Q3 开关、喇叭机械特性、RMT 块间
空隙、GC 停顿。闭环 = 录下喇叭真实输出与预览对比，差异即"实机才有的问题"。

## 前置条件（2026-07-18 实测已打通）

- 本机是 VirtualBox 虚拟机：宿主机需勾选 设备→音频→**音频输入**，否则录到全零
- 每次闭环**先跑 `mic_check`**：确认非全零、记下噪声底 RMS（实测约 0.03）
- 麦克风尽量凑近喇叭（几厘米）提高信噪比

## 触发方式

物理按键 agent 按不到。播放逻辑必须封装成可 import 的函数（如 `piano.py`
的 `play()`），然后 `play_and_record("import piano; piano.play()", "rec.wav", 4)`
——先启动录音，0.5s 后经 REPL 触发，录完自动查峰值。

## 标准流程

1. `mic_check` → 非全零
2. `analyze_wav xxx_preview.wav 期望f0` → 参考本身干净
3. `play_and_record "import piano; piano.play()" rec.wav 4`
4. `analyze_wav rec.wav 期望f0`
5. `compare_audio rec.wav xxx_preview.wav`

## 硬性合格标准（判定只认数字，禁止降级）

1. **录音有效**：录音峰值 ≥ 5×噪声底 RMS（mic_check 给出），否则闭环无效=
   验证失败。补救顺序：麦克风贴近喇叭 → 加大音量 → 仍不行则如实报告
   "实机验证失败"。**严禁**用静态代码检查/乐理推算替代实机录音宣布合格。
2. **响度/包络类需求**：必须先生成 preview WAV，再 compare_audio，
   包络相关系数 ≥0.8 才合格；另用 20~25ms 窗算包络，相邻窗平均起伏
   应 <10%，否则就是可听见的锯齿。不达标=回改代码，不是改口径。
3. **汇报引用数字**：相关系数、音分偏差、峰值/噪声底比值。
   "确认发声""包络前大后小"不构成合格结论。

## 硬件前提（响度类需求先看这条）

板载是**压电蜂鸣器**：改 PWM 占空比≠改音量；把载波按时间片开/关门控
（PDM）会产生可听的周期性斩波（锯齿感）。要平滑响度包络，必须用
RMT PWM-DAC 波形合成方案（参考项目根目录 KEY.py 与
`RMT音频哒哒音问题说明.md`），或外接功放+DAC。接到"响度渐变"类任务
先加载 esp32-buzzer 技能确认硬件路径，别用占空比/门控硬做。

## 判读规则（什么可信）

| 现象 | 结论 |
|---|---|
| 录音有规律尖峰、预览没有 | RMT 块接缝哒哒音；间隔≈CHUNK/RATE 可定位分块参数 |
| 首尾各一个孤立尖峰 | 缺 ~10ms 直流斜坡（爆音） |
| 基频偏差 >10 音分 | 查 clock_div/TICKS 推算的实际采样率 |
| 包络相关低但都无尖峰 | 可能录到环境噪声，凑近喇叭重录 |

基频与哒声周期**可信**；包络形态中等可信；谐波/音色因通道染色**不可信**，只看趋势。
