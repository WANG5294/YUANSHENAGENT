# 版本迭代记录：小兰 v0 → Yuanshen v3

## 一、小兰 v0：完整功能的起点

小兰 v0（`小兰/小兰V0/xiaolan_v0.py`，858 行）是专注于 ESP32 数字钢琴开发的单片机 agent，基于 mini-claude-code(v4) 架构，接 Kimi 大模型（Anthropic 兼容端点，消息角色为 system/user/assistant，工具结果放在 user 消息的 tool_result 块中）。因 Kimi K3 是思考模型、思考内容会吃掉 max_tokens 预算（实测 max_tokens=2000 时思考耗尽预算、正文一个字没返回），v0 统一关闭 thinking 以保证工具调用输出稳定。

### 1. Agent 主循环

核心是标准 agent 循环：LLM 返回 tool_use → 本地执行工具 → tool_result 以 user 消息回灌 → 继续请求，直到 `stop_reason != "tool_use"` 输出纯文本。单任务上限 100 轮；耗尽后不再执行工具，程序注入一条指令，要求模型直接输出进度报告（已完成什么 / 卡在哪里 / 下一步建议）。

### 2. 工具体系（对模型完全同构的三个来源）

- **基础本地工具**：`bash` / `read_file` / `write_file` / `edit_file` / `TodoWrite`。bash 有命令黑名单拦截，文件操作经 `safe_path` 限制在工作目录内。
- **Skill 机制**：按 SKILL.md 规范（YAML frontmatter 的 name + description，正文按需注入）加载领域知识，v0 只带一个 `esp32-piano-debug` 技能。
- **Task 子代理**：唯一类型 `log-summary`，专职压缩冗长内容（串口日志、MicroPython Traceback、长文件），只读、限时 5 分钟、不可嵌套，返回不超过 200 字的结论，保持主 agent 上下文干净。

设备侧通过**自研最小 MCP 客户端**连接本地 stdio 服务器 `piano_workflow/esp32_piano_mcp.py`：换行分隔的 JSON-RPC 2.0，initialize 握手 → tools/list 发现 → tools/call 调用，后台线程读 stdout 入队列。12 个 MCP 工具覆盖设备通道（list_ports / upload / run_script / repl_exec / device_ls / device_rm / soft_reset）与音频闭环（mic_check / record_audio / play_and_record / analyze_wav / compare_audio），与本地工具拼在同一个 tools 列表里，模型无需区分来源。

### 3. 问候语金丝雀与上下文压缩

v0 最有特色的机制：system prompt 以最高优先级要求每轮回复第一句必须一字不差地是"你好，我是小兰，很高兴可以帮到你。"。主 REPL 每轮检查这句问候——只要它在，说明 system prompt 仍在生效；一旦缺失，判定上下文过长已把 system 挤失效，自动触发 `compress_context`：把全部历史（截尾 30000 字符）交给 LLM 压缩成结构化摘要（用户目标 / 已完成步骤 / 文件状态 / 未解决问题 / 下一步），整体替换 messages 后重试。压缩只在 end_turn 后进行，不会切断悬挂的 tool_use/tool_result 配对。

### 4. 安全红线与工程细节

system prompt 明令：禁止 erase_flash / esptool 刷写、禁止 sudo / rm -rf 等危险命令、文件操作限工作目录、串口限 /dev/ttyACM* 与 /dev/ttyUSB*、GPIO34/35（输入专用）禁配输出、不做与单片机无关的事。工作循环强调"只有硬件验证通过才算完成，代码写完不算完成；一次只改一个变量，不瞎猜"。

启动时若缺依赖，自动原地 re-exec 切换到 `piano_workflow/.venv` 解释器——判据是 `sys.prefix` 而非路径 resolve（venv 的 bin/python 是指向系统 python 的符号链接，resolve 会误判）。这套引导和 `.env` 读取被之后所有版本继承，v1–v3 均自动兜底沿用 `小兰V0/.env` 里已存的 API Key。

## 二、迭代脉络：v1 → v2 → v3

### 小兰 v1（632 行）：做减法的最小可实现版

v1 反向精简，验证"编写 → 上传 → 运行 → 看串口输出"这条主干本身是否可靠：

- **移除**麦克风闭环五工具（不注册）、Task 子代理、TodoWrite、问候语金丝雀与上下文压缩；
- **新增循环摘要**：模型每轮回复第一段必须报告上一轮循环的工具/成败/耗时，其中轮次与秒数由程序注入的**【循环状态】**提供，模型照抄不瞎猜——这是后续所有版本"程序管事实、模型管叙述"思路的起点；
- **新增四个斜杠命令**（本地处理，不进大模型）：`/model` 切换大模型（缺 Key 时给保存指引）、`/tool` 列工具、`/skill` 列技能、`/work` 探测环境能力；
- **技能重构**：从单个调试技能改为硬件手册功能分块——`esp32-pinmap` / `esp32-buzzer` / `esp32-led` / `esp32-keys` / `esp32-mcu` / `esp32-peripherals`，允许分块间有意重叠（各自视角不同）。

### 小兰 v2（1014 行）：加回功能并工程化闭环

- **恢复全部 12 个 MCP 工具**与麦克风闭环（VirtualBox 勾选音频输入后实测打通），新增 `mic-closed-loop` 技能，`/work` 增加麦克风实测；
- **循环摘要修正**：只总结上一轮（成败/技能/工具/做了什么），自然语言上限 100 字，严禁复述加载的 skill 正文；
- **主线任务 TodoList**：用户只描述目标（不用写"请编写/烧录/运行"，流程在 system prompt 里），每个任务自动初始化四步主线：编写代码 → 烧录代码 → 测试代码 → 完成；每轮【循环状态】回灌原始需求与进度，防止长任务走偏；
- **文件组织与归档**：每个任务开一个项目文件夹 `files/<时间戳_摘要>/`，产出的 py / wav（纯文件名自动落位）与流程.md、transcript.json、经验提取.md 同放；
- **经验提取**：任务结束后从流程中提取候选技能，展示全文经用户 y/n 确认后写入 `skills/exp-*/SKILL.md`，无人值守默认不保存。

### Yuanshen v3（1087 行）：prompt 架构重构，更名"Yuanshen"

v3 功能面沿用 v2（循环摘要、主线四步卡点、归档、经验确认、安全红线全部继承），核心变化在 **prompt 架构与输入通道**：

- **三层 prompt 架构**：system prompt 只放"每个任务都用得上"的规则与小事实——身份范围、主线规则、**工具功能精简总览**（一行一个）、**【当前接线】**（wiring.md 注入）、技能列表（仅 name+description）、验证与安全红线；工具**完整**用法放 tool prompt（description + input_schema）；大块知识（硬件分块文档、经验教训）走 Skill 按需注入。放置原则：变化慢/体积大/按需用 → Skill；变化快/体积小/全局相关 → system；工具用法 → tool prompt。
- **输入栏分流**：普通文本一律视为任务目标；`/doc <md路径>` 是与任务分离的**硬件文档导入通道**，校验 skill 格式（name/description 缺一拒绝装载）后复制进 skills/ 即时生效；新增 `/wiring` 查看/编辑当前接线。**换硬件的标准动作变成：写分块 md → /doc 导入 → 更新 wiring.md → 直接下任务，主程序零修改**。
- **wiring.md 常驻接线事实**：十几行的常驻成本可忽略，却能防住"GPIO12 外接上拉导致无法启动"这类事故。
- **烧录保护与验证红线**：upload 目标为 main.py 时由主程序拦截，先自动备份板上原程序到 backups/ 再部署，并置程序级卡点（未部署 main.py 不允许把"烧录代码"标记完成）；实机验证红线量化——录音峰值 ≥5×噪声底、响度/包络类须 compare_audio 相关系数 ≥0.8、汇报必须引用数字，严禁静态检查替代实机验证。
- **更名**：从"小兰"改为"Yuanshen"，对应任务书 5.2 的全部基本能力（文件传输/程序执行/复位/串口监控/日志检索/错误报告）与进阶项（远程文件管理、麦克风自动化验证）。

### 一条主线

四个版本的演进逻辑：v0 先把完整能力堆出来 → v1 砍到最小验证主干、确立"程序注入事实、模型照抄"的循环摘要 → v2 加回闭环并补齐任务管理、归档与经验沉淀 → v3 重新分层安放知识（system / tool / skill），把硬件说明变成可插拔文档，实现换硬件不改代码。
